パスワードを推測で破る攻撃には、1アカウントに大量のパスワードを試す**ブルートフォース**と、
1つの定番パスワードを大量のアカウントに横に試す**パスワードスプレー**がある。仕組みを整理して、
Go での基本的な防御（レート制限とハッシュ）まで押さえる。

## 攻撃の仕組み — 縦に総当たりか、横に薄く広げるか

どちらも「パスワードを推測する」攻撃だが、試し方の向きが違う。

| 攻撃 | やること | 狙い |
| --- | --- | --- |
| **ブルートフォース** | 1つのアカウントに大量のパスワードを次々試す | 弱いパスワードなら時間で必ず割れる |
| **パスワードスプレー** | 1つの定番パスワード（`Password1!` 等）を大量のアカウントに1回ずつ試す | **アカウント単位のロックアウトをかいくぐる** |

- ブルートフォースは「1アカウント × 多パスワード」なので、失敗回数がそのアカウントに集中する。
  だから**アカウントロックアウト**が効く
- パスワードスプレーは「1パスワード × 多アカウント」に薄く広げる。各アカウントの失敗は1回だけなので、
  素朴なロックアウトでは検知できない。**IP・全体での試行レート**で見る必要がある

> ⚠️ ログイン失敗を「アカウント単位」でしか数えていないと、スプレーは丸ごと素通りする。
> **「誰が」だけでなく「どこから・全体で何回」** も同時に数える。

## 対策の原則 — 4つを重ねる

単発の対策では抜ける。次の4つを重ねて「1つ破られても次で止まる」状態にする。

- **レート制限**: 単位時間あたりの試行回数に上限をかける。総当たりの速度そのものを殺す
- **ロックアウト／バックオフ**: 失敗が続いたら一時ロック、または待ち時間を指数的に伸ばす
- **MFA**: パスワードが割れても、二要素目がなければ通れない。最も効く一手
- **強いパスワードハッシュ**: 万一 DB が漏れても、bcrypt 等で**1件あたりの検証を意図的に遅く**しておけば、
  オフラインでの総当たりが現実的な時間で終わらない

> 💡 レート制限は「オンライン攻撃（サーバー越しの試行）」を、強いハッシュは「オフライン攻撃
> （漏れたハッシュへの総当たり）」を遅くする。**守る場所が違う**ので両方いる。

## Go での防御① — レート制限

`golang.org/x/time/rate` のトークンバケットで、ログイン試行に上限をかける。
キーごと（IP やアカウント）に limiter を持たせるのが基本形。

```go
import "golang.org/x/time/rate"

// 1秒あたり0.2回（=5秒に1回）まで、瞬間的には3回までバースト許可
func newLoginLimiter() *rate.Limiter {
	return rate.NewLimiter(rate.Every(5*time.Second), 3)
}

// リクエスト単位でキー（例: IP）ごとの limiter を引く
func (s *server) allowLogin(key string) bool {
	s.mu.Lock()
	lim, ok := s.limiters[key]
	if !ok {
		lim = newLoginLimiter()
		s.limiters[key] = lim
	}
	s.mu.Unlock()
	return lim.Allow() // トークンがなければ false → 429 を返す
}
```

- `Allow()` はトークンがあれば消費して `true`、なければ即 `false`。ブロックしたいなら `Wait(ctx)`
- スプレー対策には **IP 単位**、ブルートフォース対策には **アカウント単位**の両方を見るとよい
- 本番でプロセスをまたぐなら、この map を Redis 等の共有ストアに置き換える

## Go での防御② — パスワードハッシュと定数時間比較

パスワードは平文でもただの SHA-256 でも保存しない。**わざと遅い** `bcrypt` を使う。

```go
import "golang.org/x/crypto/bcrypt"

// 登録時: コスト（計算の重さ）を指定してハッシュ化。ソルトは自動で埋め込まれる
func hashPassword(pw string) (string, error) {
	h, err := bcrypt.GenerateFromPassword([]byte(pw), bcrypt.DefaultCost) // 既定10
	return string(h), err
}

// 検証時: 保存ハッシュと入力を比較。内部で定数時間比較される
func checkPassword(hash, pw string) bool {
	return bcrypt.CompareHashAndPassword([]byte(hash), []byte(pw)) == nil
}
```

- `DefaultCost`（10）は 2 の 10 乗回のストレッチ。ハードウェアが速くなったらコストを上げる
- **自前で `==` 比較しない**。トークンや API キーなど生の秘密値を比べるときは、
  比較時間から中身を推測される（タイミング攻撃）のを防ぐため `crypto/subtle` を使う

```go
import "crypto/subtle"

// 長さの違いも時間に出さない。等しければ 1、違えば 0
func secretEqual(a, b []byte) bool {
	return subtle.ConstantTimeCompare(a, b) == 1
}
```

> 🧭 C#（ASP.NET Core Identity）でも定石は同じ。`PasswordHasher<T>` が PBKDF2 でハッシュ＋検証を担い、
> `SignInManager` の **lockout**（`MaxFailedAccessAttempts` / `DefaultLockoutTimeSpan`）で失敗回数を止める。
> Go では自分で組む部品を、Identity は框組みとして持っている、という違い。

## まとめ

- **ブルートフォース＝縦に総当たり／パスワードスプレー＝横に薄く広げてロックアウト回避**。数える軸が違う
- 対策は **レート制限・ロックアウト・MFA・強いハッシュ**を重ねる（1つでは抜ける）
- レート制限は**オンライン攻撃**を、強いハッシュは**漏洩後のオフライン攻撃**を遅くする——守る場所が別
- Go では `x/time/rate` でキーごとにレート制限、`bcrypt` でわざと遅いハッシュ、生の秘密値は `crypto/subtle`
- 失敗ログは「アカウント単位」だけでなく **IP・全体の試行レート**でも監視する
