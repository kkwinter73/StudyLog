「並行処理」を支えるのが **プロセス** と **スレッド**。この2つの違いと、OS スレッドのコストを知ると、
Go の [goroutine](/posts/goroutine-channel-basics/) がなぜ「軽量スレッド」と呼ばれるのかが腑に落ちる。
ここでは goroutine の手前にある OS の仕組みを押さえる。

## プロセスとスレッド — まず定義

ざっくり言うと、**プロセス＝実行中のプログラム1つ分の箱**、**スレッド＝その箱の中で実際に動く実行の流れ**。

| | プロセス | スレッド |
| --- | --- | --- |
| 何か | 実行中のプログラム（アプリ1つ） | プロセス内の実行の流れ |
| メモリ | **専用のメモリ空間を持つ**（他プロセスと隔離） | **プロセスのメモリを共有**する |
| 関係 | 1プロセスに1つ以上のスレッド | 必ずどれかのプロセスに属する |
| 隔離 | 強い。1つ落ちても他に波及しにくい | 弱い。同じメモリを触るので影響し合う |

> 💡 「プロセスはマンション1棟、スレッドはその中の住人」。住人（スレッド）は同じ建物（メモリ）を
> 共有して動き、別の棟（プロセス）とは壁で隔てられている。

## 何を共有し、何を分けるか

ここが並行処理のキモ。**同じプロセス内のスレッドはメモリを共有する**。

- 共有する: コード・ヒープ（変数やオブジェクト）・開いたファイルなど
- 各自で持つ: スタック（呼び出し中の関数の状態）・実行位置

共有できるから、スレッド間のデータ受け渡しは速い。だが裏返しに、**同じデータを同時に書き換えると壊れる**
（データ競合）。だからロックや、Go なら channel での受け渡しが要る。

> ⚠️ プロセス間は隔離されているぶん安全だが、データのやり取りには専用の仕組み（IPC）が要り重い。
> スレッド間は速いが、共有メモリの扱いに注意が要る ——この**トレードオフ**が並行処理の出発点。

## OS スレッドはタダではない

スレッドはプロセスより軽いが、それでも **OS が管理する資源**でありコストがある。

- 1スレッドごとに **MB 単位のスタック**用メモリを確保する
- 生成・破棄に OS への依頼（システムコール）が要る
- どのスレッドに CPU を割り当てるか切り替える **コンテキストスイッチ**にも時間がかかる

つまり「1リクエスト1スレッド」で何万も立てると、メモリと切替コストで頭打ちになる。
この限界が、次の goroutine の動機になる。

> 🧭 C#/.NET で生 `Thread` を大量に作らず `Task`／`ThreadPool` を使うのも同じ理由。
> OS スレッドは有限で重いので、上に軽量な層をかぶせて使い回す発想は共通している。

## goroutine への橋渡し

Go の **goroutine** は OS スレッドそのものではなく、その上に乗る**ずっと軽い実行単位**。

- スタックは数 KB から始まり、必要に応じて伸びる（OS スレッドの MB 単位より桁違いに小さい）
- 何万・何十万でも立てられる
- Go ランタイムが、少数の OS スレッドの上に多数の goroutine を**載せ替えて動かす**（M:N スケジューリング）

```go
// OS スレッドを意識せず、go をつけるだけで実行の流れが1本増える
go handle(req) // これが「軽量スレッド」= goroutine
```

「スレッドは重いから乱発できない」という前提を、ランタイム側で肩代わりして崩したのが goroutine。
だから[goroutine と channel の基本](/posts/goroutine-channel-basics/)は、この土台の上で読むと腑に落ちる。

## まとめ

- **プロセス＝隔離されたメモリ箱、スレッド＝その中で動く実行の流れ**
- 同じプロセスのスレッドは**メモリを共有**する。速い反面、データ競合に注意が要る
- OS スレッドは MB 単位のメモリ＋生成・切替コストがあり、**無限には増やせない**
- **goroutine** は OS スレッドの上に乗る軽量単位。数 KB 始まりで何万でも立てられる
- Go ランタイムが少数の OS スレッドに多数の goroutine を載せる（M:N）——これが「軽量」の正体
