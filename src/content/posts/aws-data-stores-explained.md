> 📚 シリーズ「AWSをちゃんと理解する」(3 / 6)

「データを保存する」と一口に言っても、データの **性質によって最適な置き場所が違う**。
代表的な4タイプを、何を解決するかで整理する。

## データの置き場所は4タイプ

| タイプ | AWS | 向いているもの |
| --- | --- | --- |
| 関係データベース(RDB) | RDS | 表＋関係で持つ業務データ |
| インメモリキャッシュ | ElastiCache | よく使う結果を一時的に高速で |
| オブジェクトストレージ | S3 | ファイルそのもの（画像・バックアップ） |
| NoSQL | DynamoDB | 単純な key 引き・超スケール |

## RDS ＝ マネージドな関係データベース

**RDB** は、データを表（テーブル）と行・列で持ち、テーブル同士の関係を SQL で扱う方式。
**RDS** はそれを AWS が運用してくれるサービス。ここでは **PostgreSQL** を使用。
バックアップ・パッチ適用・冗長化といった「DB のお守り」を任せられるのが利点。

> 🧭 .NET の Entity Framework + SQL Server に当たる層。エンジンが PostgreSQL になるだけで考え方は同じ。

## ElastiCache ＝ 机の上の付箋（高速な一時置き場）

DB へ毎回問い合わせると重い。そこで **よく使う結果をメモリ上に覚えさせて即返す** のがキャッシュ。
**ElastiCache** はそのマネージド版で、エンジンは **Valkey**（Redis 互換の OSS フォーク）。

例えるなら、引き出し（DB）を毎回開けず、机の上の付箋（メモリ）に書いておく。速いが揮発しやすい
ので「消えても困らない一時データ」を置く。

## S3 ＝ ファイルの巨大倉庫

画像・動画・バックアップ・ビルド成果物のような **ファイルそのもの** を入れる倉庫が **S3**。
容量は実質無限・安価で、URL でファイルを取り出せる。

> 💡 大きなファイルを RDB に入れると重く高くつく。**ファイルは S3、その置き場所(URL)だけ DB に記録**、
> が定石。

## DynamoDB ＝ 即引ける NoSQL（ここでは state ロック用）

**DynamoDB** は key を渡せば即値が返る NoSQL。スキーマが緩く、巨大スケールに強い。
この構成では用途が特化していて、**Terraform の state ロック**に使っている
（複数人が同時にインフラ変更しないための排他制御＝鍵の取り合いを記録する小さな表）。

## どう使い分けるか

```text
関係のある業務データ      → RDS (PostgreSQL)
よく使う結果の高速化      → ElastiCache (Valkey)
ファイルそのもの          → S3
単純な key-value・排他制御 → DynamoDB
```

## まとめ

- データは性質で置き場所が変わる ——「とりあえず DB」にしない
- RDS は関係データベースのマネージド版（ここでは PostgreSQL）
- ElastiCache は高速な一時キャッシュ（Valkey）。消えてよいデータ向け
- S3 はファイルの倉庫。DB には URL だけ持たせる
- DynamoDB は即引ける NoSQL。ここでは state ロックに利用
