> 📚 シリーズ「コンテナをちゃんと理解する」(1 / 5)

`docker compose up` の中身を理解する出発点は「コンテナとは何か」。
[カーネルの役割](/posts/kernel-role-for-containers/)で触れた通り、コンテナは軽量VMではなく
**カーネルの機能で隔離されたただのプロセス**。その隔離を担う namespace と cgroup を仕組みから見る。

## VM とコンテナ — もう一度

決定的な違いは[カーネル](/posts/kernel-role-for-containers/)を複製するか共有するか。

| | VM | コンテナ |
| --- | --- | --- |
| OS(カーネル) | ゲストOSを丸ごと別に起動 | **ホストのカーネルを共有** |
| 起動 | 秒〜分（OS起動が要る） | ほぼ一瞬（プロセス起動と同じ） |
| 重さ | 重い（OS何台分） | 軽い（プロセス1個分） |
| 隔離の強さ | 強い（完全に別マシン） | 中（同じカーネル上で論理的に分離） |

> ⭐ コンテナはホストから見れば[普通のプロセス](/posts/process-vs-thread/)。`ps` にも出る。
> ではなぜ「自分専用のマシン」に見えるのか——それが namespace と cgroup の仕事。

## namespace — 「見える範囲」を区切る

namespace は、プロセスから**見える世界を限定**するカーネル機能。これで「自分以外いない」ように錯覚させる。

| namespace | 区切るもの |
| --- | --- |
| PID | プロセス一覧（コンテナ内は自分が PID 1 に見える） |
| NET | ネットワーク（専用のIP・ポート空間を持つ） |
| MNT | ファイルシステムのマウント（自分用のルート `/`） |
| UTS | ホスト名 |
| IPC | プロセス間通信の資源 |
| USER | ユーザー/グループID |

> 💡 コンテナの中で `ps` してもホストの他プロセスが見えないのは PID namespace のおかげ。
> 「自分専用の `/` がある」ように見えるのは MNT namespace。**見える範囲を区切っているだけ**。

## cgroup — 「使える量」を区切る

namespace が「見える範囲」なら、cgroup（control group）は **使えるリソースの量**を制限する。

- CPU をどれだけ使えるか（例: 0.5コア分まで）
- メモリの上限（超えたら OOM で kill）

```bash
# メモリ512MB・CPU 0.5コアに制限してコンテナを動かす
docker run --memory=512m --cpus=0.5 myapp
```

これがないと1つのコンテナがホストの資源を食い尽くせてしまう。cgroup で「お隣に迷惑をかけない」枠を作る。

## docker run は何をしているか

`docker run` 1発の裏では、ざっくりこうなっている。

```text
1. イメージからファイルシステム(自分用の /)を用意
2. namespace を作る（PID/NET/MNT… で見える範囲を隔離）
3. cgroup で CPU/メモリの上限を設定
4. その中でプロセス(CMD)を起動
```

> 🧭 「VM を1台立てる」感覚で捉えると重く感じるが、実体は**プロセスを隔離して起動するだけ**。
> だから起動が速く、たくさん動かせる。この身軽さがコンテナ普及の理由。

## まとめ

- コンテナは軽量VMではなく、**カーネルの機能で隔離されたただのプロセス**
- VM はカーネルごと複製、コンテナは**ホストのカーネルを共有**するから軽い
- **namespace** が「見える範囲」(PID/NET/MNT…)を区切る
- **cgroup** が「使える量」(CPU/メモリ)を区切る
- `docker run` は「FS用意→namespace→cgroup→プロセス起動」をやっている
