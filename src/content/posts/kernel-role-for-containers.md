「コンテナは VM より軽い」とよく言われるが、なぜ軽いのかは **OS カーネルの役割**を知ると一気に腑に落ちる。
この記事ではコンテナそのものより一段下、**カーネルがハードウェアとアプリの間で何をしているか**を押さえる。

## カーネルとは — ハードウェアの前に立つ管理人

アプリは CPU・メモリ・ディスク・ネットワークといった**ハードウェアを直接は触らない**。
間に立って取り次ぐのが **カーネル**（OS の中核）。アプリは「ファイルを読みたい」「メモリが欲しい」と
カーネルに**お願い**し、カーネルがハードウェアを操作して結果を返す。

```text
[ アプリ（あなたのGoバイナリ等） ]
            ↓ お願い（システムコール）
[ カーネル：資源の管理人 ]
            ↓ 操作
[ ハードウェア：CPU / メモリ / ディスク / NIC ]
```

なぜ直接触らせないか。複数のアプリが同時にハードウェアを奪い合うと壊れるからだ。
カーネルが**唯一の管理人**として割り振ることで、各アプリは「自分専用のマシンがある」かのように動ける。

## カーネルが担う4つの仕事

ハードウェアの取り次ぎは、ざっくり4種類に分けられる。

| 仕事 | 何をしているか | アプリから見ると |
| --- | --- | --- |
| プロセス管理 | どのアプリにいつ CPU を使わせるか割り振る | 自分が CPU を独占しているように見える |
| メモリ管理 | 物理メモリを各アプリに切り出して貸す | 自分専用の広いメモリがあるように見える |
| ファイルシステム | ディスク上のデータをファイル/ディレクトリに見せる | `/var/log/app.log` のような名前で読み書きできる |
| デバイス/ネットワーク | NIC やディスクなど機器を抽象化して扱う | ソケットやファイル経由で機器を意識せず使える |

> 💡 共通する発想は **抽象化**。生のハードウェアを、アプリが扱いやすい「プロセス」「メモリ空間」
> 「ファイル」「ソケット」という形に見せ替えてくれている。

## ユーザー空間とカーネル空間 — 境界とシステムコール

メモリ上の世界は2つに分かれている。アプリが動く **ユーザー空間** と、カーネルが動く **カーネル空間**。
アプリはカーネル空間に直接踏み込めず、決められた窓口 ——**システムコール**—— 経由でしか頼めない。

```text
ユーザー空間   | アプリのコード（権限：制限あり）
   --- システムコール（open / read / write …）---
カーネル空間   | カーネルのコード（権限：ハードウェア全権）
```

この境界があるおかげで、1つのアプリがクラッシュしても OS 全体やほかのアプリは巻き込まれない。
権限の壁で隔離されているからだ。

> 🧭 C#/.NET だと `File.ReadAllText()` を呼ぶと、内部で最終的に OS のシステムコール（`read` 等）に
> 行き着く。Go の `os.ReadFile()` も同じ。**高水準APIの底はどの言語でもカーネルに繋がっている**。

## コンテナ理解への橋渡し — 1つのカーネルを共有する

ここが本題。**VM とコンテナの決定的な違いはカーネルの扱い**にある。

- **VM**: ゲスト OS（＝カーネル）を丸ごと別に立てる。OS が何台も動くので重い。
- **コンテナ**: **ホストのカーネルをそのまま共有**し、アプリだけを分離する。OS は1つなので軽い。

ではカーネルが1つなのに、どうやって「自分専用のマシン」感を出すのか。
カーネルの2つの機能で、プロセスから見える世界を区切っているだけだ。

| 機能 | 役割 | 区切るもの |
| --- | --- | --- |
| namespace（名前空間） | 見える範囲を限定する | プロセス一覧・ネットワーク・ファイルツリー・ホスト名など |
| cgroups | 使える量を制限する | CPU・メモリの上限 |

> ⭐ コンテナは「軽量な VM」ではなく、**カーネルの分離機能で隔離された、ただのプロセス**。
> ホストから `ps` で見れば普通のプロセスとして見える。これがコンテナが軽くて速い理由。

つまり、カーネルが元々持っていた「プロセスごとに資源を割り振り、隔離する」仕事を一段強めたものが
コンテナの正体だ。だから[コンテナとは何か](/posts/aws-compute-explained/)を本当に理解するには、
その下のカーネルの役割を先に押さえておくと早い。

## まとめ

- カーネルはハードウェアとアプリの間に立つ**唯一の管理人**。アプリは直接ハードを触らない
- 仕事は大きく **プロセス／メモリ／ファイル／デバイス・ネットワーク** の4つ。共通点は「抽象化」
- アプリは **システムコール**経由でしかカーネルに頼めない。この境界が隔離を生む
- **VM はカーネルごと複製、コンテナはホストのカーネルを共有**。だからコンテナは軽い
- コンテナの隔離は **namespace（見える範囲）＋ cgroups（使える量）** というカーネル機能で実現される
