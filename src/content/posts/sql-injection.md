ユーザー入力を文字列連結でSQLに埋め込むと、入力次第でクエリの意味を書き換えられる——これがSQLインジェクション。
結論を先に言うと、**SQLのコードと、入力データを混ぜて組み立てない**。プレースホルダで両者を分離すれば、ほとんどの注意点はこの一文から導出できる。

## 攻撃の仕組み — なぜ入力でクエリが変わるのか

ログイン処理で、入力をそのまま文字列に埋め込んで [SQL](/posts/rdbms-basics/) を組み立てたとする。

```go
// 危険。絶対にやってはいけない
query := "SELECT * FROM users WHERE name = '" + name + "' AND pass = '" + pass + "'"
```

`name` に `alice' --` が入ると、組み上がるSQLはこうなる。

```sql
SELECT * FROM users WHERE name = 'alice' --' AND pass = '...'
```

`--` 以降はコメント扱いになり、**パスワード判定ごと消える**。`' OR '1'='1` を入れれば常に真になり、
テーブル全件が返る。DBの権限次第では `; DROP TABLE users; --` のように別の文を追加され、破壊やデータ流出にもつながる。

> ⚠️ 問題の本質は「入力に危ない文字が含まれること」ではなく、**入力（データ）がクエリ（コード）として解釈されてしまう**こと。
> だから「危ない文字を消す」発想（エスケープの自作）は穴が残りやすく、根本解決にならない。

## 対策の原則 — コードとデータを分離する

唯一の確実な原則は、**SQL文の構造を先に確定させ、入力は後から「値」としてだけ渡す**こと。
値がどんな文字列でも、それは常にデータであってSQLの一部にはならない、という状態を作る。

| やり方 | コードとデータ | 安全性 |
| --- | --- | --- |
| 文字列連結（`+` / `fmt.Sprintf`） | 混ざる | 危険 |
| 自作エスケープ | 混ざったまま加工 | 抜けが出やすい |
| **プレースホルダ／パラメータ化** | 分離される | 安全（基本形） |

この「分離」を言語・ライブラリのレベルで提供する仕組みが、プレースホルダ（`?` や `$1`）とプリペアドステートメント。

## Goでの防御 — database/sql のプレースホルダ

Goの標準ライブラリ `database/sql` は、クエリと引数を別々に渡すAPIになっている。**SQLに値を埋め込まず、`?` を置いて引数で渡す**だけでよい。

```go
// 安全。name は値としてだけ渡り、SQLの一部にはならない
row := db.QueryRow(
    "SELECT id, name FROM users WHERE name = ? AND pass = ?",
    name, pass,
)
var id int
var got string
if err := row.Scan(&id, &got); err != nil {
    // sql.ErrNoRows なら該当なし
}
```

プレースホルダ記法はドライバによって異なる（MySQLは `?`、PostgreSQL(`pq`/`pgx`)は `$1, $2`）。
`Query` / `QueryRow` / `Exec` はいずれも同じ形で引数を受け取る。

```go
// 複数行取得も同じ。値は必ず引数で
rows, err := db.Query("SELECT id FROM users WHERE age >= ?", minAge)
```

同じクエリを繰り返すなら、プリペアドステートメントで文を一度だけ準備して使い回せる（分離は同様に保たれる）。

```go
stmt, err := db.Prepare("SELECT id FROM users WHERE name = ?")
if err != nil { /* ... */ }
defer stmt.Close()

row := stmt.QueryRow(userName) // 呼ぶたびに値だけ差し替わる
```

> ⚠️ `fmt.Sprintf` でSQLを組み立てて `db.Query(built)` に渡すのは、連結と同じで**危険**。
> プレースホルダで渡せない部分（テーブル名・列名・`ORDER BY` の向き）は、そもそも入力から作らず、
> **許可リスト**（想定される値だけを switch で受ける）で決める。

> 🧭 C#ではこう／Goではこう: ADO.NET の `SqlCommand` に `Parameters.AddWithValue("@name", name)` で渡すのや、
> Entity Framework の LINQ が自動でパラメータ化するのと同じ考え方。Goでは `?` プレースホルダ＋引数がその役割を果たす。

## ORMを使ってもゼロにはならない

`database/sql` を直接使わず、GORM や sqlc などを挟むとクエリ組み立てが自動でパラメータ化され、事故は大きく減る。
ただし**生SQLを書く抜け道**（`Raw`, `Exec`, 文字列でのWHERE指定）は残り、そこに入力を連結すれば結局インジェクションになる。

> 💡 ORMは「デフォルトが安全」なだけで免罪符ではない。生SQLを書く箇所だけ、レビューで重点的に見る。
> 入力を組み立てに使っていないか（`?`/引数になっているか）が唯一のチェックポイント。

## まとめ

- SQLインジェクションの本質は、**入力（データ）がSQL（コード）として解釈される**こと
- 対策の原則はただ一つ、**コードとデータを分離する**。エスケープの自作に頼らない
- Goでは `database/sql` の**プレースホルダ**（`?` / `$1`）に値を引数で渡す。`fmt.Sprintf` でSQLを組まない
- 繰り返すクエリは**プリペアドステートメント**で準備して使い回す
- 値にできない部分（テーブル名・列名）は入力からではなく**許可リスト**で決める
- ORMは事故を減らすが、**生SQLの抜け道**は残る。そこだけは重点レビュー
