> 📚 シリーズ「Linux をちゃんと使う」(2 / 6)

Linux 操作の土台になる基本コマンドと、それらを**繋いで使う**パイプ・リダイレクト。
名前は知っていても手が止まりがちなところを、使える形で整理する。

## ファイルを見る・動かす

まずは日常の数個。これだけで大半の操作が回る。

| コマンド | 何をする | よく使う形 |
| --- | --- | --- |
| `ls` | 一覧 | `ls -la`（隠しファイル＋詳細） |
| `cat` | 中身を表示 | `cat app.log` |
| `cp` | コピー | `cp -r src/ dst/`（ディレクトリは `-r`） |
| `mv` | 移動・改名 | `mv old.txt new.txt` |
| `rm` | 削除 | `rm file` / `rm -r dir`（ディレクトリ） |

> ⚠️ `rm -rf` は確認なしで再帰削除する。パスを取り違えると戻せない。**実行前にパスを声に出して確認**。

## 探す — grep と find

「中身で探す」のが `grep`、「ファイルそのものを探す」のが `find`。役割が違う。

```bash
# ファイルの中身から "error" を含む行を探す（-i 大小無視, -n 行番号, -r 再帰）
grep -rin "error" /var/log/

# 名前や条件でファイルを探す
find . -name "*.go"            # 拡張子 .go
find /tmp -type f -mtime +7    # /tmp の7日より古いファイル
```

> 💡 `grep=中身検索`／`find=ファイル検索` と覚える。両者は後述のパイプで組み合わせると強い。

## パイプ — 小さなコマンドを繋ぐ

`|`（パイプ）は、**前のコマンドの出力を次のコマンドの入力に渡す**。
Unix の「小さな道具を繋いで大きな仕事をする」思想の核。

```bash
# プロセス一覧から nginx の行だけ抜く
ps aux | grep nginx

# ログのエラー行を数える
grep "ERROR" app.log | wc -l

# ディスク使用量の上位5件
du -sh * | sort -rh | head -5
```

> 🧭 PowerShell のパイプにも似ているが、PS が**オブジェクト**を流すのに対し、Unix のパイプは
> **テキスト（バイト列）**を流す。だから各コマンドはテキスト整形（grep/sort/awk）に寄っている。

## リダイレクト — 出力先を付け替える

`>` 系は[出力ストリーム](/posts/stdin-stdout-stderr/)の行き先を変える。パイプが「次のコマンドへ」なら、
リダイレクトは「ファイルへ」。

```bash
echo "hello" > out.txt     # 上書き保存
echo "world" >> out.txt    # 追記
./app > out.txt 2>&1       # 標準出力と標準エラーを両方 out.txt へ
./app 2>/dev/null          # エラーを捨てる
```

`>` は上書き、`>>` は追記。`2>&1` の意味と順序の注意は[標準入出力の記事](/posts/stdin-stdout-stderr/)で詳しく扱った。

## まとめ

- 日常操作は `ls / cat / cp / mv / rm` の数個でほぼ回る（`rm -rf` だけは慎重に）
- **`grep`＝中身検索、`find`＝ファイル検索**。役割を分けて覚える
- **パイプ `|`** は出力を次のコマンドへ。小さな道具を繋ぐのが Unix の作法
- **リダイレクト** は `>`上書き・`>>`追記・`2>&1`で出力先をファイルへ
- パイプ＋grep の組み合わせ（`ps aux | grep ...`）は調査の定番
