「登録は多いのに課金が伸びない」——どこで人が抜けているのか。それを可視化するのが **ファネル分析**。
ユーザーが目標（購入・登録・課金）に至る一連のステップを漏斗に見立て、各段で**何人残り何人抜けたか**を測る。
**離脱分析**はその「ステップ間の落ち」を見ること —— 表裏一体だ。ステップの定義・コンバージョン率の測り方・
数え方・そして間違えやすい設計上の勘所までを押さえる。

## ファネルとは — ステップの列に見立てる

ファネルは「目標までの**順序のあるステップ**」の列。各段は必ず**計測できるイベント**で表す。

```text
訪問 ──▶ カート投入 ──▶ 購入手続き ──▶ 購入完了
1000       400            250            180     ← 各段に到達した人数
```

- 上から下へ進むほど人数は減る（だから「漏斗」）。**減り方のどこが急か**が知りたいこと
- 各ステップは「`page_view` した」「`add_to_cart` した」のように**明確なイベント**で定義する
- ステップの順序を固定するか（strict）、任意順でも通過と見なすかを先に決める

> 💡 ファネルは「作る対象」の分析にも使える。オンボーディングの各画面、フォームの各項目、
> API の各呼び出し段 —— **順序があって離脱が起きる所**はすべてファネルで見られる。

## コンバージョン率と離脱率 — 全体とステップ間を分ける

数字は2種類ある。混同すると原因を見誤る。

| ステップ | 到達人数 | 全体通過率 | ステップ間通過率 | 離脱率 |
| --- | --- | --- | --- | --- |
| 訪問 | 1000 | 100% | — | — |
| カート投入 | 400 | 40% | 40% | 60% |
| 購入手続き | 250 | 25% | 62.5% | 37.5% |
| 購入完了 | 180 | 18% | 72% | 28% |

- **全体コンバージョン率** = 最終段 ÷ 最初段（例では 18%）。事業KPIとして見る数字
- **ステップ間コンバージョン率** = その段 ÷ 一つ前の段。**離脱率 = 1 − ステップ間通過率**
- 「一番落ちる所」は**ステップ間**で探す。上の例なら訪問→カート投入の **60% 離脱**が最大のボトルネック

> ⚠️ 全体通過率だけ見ると「購入完了18%が低い」と最終段を責めがち。だが真の穴は最初の一段。
> **改善はステップ間で一番落ちる1箇所に絞る**のが定石。

## どう数えるか — イベントログから出す

元データは「誰が・いつ・何をしたか」のイベントログ。ここからステップ到達**ユーザー数**を数える。

```sql
-- 各ステップに到達したユニークユーザー数（同一ユーザーの重複はDISTINCTで排除）
SELECT
  COUNT(DISTINCT CASE WHEN event = 'visit'    THEN user_id END) AS s1_visit,
  COUNT(DISTINCT CASE WHEN event = 'add_cart' THEN user_id END) AS s2_cart,
  COUNT(DISTINCT CASE WHEN event = 'checkout' THEN user_id END) AS s3_checkout,
  COUNT(DISTINCT CASE WHEN event = 'purchase' THEN user_id END) AS s4_purchase
FROM events
WHERE ts >= '2026-07-01' AND ts < '2026-07-08';
```

- 数えるのは**行数ではなくユニークユーザー数**（`COUNT(DISTINCT user_id)`）。1人の連打で膨らませない
- 厳密には「前の段を踏んだ人だけを次段の母数にする」が、まずは上のような**到達人数の並び**で十分見える
- 集計は母集団が大きいほど**DB側（SQL）で数える**。手元に全件引いてから数えない

> 🧭 C# の LINQ でも `GroupBy` / `Select(g => g.DistinctBy(...))` で同じ集計は書けるが、
> それは**データを全部メモリに載せてから**の話。数万行を超えるならクエリを DB に投げる方が速い。
> [SQL の基本](/posts/sql-basics/)の `GROUP BY` と同じ発想を集計側に寄せる。

## 間違えやすい設計上の勘所

ファネルは「数え方」で結論が変わる。ここを外すと数字が嘘をつく。

- **コホートを固定する**: 「7/1〜7/7に訪問した人」を最後まで**同じ母集団で追う**。段ごとに別の人を数えない
- **分母をどこに置くか**: 全体率（最初段基準）か、直前段基準か。**表に両方載せる**と誤読が減る
- **時間窓**: どこまでを「通過」と見なすか。同一セッション内か、7日以内か。窓が変われば率も変わる
- **セグメントで割る**: 平均は嘘をつく。**モバイル/PC・流入元・新規/既存**で割ると、全体では見えない穴が出る
- **相関≠因果**: 離脱が多い＝悪い、とは限らない。**あえて絞る段**（有料の壁など）は落ちて当然
- **生存者バイアス**: 最終段まで来た人だけ見て語らない。抜けた人にこそボトルネックの手がかりがある

> ⭐ 「どこで落ちるか（量的）」の次は必ず「**なぜ落ちるか（質的）**」。SQL でボトルネック段を1つ特定し、
> そこだけセッション録画・アンケート・A/B で深掘りする。定量→定性の順が最短で効く。

## まとめ

- ファネル分析は**目標までの順序あるステップ**で各段の残存を測る。離脱分析はその**ステップ間の落ち**を見ること（表裏一体）
- 数字は**全体コンバージョン率**と**ステップ間コンバージョン率**を分ける。ボトルネックは**ステップ間**で探す
- 元はイベントログ。**`COUNT(DISTINCT user_id)`** でユニークに数え、集計は DB 側に寄せる
- **コホート固定・分母・時間窓・セグメント**を決めてから数える。ここで結論が変わる
- 量的に「どこで」を1箇所に絞り、質的に「なぜ」を深掘る。**定量→定性**が最短
