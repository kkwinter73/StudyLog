> 📚 シリーズ「Goで学ぶアーキテクチャとテスト」(3 / 7)

Robert C. Martin（Uncle Bob）が提唱した設計原則。あの「同心円」の図が有名だが、
本質はたった一つのルールにある。

## 同心円の構造

外側ほど「詳細（変わりやすい）」、内側ほど「方針（変わりにくい）」。

```text
┌─ Frameworks & Drivers（最外層）──────────────┐
│  Web フレームワーク, DB, 外部 API             │
│  ┌─ Interface Adapters ───────────────┐     │
│  │  Controller, Gateway, Presenter     │     │
│  │  ┌─ Application Business Rules ─┐    │     │
│  │  │  Use Cases（ユースケース）    │    │     │
│  │  │  ┌─ Enterprise Rules ──┐    │    │     │
│  │  │  │  Entities（ドメイン）  │    │    │     │
│  │  │  └────────────────────┘    │    │     │
│  │  └────────────────────────────┘    │     │
│  └─────────────────────────────────────┘     │
└───────────────────────────────────────────────┘
```

## 依存性のルール（最重要）

> ⭐ **依存は必ず「外 → 内」の一方向だけ。内側は外側の存在を知らない。**

- Entity は UseCase を知らない
- UseCase は Controller を知らない
- Controller は Framework を知ってよい（外側だから）

内側のコードに外側の import が現れたら、それは設計違反。この一点さえ守れば、円の数や名前は本質ではない。

## 各層の責務

```text
① Entity（ドメインモデル）
   ビジネスルールそのもの。最も変わりにくい。何にも依存しない。
   例: User, Order とそのバリデーション

② Use Case（ユースケース）
   アプリ固有のルール。「登録する」「確定する」等の操作。
   Entity を使い、Repository などのインターフェースを定義する。

③ Interface Adapter
   Use Case と外部をつなぐ変換層。
   Controller(入力変換) / Presenter(出力変換) / Gateway(DB変換)

④ Frameworks & Drivers
   Web サーバー・DB ドライバ・外部 API クライアントなど具体技術。
```

## 何がうれしいのか（MySQL → PostgreSQL 移行）

| 層 | クリーンアーキテクチャの場合 |
| --- | --- |
| ① Entity | 変更なし |
| ② Use Case | 変更なし（Repository インターフェースはそのまま） |
| ③ Adapter | Repository 実装を PostgreSQL 用に書き換え |
| ④ Framework | ドライバを差し替え |

変更は外側2層だけ。ロジックには一切触らない。もしロジックに SQL が直書きされていたら、全部書き直し＋テストやり直しになる。

## よくある誤解

> ⚠️ 4つの円は「絶対4層に割れ」という意味ではない。

- ❌ **4層きっちり分けないとダメ** → 小さいアプリで4層は過剰。原則を理解して粒度を選ぶ。
- ❌ **フォルダ構成 = アーキテクチャ** → フォルダを分けても依存の方向が守られなければ無意味。
- ❌ **Entity = DB のテーブルモデル** → Entity はビジネスルールを持つドメインオブジェクト。ORM のモデルは外側に置く。
- ❌ **全プロジェクトに適用すべき** → CRUD だけの API なら過剰設計。ロジックが複雑なときに価値が出る。

## まとめ

- 同心円は「外＝詳細／内＝方針」を表す
- 本質は1つ ——「依存は外→内、内側は外側を知らない」
- Entity / UseCase / Adapter / Framework の4責務
- フォルダ分けや層数より、依存の方向を守ることが重要
