> 📚 シリーズ「Webを支える通信」(5 / 10)

[HTTPメッセージ](/posts/http-message-anatomy/)の中身は世代を通じてほぼ同じ。変わったのは**運び方**だ。
HTTP/1.1・2・3を「何がボトルネックで、それをどう解いたか」で辿ると、
`h2` や `HTTP/3` という表示の意味が腑に落ちる。[TCPとUDP](/posts/tcp-vs-udp/)の話がここで効いてくる。

## HTTP/1.1 — 1本の道に1台ずつ

長く主流だった1.1は、1つのTCP接続で**リクエストを1件ずつ順番に**処理する。
前のレスポンスが返るまで次を送れない（＝先頭が詰まると後ろが全部待つ）。

```text
接続1: [req A]──[res A]──[req B]──[res B]  Aが遅いとBが待つ
```

- 回避策はブラウザが**接続を複数張る**（同時6本など）ことだったが、接続ごとに握手のコストがかかる
- ヘッダは毎回テキストで丸ごと送られ、`Cookie` など重複が多い

> ⚠️ この「1本の道で先頭が詰まると後続が待つ」問題を **HTTP レベルの Head-of-Line ブロッキング** と呼ぶ。2・3の主目的はこれの解消。

## HTTP/2 — 1本の道を多重化する

2は1つのTCP接続の中に**ストリーム**という仮想の車線を複数作り、リクエストを**並行**に流す（多重化）。

| 工夫 | 効果 |
| --- | --- |
| 多重化（ストリーム） | 1接続で複数リクエストを同時進行。接続を大量に張らずに済む |
| バイナリフレーム | テキストでなく固定形式で高速・確実にパース |
| ヘッダ圧縮（HPACK） | 重複ヘッダを圧縮して転送量を削減 |
| サーバプッシュ※ | 要求前にリソースを先送り（※後に非推奨化） |

これでHTTP層の行列は解けた。だが**土台のTCPが1本**なので、パケットが1つ欠けるとTCPの再送待ちで
その接続の全ストリームが止まる — 問題が**TCP層のHead-of-Lineブロッキング**へ移っただけだった。

> 💡 開発者ツールのプロトコル表示で `h2` と出ればHTTP/2。多くのサイトはHTTPSと同時に有効化している。

## HTTP/3 — 土台をUDPに載せ替える

3は残ったTCP層の詰まりを、**トランスポートごと変える**ことで解いた。TCPをやめ、
[UDP](/posts/tcp-vs-udp/)の上に **QUIC** という新プロトコルを載せる。

- QUICが順序保証・再送・輻輳制御を**ストリーム単位**で持つ → 1つ欠けても他のストリームは止まらない
- TCP＋TLSで2段だった握手を**1回に統合** → 接続確立が速い（[TLSの話は次回](/posts/https-tls-explained/)）
- 接続をIPでなく接続IDで識別 → Wi-Fi↔モバイル切替でも接続が切れにくい

> 🧭 C#/.NET も Go も、標準の`HttpClient`/`net/http`が2までは自動対応し、3は設定で有効化する段階（環境依存）。
> アプリのコードは基本そのまま、下の運び方だけが速くなるのがこの進化の嬉しさ。

## 世代の対比

| | HTTP/1.1 | HTTP/2 | HTTP/3 |
| --- | --- | --- | --- |
| 土台 | TCP | TCP | UDP + QUIC |
| 多重化 | ×（接続を複数） | ○（ストリーム） | ○（ストリーム） |
| HoLブロッキング | HTTP層で発生 | TCP層に残る | ほぼ解消 |
| ヘッダ | テキスト | HPACK圧縮 | QPACK圧縮 |

## まとめ

- HTTPの世代交代はメッセージの中身でなく**運び方（トランスポート）**の進化
- **1.1**: 1接続1件ずつ → 先頭が詰まると後続が待つ（HoLブロッキング）
- **2**: 1接続を**多重化**＋バイナリ＋ヘッダ圧縮。ただし詰まりがTCP層に移った
- **3**: TCPをやめ**UDP＋QUIC**へ。ストリーム単位の信頼性で詰まりを解消、握手も統合
- アプリのコードは概ね不変で、下の層だけが速くなる
