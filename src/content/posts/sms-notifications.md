[push通知](/posts/push-notifications/)はアプリを入れた端末にしか届かない。対して **SMS通知**は
**アプリ不要でほぼ全ての携帯に届く**——だから OTP（ワンタイムパスワード）や重要連絡の定番だ。
代わりに **1通ごとにお金がかかる**のが決定的な違い。仕組み・送り方・そして「1通いくら」を決める
**コスト構造**、到達性と規制、push との使い分けまでを一気に押さえる。

## 仕組み — アプリからキャリア網まで

自分のサーバから端末へ直接は送れない。間に **SMS配信事業者（CPaaS/アグリゲータ）** が挟まる四者リレー。

```text
①アプリサーバ ─API─▶ ②SMS配信API ─接続─▶ ③キャリア(SMSC) ─▶ ④端末
 (自分のbackend)     (Twilio等)         (各社の交換機)
```

- **② 配信事業者**（Twilio / Vonage / AWS SNS 等）が、世界中のキャリアとの接続を**束ねて代行**する
- **③ SMSC**（Short Message Service Center）がキャリア側でメッセージを蓄積・転送する
- 自前でキャリアと直接契約するのは重い。だから CPaaS を経由するのが実務の定番

> 💡 push通知の[四者モデル](/posts/push-notifications/)とそっくり。違いは相手が「OS の push service」でなく
> 「**キャリアの電話網**」であること。だからアプリの有無に関係なく、電話番号さえあれば届く。

## 送り方 — API で1通投げる

送信自体は「宛先番号・送信元・本文を配信APIに POST」するだけ。Go で Twilio 風の例:

```go
form := url.Values{
    "To":   {"+819012345678"}, // E.164 形式（国番号付き）
    "From": {"+815012345678"}, // 送信元番号 or 送信者ID
    "Body": {"認証コードは 123456 です"},
}
resp, err := http.PostForm(endpoint, form) // 実際は Basic 認証などを付与
```

- 宛先は **E.164**（`+81…` の国際形式）で渡すのが基本
- 送った時点では「受け付けた」だけ。**本当に届いたか**は配信事業者が **Webhook（配信レポート/DLR）** で後追い通知する
- 到達確認は[Webhook の受け口](/posts/webhook-implementation/)と同じ発想で、非同期に受けて突合する

> 🧭 .NET なら Twilio の公式 SDK か **Azure Communication Services（SMS）** が使える。Go は SDK か HTTP 直叩き。
> どちらでも「送信＝受付、到達＝Webhook で後追い」の二段構えは共通。

## コスト構造 — 「1通いくら」を決めるもの

SMS の最大の特徴は **従量課金**。何が単価を決めるかを知らないと請求で驚く。

| 要素 | 内容 |
| --- | --- |
| **セグメント数** | 半角160字（日本語や絵文字は**70字**）で1通。超えると分割され**通数が増える** |
| **宛先の国・キャリア** | 単価は国ごとに大きく違う。日本は US 等より高め |
| **番号の種別** | 長番号／短縮番号（ショートコード）／送信者ID（英数名）で費用も到達性も違う |
| **登録料** | 米 A2P 10DLC のように、事業者・用途の**事前登録に初期費/月額**がかかる制度がある |

- **本文の長さ**が効く。日本語は70字で1セグメントなので、長文は**そのまま通数（＝金額）増**
- **送信者ID**（英数の差出人名）は日本や米国では制約が強く、使えない/要審査のことが多い
- ショートコードは高スループットだが**初期費・月額が高い**。OTP大量送信など用途次第

> ⚠️ 「1通◯円」だけ見て見積もると外す。**セグメント分割・国別単価・番号や登録の固定費**を合わせて
> 初めて実コストになる。特に**日本語本文は70字**で刻まれる点を忘れない。

## 到達性と規制 — 送れても届くとは限らない

SMS は電話網ゆえに**規制が厳しい**。守らないとブロックや番号停止になる。

- **オプトイン必須**: 同意なく送らない。**STOP 等での配信停止（オプトアウト）**受付も用意する
- **送信者の明示**: なりすまし・フィッシング対策で、差出人や用途の登録・審査が各国で強化されている
- **フィルタリング**: URL の多用・スパム的文面はキャリアに**無言でブロック**されうる（送信は成功扱いでも届かない）
- **時間帯・頻度**: 深夜送信や連投は規制・ユーザー体験の両面で避ける

> ⭐ SMS は「送信成功」と「到達」が別。DLR（配信レポート）を受けて**実到達を計測**し、
> 届かない番号・キャリアを可視化する。push と同じく**配信は保証されない**前提で組む。

## push通知との使い分け

無料で何度でも押せる push と、確実だが有料の SMS。**役割で選ぶ**。

| | SMS | push通知 |
| --- | --- | --- |
| 前提 | 電話番号のみ（アプリ不要） | アプリ/ブラウザの登録が必要 |
| コスト | **1通課金** | ほぼ無料 |
| 到達 | ほぼ全端末・開封率高い | アプリ未導入には届かない |
| 向く用途 | OTP・重要連絡・本人確認 | アプリ内のお知らせ・再訪促進 |

- **お金を払ってでも確実に届けたい1回**（認証コード、決済確認）は SMS
- **無料で何度も接点を持ちたい**（キャンペーン、更新通知）は push
- 実務では**併用**（重要は SMS、日常は push、SMS が届かなければメールへフォールバック）が定石

## まとめ

- SMS通知は**アプリ→配信API→キャリア網→端末**の四者リレー。電話番号だけで**アプリ不要**で届く
- 送信は API に POST するだけ。**到達は Webhook（DLR）で後追い**、送信成功≠到達
- コストは**従量**。**セグメント（日本語70字）・国別単価・番号種別・登録料**が単価を決める
- 電話網ゆえ**規制が厳しい**。オプトイン・オプトアウト・送信者登録を守る
- **確実な1回は SMS、無料の反復は push**。用途で使い分け、フォールバックを設計する
