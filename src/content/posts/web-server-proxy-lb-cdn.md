> 📚 シリーズ「Webを支える通信」(9 / 10)

[リクエストの旅](/posts/how-a-web-request-travels/)ではブラウザとアプリを直結で描いたが、実運用ではその間に
**いくつもの箱**が並ぶ。Webサーバ・リバースプロキシ・ロードバランサ・CDN — 名前は違えど、
どれも「アプリにやらせなくていい仕事を手前で肩代わりする」中継役だ。役割で切り分ければ怖くない。シリーズ最終回。

## 全体像 — リクエストが通る順

```text
ブラウザ
  │  （近ければ）
  ▼
CDN ──── 静的ファイルはここで返す（世界中の拠点にキャッシュ）
  │  （動的なものだけ奥へ）
  ▼
ロードバランサ ── TLS終端＋複数台へ振り分け
  │
  ▼
リバースプロキシ / Webサーバ ── ルーティング・静的配信
  │
  ▼
アプリ（C# / Go）── 本来のビジネスロジックだけ
```

奥へ行くほど「そのアプリにしかできない仕事」に絞られ、手前ほど「みんなの共通作業」を担う。

## Webサーバとリバースプロキシ

- **Webサーバ**（Nginx / Apache 等）: HTTPを受け、**静的ファイル**（HTML/CSS/画像）をそのまま返すのが本業
- **リバースプロキシ**: 受けたリクエストを**背後のアプリへ中継**し、返ってきた応答をクライアントへ返す代理人

同じソフト（Nginx等）が両方を兼ねることが多い。動的な処理はアプリへ回し、静的配信やTLS終端は手前で済ませる。

> 💡 「フォワードプロキシ」がクライアント側に立って**出ていく**通信を代理するのに対し、
> リバースプロキシはサーバ側に立って**入ってくる**通信を代理する。向きが逆。

> 🧭 C#のKestrelやGoの`net/http`を**そのまま外部公開せず**、前段にNginx/ALBを置くのは定番。
> アプリは信頼できる内側の通信だけ相手にし、TLSや攻撃の矢面は手前が引き受ける（[認証境界の話](/posts/auth-boundary-edge-vs-app/)に通じる）。

## ロードバランサ — 複数台へ振り分ける

1台では捌けない・落ちると困るので、同じアプリを**複数台**並べて前に**ロードバランサ(LB)**を置く。

- リクエストを空いている台へ分散（ラウンドロビン等）
- **ヘルスチェック**で不調な台を自動で外す → 1台落ちてもサービスは継続
- ここで**TLSを終端**し、背後へは平文で流す構成が多い（[⑥TLS](/posts/https-tls-explained/)）

前回⑧までに出た **L4/L7** がここで効く（[①階層モデル](/posts/network-layers-tcpip-osi/)）:

| 種別 | 見る層 | 振り分けの判断材料 |
| --- | --- | --- |
| L4 LB | トランスポート層 | IP / ポートだけ（中身は見ない・速い） |
| L7 LB | アプリケーション層 | HTTPのパス・ヘッダ（`/api`は別群へ 等） |

> ⚠️ 複数台に分散すると、[⑦のセッション](/posts/cookie-session-token/)を各台のメモリに持つ設計は破綻する
> （次のリクエストが別の台に行く）。だからセッションは共有ストアへ、あるいはトークン方式にする。

## CDN — ユーザーの近くにキャッシュ

**CDN**は世界中に配置した拠点(エッジ)に**静的コンテンツをキャッシュ**し、ユーザーに一番近い拠点から返す。

- 物理的に近い＝速い。かつ大量アクセスを**エッジで吸収**してオリジン(自前サーバ)の負荷を下げる
- キャッシュの可否・期間は[④で触れた](/posts/http-message-anatomy/)`Cache-Control`/`ETag`で制御する
- 動的な応答や個人別データはキャッシュせず奥のアプリへ通す

> 💡 「静的は手前(CDN/Webサーバ)で、動的は奥(アプリ)で」。この住み分けが速さとスケールの基本形。

## まとめ

- リクエストは **CDN → LB → プロキシ/Webサーバ → アプリ** と、手前ほど共通作業を肩代わりする箱を通る
- **Webサーバ/リバースプロキシ**: 静的配信・TLS終端・背後アプリへの中継。アプリを直接公開しない
- **ロードバランサ**: 複数台へ分散＋ヘルスチェックで可用性を確保。**L4は速さ/L7は賢さ**
- **CDN**: 静的をユーザー近くのエッジにキャッシュし、速度とオリジン負荷を改善
- 分散環境では**セッションを各台のメモリに持たない**（共有ストア/トークンへ）

ここまでで「リクエストが通る箱」は揃った。最終回は、HTTPの一方通行を超える双方向・push通信を扱う。
