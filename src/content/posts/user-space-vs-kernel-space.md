OS の上では、コードが動く世界が **ユーザー空間** と **カーネル空間** の2つに分かれている。
[カーネルの役割](/posts/kernel-role-for-containers/)では1節だけ触れたこの境界を、ここでは正面から扱う。
なぜ分けるのか・何が違うのか・どう行き来するのかの3点で理解する。

## 2つの空間 — 何がどう違うのか

同じメモリ上の世界だが、**権限とできること**がまるで違う。

| | ユーザー空間 | カーネル空間 |
| --- | --- | --- |
| 誰が動く | アプリ（あなたの Go バイナリ等） | カーネル（OS の中核）とドライバ |
| 権限 | 制限あり。ハードウェアに直接触れない | 全権。CPU・メモリ・機器を直接操作できる |
| できること | 計算・自分のメモリの読み書き | プロセス管理・メモリ割当・デバイス操作 |
| 失敗したら | そのプロセスが落ちるだけ | OS 全体が巻き込まれうる（カーネルパニック） |

> 💡 CPU 自体に「特権モード／非特権モード」の区別がハードウェアとして備わっている。
> ユーザー空間のコードは非特権モードで動かされ、ハードを触る命令は実行できないようになっている。

## なぜ分けるのか — 保護と安定性

理由はシンプルで、**1つの暴走したアプリに OS 全体を巻き込ませない**ため。

- アプリにハードを直接触らせると、バグった1本がメモリやディスクを壊し、全体が道連れになる
- だからアプリは非特権モードに閉じ込め、危険な操作はすべてカーネルに**代行を頼む**形にする
- カーネルは頼まれた操作が妥当か検査してから実行する。これが**保護の壁**になる

> ⭐ 「アプリは直接ハードを触らず、カーネルに頼む」という大原則は、この権限の壁から来ている。
> 壁があるからこそ、アプリが1つ落ちても OS とほかのアプリは生き残れる。

## 境界をまたぐ — システムコールとモード切替

アプリがファイル読み込みやネットワーク通信などハード絡みの操作をしたいとき、
**システムコール**という決められた窓口からカーネルに頼む。このとき CPU は
**ユーザーモード → カーネルモードへ切り替わり（モード切替）**、処理が終わると戻ってくる。

```text
アプリ:   read() を呼ぶ
   │  ── モード切替（ユーザー → カーネル）──▶
カーネル:  権限を確認し、ディスクから実際に読む
   │  ◀── モード切替（カーネル → ユーザー）──
アプリ:   読めたデータを受け取って続行
```

この往復には**わずかなコスト**がある。だから性能が要る場面では「システムコールの回数を減らす」
工夫（バッファリングでまとめて読み書きする等）が効いてくる。

> 🧭 C#/.NET の `FileStream` も Go の `bufio` も、内側でこの往復を**まとめて減らす**ための仕組み。
> 1バイトずつ read するより、4KB まとめて read する方が境界をまたぐ回数が減って速い。

## どこで効いてくるか

この2層構造を知っていると、いろいろな話が一段すっきり見える。

- **コンテナ**: ホストの1つのカーネル（カーネル空間）を共有し、アプリ側（ユーザー空間）だけを分離する。
  だから[コンテナは軽い](/posts/kernel-role-for-containers/)
- **クラッシュの範囲**: アプリのバグはそのプロセスで止まる。OS まで落ちるのは稀（＝カーネル側の問題）
- **性能チューニング**: 「遅い」原因がシステムコールの呼びすぎ、というのはよくある話

## まとめ

- 世界は **ユーザー空間（アプリ・制限あり）** と **カーネル空間（OS中核・全権）** の2つに分かれる
- CPU の特権/非特権モードという**ハードウェアの仕組み**で権限の壁が作られている
- 分ける理由は **保護と安定性** ——暴走したアプリに OS 全体を巻き込ませないため
- 境界は **システムコール**経由でまたぎ、その都度 **モード切替**が起きる（わずかなコスト）
- まとめ読み・まとめ書きで**境界をまたぐ回数を減らす**のが性能の定石

次にやること: システムコールを実際に覗く（Linux なら `strace ./app` でアプリが呼ぶシステムコールを観察できる）。
