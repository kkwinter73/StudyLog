監視カリキュラム (3/9・後編)。[前編](/posts/what-to-measure-metrics/)で「何を測るか」を決めた。今回は
それを**どう表現し、どう集めるか**。メトリクスの型（カウンタ/ゲージ）と収集方式（pull/push）は、
CloudWatch でも自前OSSでも共通の土台になる。

## メトリクスは時系列

`top` の数字は**今この瞬間の値**でしかなく、すぐ消える。これを**時刻つきで連続的に記録**すると、
後から振り返り・比較・傾向把握ができる。これが時系列メトリクス。

```text
timestamp            metric              value
2026-07-01T21:00:00  http_requests_total 15230
2026-07-01T21:00:15  http_requests_total 15288
2026-07-01T21:00:30  http_requests_total 15355
```

- 「点」を貯めて「線」にすることで、スパイクや右肩上がりが見える
- 保存先が時系列データベース（Prometheus / CloudWatch メトリクス等）

## カウンタとゲージ

メトリクスには大きく2つの型がある。混同すると集計を間違える。

| 型 | 動き | 例 |
| --- | --- | --- |
| **カウンタ(counter)** | 増える一方（再起動で0に戻る） | 累計リクエスト数・累計エラー数 |
| **ゲージ(gauge)** | 上下する（今の値） | CPU使用率・メモリ使用量・接続数 |

- **カウンタ**は累計。そのまま見ても意味が薄く、**変化率(rate)**にして「秒あたり何件」を見る
- **ゲージ**は今の瞬間の値。そのまま見れば意味がある

```promql
# カウンタは rate で「秒あたり」に変換して使う
rate(http_requests_total[1m])
```

> 💡 [前編](/posts/what-to-measure-metrics/)の Rate（秒あたりリクエスト）は、カウンタ
> `http_requests_total` に rate をかけて得る。カウンタとゲージの違いが分かると RED が計算できる。

## pull型とpush型

値を「誰が動かして集めるか」で2方式ある。ここは製品選びに直結する。

```text
pull型:  監視側 ──「今の値ちょうだい」──▶ 被監視側   （取りに行く）
push型:  被監視側 ──「今の値これです」──▶ 監視側     （送りつける）
```

| | pull型 | push型 |
| --- | --- | --- |
| 誰が起点 | 監視側が定期的に取りに行く | 被監視側が送る |
| 代表 | **Prometheus** | **CloudWatch**・StatsD |
| 相性 | 常駐サービス（居場所が分かる） | 短命ジョブ・多数の分散ソース |

- **pull**は「ターゲット一覧」を監視側が持ち、生きているか（取れるか）も同時に分かる
- **push**は送る側さえ届けばよく、居場所が変わるものや一瞬で終わる処理に強い

> ⚠️ この違いは後で効く。[Prometheus は pull](/posts/build-monitoring-oss/)（Step 3で自作）、
> **CloudWatch は push**（Step 4）。同じ「メトリクス収集」でも向きが逆、と頭に入れておく。

## どちらが良いというより特性

pull/push は優劣でなく**向き不向き**。居場所の分かる常駐サービスは pull が管理しやすく、
サーバーレスや短命ジョブ・AWSマネージド環境は push が自然。実務では**両方混在**することも多い。

## まとめ

- その瞬間の値は消える。**時刻つきで連続記録**するのが時系列メトリクス
- 型は2つ。**カウンタ（増える一方→rateで使う）**と**ゲージ（今の値）**
- 収集は **pull（取りに行く・Prometheus）** と **push（送る・CloudWatch）**
- pull は居場所の分かる常駐向き、push は短命・分散・マネージド向き。優劣でなく特性
- カウンタ×rate が前編の Rate につながる——型が分かると RED が計算できる
