開発・検証・本番をどう分けるか。雑に混ぜると「検証の操作が本番を壊す」事故につながる。
環境分離の段階と、同じ[IaC](/posts/iac-terraform-basics/)コードを環境差だけ変えて**DRYに使い回す**考え方を押さえる。

## なぜ環境を分けるのか

dev/staging/prod を分ける目的は、ひとことで言えば**事故の隔離**。

- 検証中の壊れた変更が、本番の利用者に影響しない
- 本番のデータ・権限・課金を、開発から触れないようにする
- 「どこまでが本番か」を曖昧にしない

## 分け方の段階

分離にはレベルがあり、強いほど安全だが管理は増える。

| レベル | やり方 | 隔離の強さ |
| --- | --- | --- |
| 弱 | 同一アカウント内で名前を分ける（prefixやタグ） | 弱（権限・課金は混在） |
| 中 | 同一アカウント内でネットワーク等を分離 | 中 |
| 強 | **環境ごとにアカウントを分ける** | 強（権限・課金・障害が独立） |

## アカウント分離の利点

環境ごとにアカウントを分けると、境界がはっきりする。

- **権限の隔離**: 本番アカウントへのアクセスを限られた人だけに絞れる
- **課金の分離**: コストが環境ごとに見える（dev のコストが本番に紛れない）
- **爆発半径が小さい**: 設定ミスやインシデントが他環境に波及しにくい

> 💡 「[最小権限](/posts/ecs-iam-roles/)」をアカウント単位でも効かせる発想。
> 本番を別アカウントにすれば、開発用の権限では本番に手が届かない。

## IaC を DRY に使い回す

環境ごとにコードを丸ごとコピーすると、修正が全環境に伝播せず破綻する。
**同じ[module](/posts/terraform-in-practice/)を、環境別の[変数](/posts/terraform-in-practice/)で呼び分ける**のが基本。

```hcl
# 共通の構成は module に。違いは変数だけ
module "app" {
  source        = "./modules/app"
  env           = "prod"            # 環境名
  instance_size = "large"          # 環境ごとに変える値だけ渡す
}
```

- 構成の本体は1か所（module）。**環境差は変数に追い出す**
- dev は小さく、prod は冗長化、のような違いを変数で表現する

> ⚠️ コピペで環境を増やすと、片方だけ直して**設定がdrift**する。「同じコード＋違う変数」を徹底する。

## まとめ

- 環境分離の目的は**事故の隔離**（検証が本番を壊さない）
- 分け方は弱→強: **同一アカウントで区切る → アカウントごとに分ける**
- **アカウント分離**は権限・課金・爆発半径を独立させられて強い
- IaC は**同じ module を環境別の変数で呼ぶ**ことで DRY に使い回す
- コピペ増殖は drift の元。「コード共通・変数だけ環境別」を守る
