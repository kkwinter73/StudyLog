`assertEqual(期待値, 実際)` と書くには**期待値が分かっている**必要がある。だが複雑な計算・検索・機械学習など、
**入力に対する正解を書けない**プログラムは多い。そこで使うのが **メタモルフィックテスト**。
個々の出力の正解ではなく、**入力を変えたときに出力が満たすべき「関係」**で正しさを確かめる手法だ。

## テストオラクル問題 — 期待値が書けないとき

テストの「期待値（＝正解）」を与えてくれる仕組みを**テストオラクル**と呼ぶ。これが無い／作れない状況が
**オラクル問題**だ。次のようなプログラムでよく起きる。

- 複雑な数値計算・シミュレーション（正解を手で計算できない）
- 検索エンジンやレコメンド（"正しい検索結果"を定義しづらい）
- 機械学習モデル（この入力の正しい予測は？と聞かれても厳密な正解が無い）
- レンダリングやコンパイラ（出力が巨大で1つずつ照合できない）

> ⭐ オラクルが無いと従来型の「入力→期待値」テストが書けない。だが**「正解そのもの」は分からなくても、
> 「入力をこう変えたら出力はこうあるべき」という関係なら分かる**——ここに突破口がある。

## メタモルフィック関係（MR） — 出力どうしの「守るべき関係」

**メタモルフィック関係（Metamorphic Relation, MR）** とは、正しいプログラムなら必ず成り立つ、
**複数の入力と出力の間の関係**。正解値を知らなくても検証できるのがポイント。

| 対象 | メタモルフィック関係(MR)の例 |
| --- | --- |
| 三角関数 `sin` | `sin(x)` と `sin(π − x)` は等しい |
| リストの合計 | 並び替えても合計は変わらない（順序不変） |
| 検索エンジン | 条件を足すと結果は増えない（"A" の件数 ≥ "A かつ B" の件数） |
| 最短経路 | ノードを1つ足しても、既存の最短経路は短くならない |
| ML 分類器 | 無関係な変化を加えても予測は変わらない（頑健性） |

> 💡 `sin(1.2345)` の正確な値は覚えていなくても、「`sin(x)=sin(π−x)` のはず」は言える。
> この**関係が破れたらバグ**、という形でテストできるのがメタモルフィックテストの発想だ。

## どうテストするか — source と follow-up

手順はシンプル。1つの入力（**source ケース**）と、それをMRに従って変換した入力（**follow-up ケース**）を作り、
両方の出力が**MRを満たすか**を見る。

```go
// MR: sin(x) == sin(π − x)  ——正解値を知らなくても成り立つべき関係
func TestSin_Metamorphic(t *testing.T) {
    for _, x := range []float64{0.1, 0.7, 1.2, 2.5} {
        src := MySin(x)                 // source ケース
        follow := MySin(math.Pi - x)    // follow-up ケース（入力を MR で変換）
        if math.Abs(src-follow) > 1e-9 { // 出力どうしの関係を検証
            t.Errorf("MR違反: sin(%v)=%v だが sin(π−%v)=%v", x, src, x, follow)
        }
    }
}
```

- **source → follow-up の入力変換**（`x` → `π − x`、リスト→並び替え、検索語に条件追加…）がMRの本体
- 検証するのは「期待値との一致」ではなく「**source と follow-up の出力の関係**」
- 入力は乱数で大量生成できる。**正解を用意せずに**多数のケースを回せる

> 🧭 これは**プロパティベーステスト（PBT）**の親戚。PBTが「任意の入力で成り立つ性質」を確かめるのに対し、
> MTは特に「**入力を変換した前後の出力関係**」に注目する。.NET なら FsCheck、Go なら rapid 等の PBT ツールが実装の足場になる。

## 何が嬉しく、何に注意か

- **嬉しい点**: オラクルが無くてもテストできる。乱数入力で自動化しやすく、"テストしにくい"領域（ML・数値計算）に効く
- **注意① MRの発見が肝**: 良いMRを見つけるには**対象ドメインの理解**が要る。ここが一番の頭の使いどころ
- **注意② 必要条件にすぎない**: MRが成り立っても正しさの**証明にはならない**。例えば `MySin` が常に0を返すバグは
  `sin(x)=sin(π−x)=0` を満たして**すり抜ける**。従来型テストと**併用**して穴を塞ぐ

## まとめ

- メタモルフィックテストは、正解が分からない入力を**出力どうしの関係（MR）**で検証する手法。**オラクル問題**への回答
- MRは「正しければ必ず成り立つ入力と出力の関係」。`sin(x)=sin(π−x)`、合計の順序不変、検索の単調性など
- 回し方は **source ケース → MRで変換した follow-up ケース → 出力の関係を検証**。正解を用意せず自動化できる
- 効くのは**ML・数値計算・検索**などテストしにくい領域。**プロパティベーステスト**の親戚
- 注意は「**良いMRの発見にドメイン知識が要る**」「MR通過は**必要条件で証明ではない**」。従来型テストと併用する
