監視カリキュラム (3/9・前編)。[前回](/posts/observe-server-by-hand/)は生の数字を手で読んだ。だが
サーバーが持つ数字は無数にある。**まず何を測ればいいのか**。この問いは CloudWatch でも自前OSSでも
同じ。定番の指標セット——4大シグナルと RED——を地図として押さえる。

## 無数の数字から何を選ぶか

全部を測ってダッシュボードに並べても、多すぎて「健全かどうか」が読めない。**少数の効く指標**に絞る。
先人が整理した「まずこれを見よ」という型がいくつかある。

## 4大シグナル（Golden Signals）

Google SRE 本が挙げる、ユーザー影響を捉える4つ。まずこれで「サービスは健全か」を見る。

| シグナル | 意味 | 例 |
| --- | --- | --- |
| レイテンシ(Latency) | 応答にかかる時間 | p50/p99 応答時間 |
| トラフィック(Traffic) | どれだけ来ているか | リクエスト数/秒 |
| エラー(Errors) | 失敗の割合 | 5xx 率・失敗数 |
| サチュレーション(Saturation) | 資源の逼迫度 | CPU/メモリ/キューの詰まり |

- 最初の3つ（レイテンシ・トラフィック・エラー）は**ユーザー体験**に直結
- サチュレーションは「あとどれだけ余裕があるか」。[前回のロードアベレージやavailable](/posts/observe-server-by-hand/)がこれ

> ⚠️ レイテンシは**平均でなく分布**で見る。平均が速くても p99（遅い方の1%）が悪いと、一部ユーザーは
> ひどい体験をしている。平均は異常を隠す。

## REDメソッド

Web API のような**リクエスト駆動サービス**向けに、Golden Signals を3つに絞ったのが RED。

| 指標 | 意味 |
| --- | --- |
| **R**ate | 秒あたりリクエスト数（＝トラフィック） |
| **E**rrors | 失敗したリクエスト数/率（＝エラー） |
| **D**uration | 処理にかかった時間（＝レイテンシ） |

- サービスごとに「Rate / Errors / Duration」を並べるだけで、健全性がひと目で分かる
- これが後の CloudWatch ダッシュボード（Step 5）に並べる指標の芯になる

> 💡 資源側には **USE メソッド**（Utilization / Saturation / Errors）という別の型がある。
> ざっくり **RED＝リクエスト（サービスの外から）**、**USE＝資源（サーバの中）**、と使い分ける。

## Goの業務システムに当てはめる

抽象論で終えず、自分たちのサービスの具体的な数字に落とす。例えば HTTP API なら:

```text
Rate     : http_requests_total を秒あたりに換算
Errors   : ステータス 5xx（と必要なら4xx）の割合
Duration : ハンドラの処理時間の分布（p50 / p95 / p99）
```

- エラーの判定は[HTTPステータスコード](/posts/http-status-codes/)が基準。5xx はサーバ起因、4xx はクライアント起因で分けて考える
- 「どの数字が RED に当たるか」を先に書き出しておくと、ダッシュボード設計が迷わない

## まとめ

- 無数の数字から**少数の効く指標**に絞る。型に沿うのが近道
- **4大シグナル**＝レイテンシ・トラフィック・エラー・サチュレーション（まず健全性）
- **REDメソッド**＝Rate/Errors/Duration。リクエスト駆動サービスの芯
- レイテンシは**分布(p99)**で見る。平均は異常を隠す
- 資源側は **USE**。RED（外から）と USE（中）を使い分ける
