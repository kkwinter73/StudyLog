「別のサイトから漏れた ID とパスワードの組を、そのままうちのログイン画面に流し込む」——これが
クレデンシャルスタッフィングだ。**パスワードの使い回しに便乗する**ので、こちらが漏洩していなくても
被害が出る。仕組みと防ぎ方を整理する。

## 攻撃の仕組み — 総当たりとは別物

攻撃者はどこか別のサービスで漏れた「ユーザー名＋パスワード」の巨大なリストを手に入れ、それを
**そのまま**自サービスのログインに投げる。人が同じパスワードを使い回しているので、一定割合で通る。

総当たり（ブルートフォース）と混同しがちだが、狙いも成功率も違う。

| | ブルートフォース | クレデンシャルスタッフィング |
| --- | --- | --- |
| 使うもの | ありうる文字列を機械的に生成 | 実在の漏洩済み ID＋PW の組 |
| 対象 | 1アカウントを深く掘る | 大量アカウントを浅く広く |
| 試行あたり成功率 | 極端に低い | 低いが**ゼロではない**（0.1〜数%） |
| 効く前提 | パスワードが弱い | パスワードを**使い回している** |

> ⚠️ 試行あたり 0.2% でも、100万組を投げれば約2000アカウントが陥落する。しかも「正しい組」を
> 試すだけなので、**1アカウントあたりの試行は1回**で済む。回数ベースのロックアウトだけでは止まらない。

## 対策の原則 — 「パスワードだけ」から降りる

根本原因は「パスワードという知識だけで認証している」こと。だから防御も、そこから一段ずらす。

- **MFA（多要素認証）が最強の単発対策**。パスワードが合っていても2つ目の要素が無ければ通らない。
  盗んだ組の価値をまとめて無効化できる
- **不審ログインを検知する**: 短時間の地理的にありえない移動（impossible travel）、見慣れない
  デバイスフィンガープリント、人間離れした速度・分散（ボット検知）
- **漏洩済みパスワードを弾く**: 既知の漏洩パスワード集合と照合し、使い回しをそもそも登録させない
- **パスワードレス / パスキー**: 共有の秘密（パスワード）自体を無くす。使い回しようがない

> ⭐ 優先順位は明快。まず MFA、次に漏洩パスワード照合。この2つで攻撃の経済性が大きく崩れる。

## 実務での防ぎ方（Go）— 漏洩パスワード照合

新規登録・パスワード変更のたびに「その候補が既知の漏洩リストにあるか」を照合する。悩ましいのは
**生パスワードを外部に送りたくない**こと。ここで使うのが **k-匿名性のレンジクエリ**だ。

- パスワードを SHA-1 でハッシュ化し、**先頭5文字（プレフィックス）だけ**を問い合わせる
- サーバはそのプレフィックスに一致する**候補ハッシュの後半をまとめて**返す
- 一致判定は**手元**で行う。サーバ側は完全なハッシュも生パスワードも受け取らない

```go
// 生パスワードは外に出さず、ハッシュのプレフィックス5文字だけを問い合わせる
func isBreached(password string) (bool, error) {
	sum := sha1.Sum([]byte(password))
	hash := strings.ToUpper(hex.EncodeToString(sum[:])) // 40文字
	prefix, suffix := hash[:5], hash[5:]

	resp, err := http.Get("https://api.example.com/range/" + prefix)
	if err != nil {
		return false, err
	}
	defer resp.Body.Close()

	// レスポンスは "SUFFIX:出現回数" の行が並ぶ。手元で後半を突き合わせる
	sc := bufio.NewScanner(resp.Body)
	for sc.Scan() {
		line, _, _ := strings.Cut(sc.Text(), ":")
		if strings.EqualFold(line, suffix) {
			return true, nil // 漏洩済み → 登録を拒否
		}
	}
	return false, sc.Err()
}
```

> 💡 これは「登録させない」ための照合。既存ユーザーには、ログイン成功時に同じ判定を回して
> 該当したらパスワード変更を促す、という運用が効く。

## 実務での防ぎ方（Go）— MFA を必須にする

漏洩照合はすり抜けを減らすが、使い回しをゼロにはできない。**最後の砦は MFA**。ログイン処理は
「パスワード検証」で終わらせず、2要素目の検証を**必ず経由する**状態機械にする。

```go
// パスワードが合っても、MFA未完了なら本セッションを発行しない
func login(w http.ResponseWriter, r *http.Request, u *User, pw string) {
	if !u.CheckPassword(pw) {
		http.Error(w, "invalid credentials", http.StatusUnauthorized)
		return
	}
	if u.MFAEnabled {
		// フルセッションではなく「MFA待ち」の一時トークンだけ発行
		issuePendingMFA(w, u.ID)
		return
	}
	issueSession(w, u.ID) // MFA無効ユーザーのみここへ（=有効化を促す対象）
}
```

- パスワード検証成功＝ログイン完了、にしないのが要点。MFA 未完了の間は**限定トークン**しか渡さない
- 総当たり対策の[ログインスロットリング](/posts/brute-force-password-spray/)と併用する。ただし
  スタッフィングは「1アカウント1回」なので、**IP・デバイス単位のレート制限とボット検知**が効く

> 🧭 C#/.NET なら ASP.NET Core Identity が土台。`SignInManager` は
> `RequiresTwoFactor` を返すので二段構えは標準機能で組める。漏洩照合や TOTP は
> 外部プロバイダ・ライブラリで足すのが定石。

## まとめ

- クレデンシャルスタッフィングは**他所の漏洩を持ち込む**攻撃。使い回しに便乗するので自社無傷でも刺さる
- 総当たりと違い「正しい組を1回試す」だけ。**試行回数ベースのロックアウトだけでは止まらない**
- 最優先は **MFA**。パスワードが合っても2要素目が無ければ通さない状態機械にする
- 登録・変更時に**漏洩パスワードを照合**する。k-匿名性のレンジクエリで生パスワードを外に出さない
- 検知は impossible travel・デバイスフィンガープリント・ボット検知、根治はパスキーで**共有秘密を無くす**
