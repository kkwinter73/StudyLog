[gRPC と REST](/posts/grpc-vs-rest/)で「REST はリソース指向」と書いた。ここはその中身を掘り下げ、
**RESTらしいAPIの組み立て方**を整理する。鍵は「やりたい操作をURLに書かない」こと。名詞でモデル化し、
動詞は HTTP メソッドに任せる。

## リソース指向とは

APIを「**操作の一覧**」でなく「**リソース(名詞)の集まり**」として設計する考え方。

```text
✕ 手続き的: /createUser  /getUser  /updateUser  /deleteUser
◯ リソース: /users（名詞）に対して POST/GET/PUT/DELETE（動詞はメソッド）
```

- リソース＝APIが扱う「モノ」（ユーザー・注文・商品…）
- 「何をするか」は URL でなく**HTTPメソッド**で表す

> 💡 URL に動詞が出てきたら黄信号。`/users/1/orders` のように**名詞を階層で並べる**のがリソース指向。

## URL の決め方

リソースの「住所」を素直に表す。コレクション(複数)と個別(単数)を分けるのが基本。

| パターン | 例 | 指すもの |
| --- | --- | --- |
| コレクション | `/users` | ユーザー全体 |
| 個別 | `/users/1` | ID=1 のユーザー |
| 従属リソース | `/users/1/orders` | ユーザー1の注文一覧 |
| 絞り込み | `/orders?status=paid` | クエリで条件指定 |

- 名詞は**複数形**で揃える（`/users`）。表記ゆれを作らない
- 絞り込み・並び・ページングは**クエリ文字列**で（URLの階層に混ぜない）

## メソッドの意味 — 安全性と冪等性

HTTPメソッドには決められた性質がある。これに沿うとキャッシュやリトライが安全に効く。

| メソッド | 用途 | 安全 | 冪等 |
| --- | --- | --- | --- |
| GET | 取得 | ✅（変更しない） | ✅ |
| POST | 作成 | ✕ | ✕（複数回で複数作成） |
| PUT | 置換・更新 | ✕ | ✅（何回でも同じ結果） |
| DELETE | 削除 | ✕ | ✅ |

- **安全(safe)**: 状態を変えない。GET は副作用を持たせない
- **冪等(idempotent)**: 同じ要求を何回送っても結果が同じ。ネットワーク再送に強い

> ⚠️ POST だけは冪等でない＝**リトライで二重作成**しうる。決済のような所は[冪等キー](/posts/idempotency-key-implementation/)で
> 補う。「GET なのに更新する」ような設計は、キャッシュやクローラに壊される。

> 🧭 C# の ASP.NET Core Web API も属性ルーティングで同じ形（`[HttpGet("users/{id}")]` 等）。
> フレームワークは違っても「名詞のルート＋メソッドで動詞」の作法は共通。

## ステートレスと表現(representation)

REST はやり取りを**ステートレス**にする。サーバはリクエスト間でセッション状態を覚えない。

- 認証情報など必要な文脈は**毎回のリクエストに載せる**（[境界で認証](/posts/auth-boundary-edge-vs-app/)）
- クライアントが受け取る JSON 等は、リソースそのものでなく**その時点の「表現」**。同じリソースを JSON でも別形式でも表せる

> 💡 ステートレスだから、どのサーバに振っても処理でき**水平にスケール**しやすい。状態はDBやトークンへ。

## やりがちなアンチパターン

リソース指向を崩す典型。避けるだけで素直なAPIになる。

- **動詞URL**: `/getUser` `/user/1/activate` → 状態変更は `PATCH /users/1` や従属リソースで表す
- **GETで副作用**: `GET /users/1/delete` → 破壊的操作を GET に載せない
- **巨大なクエリを本文で**: 取得なのに POST 多用 → まずクエリ文字列で表現できないか考える
- **1つの巨大エンドポイント**: 何でも `/api` に流す → リソースごとに分ける

## まとめ

- リソース指向は**名詞(リソース)を並べ、動詞はHTTPメソッド**に任せる設計
- URLは**複数形の名詞＋階層**、絞り込みは**クエリ文字列**で表す
- メソッドの**安全性・冪等性**に沿う。POST は非冪等なので二重作成に注意
- **ステートレス**にして文脈は毎回載せる＝水平スケールしやすい
- **動詞URL・GETで副作用**などのアンチパターンを避けるだけで素直になる
