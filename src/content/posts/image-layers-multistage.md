> 📚 シリーズ「コンテナをちゃんと理解する」(3 / 5)

[Dockerfile](/posts/dockerfile-basics/)の各命令は、実は **レイヤー（層）** として積み重なる。
この構造を知ると、ビルドが速くなる書き方と、イメージを小さくする[マルチステージビルド](/posts/deploying-go-apps/)が
なぜ効くのかが腑に落ちる。

## イメージはレイヤーの積み重ね

Dockerfile の `FROM`/`COPY`/`RUN` などの命令1つにつき、基本1つのレイヤーができる。
イメージは、そのレイヤーを下から積み上げた**読み取り専用の地層**。

```text
[CMD]            ← 上の層ほど後の命令
[RUN go build]
[COPY . .]
[FROM golang]    ← 一番下が土台
```

> 💡 同じベースイメージを使う複数のイメージは、下の共通レイヤーを**共有**する。
> だからディスクも pull も効率的（同じ層は1回持てばいい）。

## ビルドキャッシュ — 変わってない層は再利用

`docker build` は、各レイヤーについて「前回から変わってなければ**キャッシュを使い回す**」。
変更があったレイヤーと、**それより上**だけを作り直す。

> ⚠️ ポイントは「変更点より下は再利用、上は全部やり直し」。だから**変わりにくいものを先に、
> 変わりやすいものを後に**書くとキャッシュが効く。順序がビルド時間を左右する。

## キャッシュを効かせる書き方

典型は「依存のインストール」と「ソースのコピー」を分けること。

```dockerfile
FROM golang:1.26
WORKDIR /src

COPY go.mod go.sum ./     # 依存定義だけ先にコピー
RUN go mod download       # 依存DL（go.mod が変わらない限りキャッシュされる）

COPY . .                  # ソースは後（ここだけ頻繁に変わる）
RUN go build -o /app .
```

- ソースを1行直しただけなら、`go mod download` のレイヤーはキャッシュが効いて**依存DLをスキップ**
- もし `COPY . .` を先頭に置くと、1文字直すたびに依存DLからやり直しになる

## マルチステージビルド — 最終イメージを小さく

ビルドにはコンパイラ等の重い道具が要るが、**実行には成果物だけあればいい**。
そこで「ビルド用ステージ」と「実行用ステージ」を分け、成果物だけ最終イメージにコピーする。

```dockerfile
# --- build stage（重い道具一式） ---
FROM golang:1.26 AS build
WORKDIR /src
COPY . .
RUN CGO_ENABLED=0 go build -o /app .

# --- run stage（成果物だけ） ---
FROM gcr.io/distroless/static-debian12
COPY --from=build /app /app    # build ステージから成果物だけ持ってくる
ENTRYPOINT ["/app"]
```

- 最終イメージにコンパイラもソースも入らない → **数MB〜十数MB**に収まる
- [Goの静的バイナリ](/posts/deploying-go-apps/)と相性抜群。小さい＝pullが速い・攻撃面が小さい

## まとめ

- イメージは命令ごとの**レイヤーの積み重ね**。共通の層は複数イメージで共有される
- `docker build` は**変更点より下を再利用、上を作り直す**（ビルドキャッシュ）
- **変わりにくいもの(依存)を先、変わりやすいもの(ソース)を後**に書くとキャッシュが効く
- **マルチステージビルド**でビルド用と実行用を分け、成果物だけの小さいイメージにする
- Go の静的バイナリ + distroless/scratch で数MB級に
