[決済処理の流れ](/posts/payment-processing-flow/)で「アプリはカード番号を持たず PSP に任せる」と
書いた。その **PSP（決済代行）の国内代表例**がベリトランス。何者で、どこに立ち、アプリから
どう使うのかを、シリーズで使った用語に紐づけて整理する。

## 何者か

ベリトランスは現在 **株式会社DGフィナンシャルテクノロジー（旧ベリトランス株式会社）** が運営する
決済代行サービス。デジタルガレージ(DG)グループで、1997年創業の老舗。

- 現行の主力サービスは **VeriTrans4G**（オンライン/対面向けの総合決済）
- 取扱高は兆円規模・支払拠点も100万超という、国内では大手の部類

> 💡 国外でいう Stripe / Adyen に相当する立ち位置の**国内 PSP**。シリーズで抽象的に「PSP」と
> 呼んでいた箱に、具体名が入ったと思えばよい。

## 決済の流れの中での立ち位置

[クレジットカード決済の仕組み](/posts/credit-card-payment-mechanism/)で見た登場人物のうち、
ベリトランスは**アクワイアラやブランドへの接続を肩代わりする層**にあたる。

```text
顧客 → 自社アプリ → 【ベリトランス(PSP)】 → 各カード会社/決済事業者
                       ↑ ここを1つにまとめてくれる
```

- アプリは個々のカードブランドやコンビニ各社と**個別契約・個別接続しなくて済む**
- カード情報はベリトランス側に預け、アプリは[トークン](/posts/payment-processing-flow/)を扱う方針にできる（PCI DSS の負担を寄せられる）

## 強み —— 多数の決済手段を集約

国内 PSP を使う最大の動機がこれ。**30種類以上の決済手段を1契約・1接続で導入**できる。

| カテゴリ | 例 |
| --- | --- |
| カード | 主要国際ブランド |
| コンビニ | コンビニ払い（後から確定する非同期決済） |
| QR・ID 決済 | PayPay などのコード決済 |
| キャリア決済 | 携帯料金合算 |
| 電子マネー | 各種 |
| 越境/海外 | PayPal, Alipay など |

> 🧭 自前で各社と繋ぐと、API も入金サイクルも[Webhook](/posts/webhook-implementation/)もバラバラ。
> PSP はそれを**1つのインターフェースに抽象化**してくれる。C#/.NET でも Go でも、繋ぐ先が
> 1本になる恩恵は同じ。

## アプリから見た使いどころ

シリーズで扱った概念が、そのまま機能として乗っている。

- **与信→確定**: オーソリと売上確定の2段階、コンビニ等の**非同期確定**にも対応
- **本人認証**: [3Dセキュア2.0](/posts/3d-secure-sca/)・セキュリティコード認証で不正対策
- **結果通知**: 決済結果は同期レスポンスだけに頼らず、通知で受ける設計が前提
- **返金など**: [返金・取り消し](/posts/refund-chargeback-dispute/)も API 経由で行える

> ⚠️ 「PSP を使う＝何も考えなくていい」ではない。[冪等性](/posts/idempotency-key-implementation/)・
> 通知の[冪等処理](/posts/webhook-implementation/)・[突合](/posts/refund-chargeback-dispute/)は、
> どの PSP でもアプリ側の責任として残る。

## まとめ

- ベリトランス（VeriTrans4G）は DGフィナンシャルテクノロジーが運営する**国内大手の PSP**
- 立ち位置は、アクワイアラ/ブランドへの**接続を肩代わりする決済代行層**
- 強みは **30種以上の決済手段を1契約・1接続に集約**できること（越境も）
- 与信/確定・3Dセキュア・結果通知・返金など、**シリーズの概念がそのまま機能**になっている
- ただし**冪等性・通知処理・突合**はどの PSP でもアプリ側の責任として残る
