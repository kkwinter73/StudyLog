クライアントとサーバの間に攻撃者が割り込み、通信を読んだり書き換えたりするのが中間者攻撃（MITM）。
結論を先に言うと、**経路が信用できない前提で TLS を正しく使い、証明書検証を絶対に外さない**——
防御はこの一文にほぼ集約される。

## 攻撃の仕組み — 「経路の途中」に座る

MITM は暗号を破るのではなく、**通信経路の途中に割り込んで自分を経由させる**攻撃。代表的な入り口はこの3つ。

- **ARP スプーフィング**: LAN 内で「ゲートウェイは自分だ」と偽の ARP 応答を撒き、被害者の通信を自分に向かせる
- **不正 Wi-Fi（Evil Twin）**: 正規と同じ SSID のアクセスポイントを立て、接続してきた端末の通信を素通しで覗く
- **ダウングレード攻撃**: 暗号化された接続を、平文や弱い暗号方式に「格下げ」させて中身を読む（HTTPS を HTTP に落とす等）

いずれも成立すると、攻撃者はリクエストとレスポンスを**盗み見（盗聴）**でき、さらに書き換えて**改ざん**もできる。

> ⚠️ 「社内ネットワークだから安全」は通用しない。ARP スプーフィングは同じ LAN にいれば成立する。
> 経路の安全は**ネットワークの場所ではなく、暗号と証明書で担保する**。

## 対策の原則 — 経路を信用しない

MITM 対策は「通信路そのものは信用できない」を出発点に組み立てる。原則はシンプル。

| 原則 | 何をするか |
| --- | --- |
| **TLS everywhere** | 内部通信も含めて全経路を TLS 化する。平文区間を残さない |
| **証明書を検証する** | 相手のサーバ証明書が信頼された CA 発行で、ホスト名と一致するか必ず確認する |
| **HSTS** | ブラウザに「このドメインは常に HTTPS」と宣言し、HTTP へのダウングレードを封じる |
| **証明書ピンニング** | 特定の証明書・公開鍵だけを受け入れる。モバイルアプリなど**経路統制したい場面で限定的に**使う |

- ピンニングは強力だが、証明書更新時に壊れやすい。**むやみに使わず、更新運用まで含めて設計できる時だけ**採用する
- 一番やってはいけないのは**「動かないから検証を無効化する」**。それは MITM への扉を自分で開ける行為。

## Go での防御 — 既定は安全、外すと危険

`net/http` は既定でサーバ証明書を検証する。**素直に使えばそれだけで MITM に強い**。

```go
// 既定の http.Client はサーバ証明書を検証する。これで十分。
resp, err := http.Get("https://api.example.com/health")
if err != nil {
    log.Fatal(err) // 証明書が不正なら、ここでエラーになる（＝守られている）
}
defer resp.Body.Close()
```

問題は、証明書エラーを消したくて次のように書いてしまうケース。**これは本番に出してはいけない**。

```go
// アンチパターン: 証明書検証を無効化 —— MITM を素通しにする
tr := &http.Transport{
    TLSClientConfig: &tls.Config{InsecureSkipVerify: true}, // ⚠️ 絶対にやらない
}
client := &http.Client{Transport: tr}
```

`InsecureSkipVerify: true` は「誰が相手でも信じる」宣言。偽サーバや割り込んだ攻撃者の証明書も
無条件で受け入れるので、TLS を張っていても**中身は丸見え・書き換え自在**になる。

代わりに、**最低バージョンを固定**して弱い方式へのダウングレードを防ぐ。

```go
// 良い方向: 検証は有効のまま、最低バージョンを TLS 1.2 に固定
cfg := &tls.Config{
    MinVersion: tls.VersionTLS12, // より厳しくするなら tls.VersionTLS13
}
tr := &http.Transport{TLSClientConfig: cfg}
client := &http.Client{Transport: tr}
```

サービス間通信で「相手も本物か」を双方向に検証したいなら **mTLS**（クライアント証明書も提示させる）を使う。
`tls.Config` の `ClientCAs` と `ClientAuth` で構成できる。

> 🧭 C# の `HttpClient` も既定で証明書を検証する。危険な同等物は
> `HttpClientHandler.ServerCertificateCustomValidationCallback` で常に `true` を返すこと——
> Go の `InsecureSkipVerify: true` とまったく同じ落とし穴なので、`return true;` を見たら疑う。

## まとめ

- MITM は暗号破りではなく**経路の途中に割り込む**攻撃。ARP スプーフィング・不正 Wi-Fi・ダウングレードが代表的な入り口
- 対策の軸は **TLS everywhere・証明書検証・HSTS**。ピンニングは更新運用まで設計できる場面だけ限定採用
- Go の `net/http` は**既定で証明書を検証する**。素直に使えば守られている
- `tls.Config{InsecureSkipVerify: true}` は MITM を素通しにする**アンチパターン。本番に出さない**
- ダウングレード対策に `MinVersion: tls.VersionTLS12/13` を固定。サービス間は **mTLS** で相互検証
