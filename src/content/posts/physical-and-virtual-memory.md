[カーネルの役割](/posts/kernel-role-for-containers/)で挙げた4つの仕事のうち、ここでは **メモリ管理**を深掘りする。
各プロセスが「自分専用の広いメモリがある」かのように動けるのは **仮想メモリ** の仕組みのおかげ。
物理メモリだけだと何に困り、仮想メモリがそれをどう解くのかを押さえる。

## 物理メモリだけだと何が困るか

物理メモリ（RAM）は実在する有限のチップ。アプリが物理アドレスを直接使うと問題が噴出する。

- **競合**: アプリ A と B がたまたま同じアドレスを使うと、互いのデータを踏み潰す
- **断片化**: 空きはあるのに連続した領域が取れず、確保に失敗する
- **容量の壁**: 物理メモリより大きなデータを扱えない

> ⚠️ そもそもアプリ同士が同じ物理アドレスを生で触れる時点で、[隔離](/posts/user-space-vs-kernel-space/)が成り立たない。
> 1本のバグが他アプリのメモリを壊し放題になってしまう。

## 仮想メモリ — 各プロセスに専用空間を見せる

解決策は、**アプリには「仮想アドレス」という見せかけの住所を使わせ**、実際の物理アドレスへの対応づけを
カーネルが裏で管理すること。各プロセスは **0 番地から始まる自分専用の広大なアドレス空間**を持つ。

```text
プロセスA の仮想空間 ─┐
                      ├─▶ カーネルが変換 ─▶ 物理メモリ（RAM）の実際の場所
プロセスB の仮想空間 ─┘
```

- A の「100 番地」と B の「100 番地」は、**別々の物理メモリ**に対応づく → 競合しない
- アプリは物理メモリの実際の配置・空き具合を一切気にしなくていい

> 💡 「仮想アドレス＝各社員に配る内線番号、物理アドレス＝実際の電話機」。
> 内線は会社ごとに 100 番から始められ、交換機（カーネル）が本物の機械へ繋いでくれる。

## ページングとアドレス変換

仮想と物理の対応は、**ページ**という固定サイズ（典型的に 4KB）の単位で管理する。
どの仮想ページがどの物理ページに対応するかは **ページテーブル**に記録され、変換は CPU の
**MMU（メモリ管理ユニット）**というハードウェアが高速に行う。

| 用語 | 意味 |
| --- | --- |
| ページ | 管理単位（例: 4KB）。この粒度で割り当て・変換する |
| ページテーブル | 仮想ページ → 物理ページ の対応表（プロセスごとに持つ） |
| MMU | 変換を担う CPU 内のハード。毎回のメモリアクセスで変換する |
| ページフォルト | 対応する物理ページが無いときに発生し、カーネルが処理する |

ページ単位なので、物理メモリは飛び飛びの空きでも割り当てられる（連続している必要がない）。
これで断片化の問題も緩む。

## どこで効いてくるか

仮想メモリを知っていると、こんな話が繋がる。

- **スワップ**: 使っていないページをディスクへ退避し、物理メモリより大きく使える(遅くなるが破綻しない)
- **隔離**: プロセスごとに別のページテーブルを持つから、互いのメモリを覗けない＝安全
- **OOM**: それでも物理＋スワップを使い切ると、カーネルがプロセスを kill する（OOM Killer）

> 🧭 C#/.NET の GC もこの仮想アドレス空間の上で managed heap を管理している。Go の GC も同じ。
> **言語のメモリ管理（GC）と、OS のメモリ管理（仮想メモリ）は別レイヤー**で、二段重ねになっている。

## まとめ

- 物理メモリを生で使うと **競合・断片化・容量の壁・隔離崩壊** が起きる
- **仮想メモリ**は各プロセスに 0 番地始まりの専用アドレス空間を見せ、カーネルが物理へ対応づける
- 対応は **ページ（4KB 等）単位**で管理し、変換は **MMU** が、対応表は **ページテーブル**が担う
- 物理が足りなければ **スワップ**でディスクへ退避。使い切れば **OOM Killer** が動く
- 言語の GC は OS の仮想メモリの**上の層**。メモリ管理は二段重ねになっている

次にやること: `/proc/<pid>/maps`（Linux）でプロセスの仮想アドレス空間がどう並んでいるか覗いてみる。
