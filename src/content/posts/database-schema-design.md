[RDBMS](/posts/rdbms-basics/) で「壊れないデータ」を持つ土台は、**テーブルの分け方**で決まる。
1枚の巨大な表に何でも詰めると、重複と矛盾がすぐ生まれる。エンティティ分割・正規化・キー設計・
あえての非正規化の4点を押さえ、**重複せず矛盾しない**スキーマを作る。

## 何がまずいのか — 1枚の表に詰めると壊れる

たとえば注文を1枚の表に全部入れると、こうなる。

```text
orders
+----+--------+-------------+-----------+--------+
| id | 顧客名 | 顧客メール  | 商品名    | 単価   |
+----+--------+-------------+-----------+--------+
| 1  | 田中   | t@ex.com    | ノート    |  300   |
| 2  | 田中   | t@ex.com    | ペン      |  100   |  ← 顧客情報が重複
| 3  | 佐藤   | s@ex.com    | ノート    |  350   |  ← 同じ商品なのに単価が違う…矛盾
+----+--------+-------------+-----------+--------+
```

- 田中のメールを直すには**複数行を全部**直す必要がある（更新の手間・漏れ）
- 同じ「ノート」の単価がバラバラ（**どれが正しいか分からない**）
- まだ注文のない顧客を登録できない（注文行がないと存在できない）

> ⚠️ これらは **更新・挿入・削除の異常(anomaly)** と呼ばれる。原因は「別々の事実を1つの表に混ぜた」こと。

## エンティティごとに表を分ける

設計の第一歩は、**独立した"モノ"(エンティティ)を別々の表にする**こと。上の例なら
顧客・商品・注文は別の事実なので、3つに分ける。

```text
customers          products          orders
+----+------+       +----+------+----+   +----+-------------+------------+
| id | name |       | id | name |単価|   | id | customer_id | product_id |
+----+------+       +----+------+----+   +----+-------------+------------+
| 1  | 田中 |       | 1  |ノート|300 |   | 1  |     1       |     1      |
| 2  | 佐藤 |       | 2  | ペン |100 |   | 2  |     1       |     2      |
+----+------+       +----+------+----+   +----+-------------+------------+
```

- 顧客の事実は `customers` に**1か所だけ**（メール修正は1行で済む）
- 商品の単価も `products` に**1か所だけ**（矛盾のしようがない）
- 注文は「どの顧客がどの商品を」という**関係だけ**を持つ

> 🧭 C#/.NET なら EF Core のエンティティクラス1つが1テーブルに対応するのと同じ発想。
> クラスを分けるのと同じ粒度で、DBのテーブルも分ける。

## 正規化 — 重複をなくす3ステップ

上の「表を分ける」を体系化したのが**正規化**。第1〜第3正規形(1NF〜3NF)まで押さえれば実務は足りる。

| 正規形 | ルール | 平たく言うと |
| --- | --- | --- |
| 第1正規形 (1NF) | 各セルは1つの値。繰り返し列を作らない | `商品1, 商品2, 商品3` のような列にしない |
| 第2正規形 (2NF) | 主キーの一部にだけ依存する列を分ける | 複合キーの片方だけで決まる列は別表へ |
| 第3正規形 (3NF) | キー以外の列に依存する列を分ける | 「郵便番号→住所」のような芋づるを別表へ |

```text
-- 3NF 違反の例: 郵便番号が決まれば住所も決まる（キー以外への依存）
customers(id, name, 郵便番号, 都道府県)   ← 都道府県は郵便番号に依存

-- 分けると矛盾しない
customers(id, name, 郵便番号)
zipcodes(郵便番号, 都道府県)
```

> 💡 合言葉は **「1つの事実は1か所に(One fact, one place)」**。同じ事実が2か所に出たら、
> いつか片方だけ更新されて矛盾する。正規化はそれを構造で防ぐ。

## 主キーと外部キーで関係を張る

分けた表は**キー**で結ぶ。ここが設計の肝。

- **主キー(PK)**: 行を一意に識別する列。**代理キー**（連番・UUID の `id`）が扱いやすい。
- **外部キー(FK)**: 他の表の主キーを指す列。存在しない相手を指せなくなる（**参照整合性**）。

```sql
CREATE TABLE orders (
  id          BIGINT PRIMARY KEY,
  customer_id BIGINT NOT NULL REFERENCES customers(id),  -- FK
  product_id  BIGINT NOT NULL REFERENCES products(id),
  quantity    INT    NOT NULL CHECK (quantity > 0)
);
```

自然キー（メールアドレス等の意味のある値）を主キーにしがちだが、**変わりうる値はPKに向かない**。

| | 代理キー(id) | 自然キー(email等) |
| --- | --- | --- |
| 変更 | 変わらない | 変わりうる（PK変更は波及大） |
| 関係 | FKが短く安定 | FKが長く不安定 |
| おすすめ | **基本こちら** | UNIQUE制約で別途一意性を守る |

> 🧭 多対多（注文と商品が両方向に複数）になったら、**中間テーブル**（`order_items` 等）を1枚挟む。
> C# のコレクションナビゲーションプロパティが、DBでは中間テーブルとして現れる。

## 正規化しすぎない — あえての非正規化

正規化は基本だが、**やりすぎると JOIN だらけで遅くなる**。読み取りが重い所は、あえて重複を持つ。

- 一覧表示で毎回 JOIN する集計値（合計金額など）を**あらかじめ列に持つ**
- 参照が集中する所に[キャッシュ](/posts/caching-cache-aside/)を挟む
- 検索を速くする列に**インデックス**を張る（FK列・検索条件の列が定番）

> ⭐ 原則は **「まず正規化、遅い所だけ測ってから崩す」**。最初から非正規化すると、
> 冒頭の「矛盾する表」に逆戻りする。崩すなら「重複は自分で同期する」と決めた上で。

> 💡 スキーマは一度作ると変えづらい。本番で変えるときは[expand-contract 移行](/posts/db-schema-migration-expand-contract/)で止めずに。

## まとめ

- 1枚の表に詰めると**更新・挿入・削除の異常**が出る。原因は事実の混在
- **エンティティごとに表を分け**、正規化(1NF→3NF)で「1つの事実は1か所」にする
- 表は**主キー(代理キー推奨)と外部キー**で結び、参照整合性で矛盾を防ぐ
- 多対多は**中間テーブル**、変わりうる値はPKにせず UNIQUE で守る
- 正規化しすぎず、**遅い所だけ測って非正規化**（同期の責任とセット）
