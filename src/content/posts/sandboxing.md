信頼できないコードを普通に実行すれば、ファイルを読まれ、通信され、権限を奪われ得る。
**サンドボックス**は、コードを**隔離された箱**に閉じ込め、**外へ害を及ぼせなくして**動かす仕組み。
基本思想は**デフォルト拒否**——箱の中からは、明示的に許した最小限のことしかできない。

## なぜ隔離するのか — 信頼できないコードを安全に動かす

「自分が書いたわけではない」「安全か分からない」コードを動かす場面は、意外と多い。

- **ブラウザ**: 開いたページの JavaScript は他人のコード。タブ/サイトごとにプロセスを隔離して守る
- **スマホアプリ**: 各アプリを箱に閉じ、他アプリや端末全体に勝手に触らせない
- **コンテナ/サーバレス**: 他人のワークロードや関数を同じ基盤で安全に同居させる
- **プラグイン・拡張・ユーザー投稿コード**: CIで走るPRのコード、SaaSのユーザースクリプトなど
- **マルウェア解析**: 危険な検体を隔離環境で動かして挙動を観察する

> ⭐ サンドボックスの目的は「悪いことをさせない」ではなく「**悪いことをされても被害を箱の中に閉じ込める**」こと。
> コードが乗っ取られる前提で、**被害範囲（blast radius）を最小化**するのが発想の核だ。

## 何を断つのか — 最小権限で囲う

箱の壁は「アクセスできる対象を絞る」ことで作る。代表的に次を制限する。

- **ファイルシステム**: 見える範囲を限定（箱の外のファイルは存在しないことにする）
- **ネットワーク**: 外部通信を遮断/限定。データを持ち出させない
- **システムコール**: カーネルに頼める操作を絞る（危険なsyscallを禁止）
- **リソース**: CPU・メモリ・実行時間の上限（暴走や資源枯渇を防ぐ）
- **権限**: 不要な特権を落とす（root相当を与えない）

> 🧭 .NET はかつて AppDomain / Code Access Security で**プロセス内サンドボックス**を提供したが、今は非推奨。
> 「信頼できないコードは**言語機能ではなく OS/プロセス/コンテナの層で隔離する**」が現在の定石。境界は下の層ほど強い。

## どう作るか — 隔離の強さとコストは裏返し

隔離は「強いほど重い」。用途に合う強度を選ぶ。土台は[OSの機能](/posts/container-namespace-cgroup/)だ。

| 方式 | 隔離の強さ | コスト | 例 |
| --- | --- | --- | --- |
| OS機能を直接 | 弱〜中 | 軽い | seccomp（syscall制限）・chroot・capabilities |
| コンテナ | 中（カーネル共有） | 軽い | Docker（namespace＋cgroup） |
| ユーザー空間カーネル | 強 | 中 | gVisor（syscallを横取りして代行） |
| 軽量VM | 強（カーネルごと別） | 中〜重 | Firecracker（Lambda等の基盤） |
| WASM | 中〜強（能力ベース） | 軽い | WASI（許可した機能だけ渡す） |

- コンテナは軽いが**ホストとカーネルを共有**する。カーネルの穴が箱の壁の穴になり得る（[ユーザー空間とカーネル](/posts/user-space-vs-kernel-space/)）
- 他人のコードを本気で隔離するなら、**gVisor や軽量VM**でカーネル境界を厚くする。AWS Lambda が microVM を使うのはこのため

## 限界 — サンドボックスエスケープ

箱は完璧ではない。壁を破って外に出る攻撃を**サンドボックスエスケープ（ブレイクアウト）**と呼ぶ。

- カーネルやハイパーバイザの脆弱性を突かれると、箱の外＝ホストへ抜けられる
- だから**単一の壁に頼らない**。seccompで syscall を絞り、権限を落とし、VM境界を重ねる——**多層防御**にする
- 「サンドボックスに入れたから絶対安全」は禁物。**パッチ適用**と**最小権限**を土台に、箱はその上の一枚と考える

> ⚠️ 隔離の強さと運用コスト・性能は必ずトレードオフ。全部を最強VMに入れる必要はない。
> **扱うコードの信頼度**（自社コードか／他人の任意コードか）に応じて強度を選ぶのが現実的。

## まとめ

- サンドボックスは信頼できないコードを**隔離箱**で動かし、**被害を箱の中に閉じ込める**仕組み。基本は**デフォルト拒否**
- 使う場面はブラウザ・スマホアプリ・コンテナ/サーバレス・プラグイン・マルウェア解析など「他人のコードを動かす」所
- 断つのは**ファイル・ネットワーク・システムコール・リソース・権限**。最小権限で囲う
- 作り方は強度順に **OS機能 < コンテナ < gVisor < 軽量VM**（＋WASM）。**強いほど重い**トレードオフ
- 箱は完璧でない（**サンドボックスエスケープ**）。多層防御・パッチ・最小権限を土台に、信頼度に応じ強度を選ぶ
