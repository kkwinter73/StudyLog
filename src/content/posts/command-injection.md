外部コマンドを呼ぶときにユーザー入力を文字列連結すると、`;` や `|` を使って任意のコマンドを
実行されてしまう。これがコマンドインジェクション。**シェルに文字列を渡さない**——
Go では最初からそうなっている——のが一番確実な対策だ、という話。

## 攻撃の仕組み — 文字列がシェルに届くと壊れる

「URL を受け取って `git clone` する」ような処理を、コマンド文字列を組み立てて
シェルに渡す形で書いたとする。

```bash
# ユーザー入力 url をそのまま連結してシェルに渡す（危険）
git clone $url
```

`url` に普通の URL が来る前提だが、攻撃者は URL のつもりで**シェルのメタ文字**を送り込む。

```text
入力: https://example.com/repo.git; rm -rf /
実行: git clone https://example.com/repo.git; rm -rf /
                                         ^^^^^^^^^^^^ 別コマンドとして実行される
```

`;` `|` `&&` `$()` `` ` `` などはシェルにとって「コマンドの区切り」や「別コマンドの実行」を
意味する。文字列がシェルの目に触れた瞬間、これらは**データではなく命令**として解釈される。

> ⚠️ 原理は[SQLインジェクション](/posts/sql-injection/)とまったく同じ。データのつもりの入力が、
> パーサ（あちらは SQL、こちらはシェル）に**構文**として解釈されるのが根っこ。

## 対策の原則 — シェルを介さない

小手先のエスケープ（メタ文字を潰す）より、**そもそもシェルに文字列を組ませない**方が確実。

- **文字列を組み立てない**: `"git clone " + url` のような連結をやめる
- **引数はリストで渡す**: コマンド名と各引数を配列で分けて渡す。区切りをシェルに判断させない
- **シェル自体を避ける**: `sh -c "..."` を経由しなければ、メタ文字はただの文字になる
- **検証・許可リスト**: それでも入力が形式を満たすか（URL か、想定のホストか）を確認する

> 💡 引数をリストで渡せば、`; rm -rf /` は**「そういう名前の1個の引数」**として git に渡るだけ。
> 別コマンドにはなりようがない。

## Go での防御 — exec.Command はシェルを経由しない

Go の `os/exec` は、コマンドと引数を**個別の値**として受け取り、OS に直接渡す。
シェル（`sh`）を挟まないので、メタ文字が解釈される隙がない。

```go
// 安全: git に url を「1個の引数」として渡す。シェルを経由しない
cmd := exec.Command("git", "clone", url)
if err := cmd.Run(); err != nil {
	log.Fatal(err)
}
```

`url` に `repo.git; rm -rf /` が来ても、それは `git` の第2引数の**文字列そのもの**でしかなく、
`rm` が動くことはない（git が「不正な URL」と怒るだけ）。一方、次はやってはいけない書き方。

```go
// 危険: わざわざシェルを呼び、文字列を連結している
cmd := exec.Command("sh", "-c", "git clone "+url)
```

`sh -c` に渡した瞬間、中身はシェルが解釈する。`;` 以降が別コマンドとして走る。
**`exec.Command` の第1引数を `sh`/`bash` にした時点で危険信号**、と覚えておく。

> 🧭 C# の `Process` も同じ構図。`ProcessStartInfo.Arguments`（1本の文字列）ではなく
> `ArgumentList`（リスト）に積み、`UseShellExecute = false` にすればシェルを経由しない。
> Go の `exec.Command` は既定でこの「シェルなし・引数リスト」側になっている。

## それでもシェルが要るとき

パイプやリダイレクトなど、どうしてもシェル機能が要る場面はある。そのときも
ユーザー入力を文字列に混ぜない工夫で被害を抑えられる。

- **入力は環境変数経由で渡す**: スクリプト本文には `$URL` と書き、値は `cmd.Env` で渡す。
  文字列連結でスクリプトを組まない
- **許可リストで弾く**: 入力を正規表現で検証し、想定パターン以外は実行前に拒否する
- **可能なら分解する**: パイプでつなぎたい2コマンドを、Go 側で `exec.Command` を2つ作って
  `StdoutPipe` でつなぐ。シェルを呼ばずに同じことができる

> ⭐ 判断の軸はひとつ。「その入力は**構文として解釈される場所**に届くか？」
> 届くなら（シェルでも SQL でも[パス](/posts/path-traversal/)でも）データとして扱う経路に切り替える。

## まとめ

- コマンドインジェクションは、ユーザー入力が**シェルに構文として解釈される**ことで起きる
- `;` `|` `&&` `$()` はシェルの区切り・別コマンド実行の記号。データに混ざると命令になる
- 対策の芯は**シェルを介さない・引数はリストで渡す**。エスケープより確実
- Go の `exec.Command("git", "clone", url)` はシェルなしで引数を個別に渡すので安全。
  `exec.Command("sh", "-c", ...)` は自らシェルを呼ぶので危険
- C# なら `ArgumentList` + `UseShellExecute = false`。考え方は Go と同じ
