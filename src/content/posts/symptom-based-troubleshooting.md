監視カリキュラム 補足。[三本柱](/posts/observability-basics/)（ログ・メトリクス・トレース）は道具の話だが、
いざ障害が起きたときに迷うのは「**どの道具から手に取るか**」。症状 → まず見る場所 → 疑う先、の対応を
早見表として持っておくと初動が速くなる。

## 早見表

| 症状 | まず見る場所 | 疑う先 |
| --- | --- | --- |
| **エラー率が上がった** | ログ — 4xx/5xx の中身と、どのエンドポイントか | 特定リクエストだけ → アプリ側／全体 → インフラ・依存先 |
| **レイテンシが上がった（エラーは出ていない）** | トレース — どのスパンが遅いか | DB・外部 API 呼び出し |
| **特定エンドポイントだけ遅い/エラー** | そのサービスのログとトレース | デプロイ直後なら直近の変更 |
| **全体的に遅い＆リソースも逼迫** | メトリクス — CPU・メモリ・コネクション数 | スケール不足・リーク |

> 💡 ざっくり「**エラーはログ、遅さはトレース、全体はメトリクス**」。その壊れ方について
> いちばん情報が濃い柱から入る、というだけの話で、丸暗記しなくても導出できる。

> 🧭 C#/.NET の Application Insights でも同じ構図。Failures ブレード（エラーから入る）と
> Performance ブレード（遅さから入る）が、この早見表の最初の2行にそのまま対応する。

## なぜこの対応になるか — 「エラーの有無」と「範囲」の2軸

早見表は2つの問いの組み合わせでできている。

- **エラーは出ているか？** 出ていれば、エラーメッセージ・スタックトレースという**文字の手がかり**が
  ある——だから[ログ](/posts/cloudwatch-logs/)。出ていないのに遅いなら、知りたいのは
  「どこで時間を食っているか」という**時間の内訳**——だから[トレース](/posts/distributed-tracing-otel/)。
- **範囲は特定か、全体か？** 特定のエンドポイント・リクエストだけなら、そのコードパス固有の原因
  （＝アプリ側）。全体に及ぶなら、共有しているもの（インフラ、DB、依存先、リソース）が容疑者。

> ⭐ 範囲の切り分けが先、深掘りは後。「全体か特定か」を最初の1分で確かめるだけで、
> 疑う先が半分に絞れる。

## 各症状の掘り下げ

### エラー率が上がった → ログ

まず 4xx か 5xx かを見る。**4xx はクライアント側**（不正なリクエスト、認証切れ、クライアントの
バグやリリース）、**5xx はサーバ側**。次に「どのエンドポイントに偏っているか」で範囲を確かめ、
特定なら該当ハンドラのログ、全体なら DB 接続や依存サービスのエラーを疑う。

### レイテンシが上がった（エラーなし） → トレース

エラーが出ない遅さは、たいてい**どこかの呼び出しが遅い**だけ。トレースでリクエストの
[スパンの内訳](/posts/distributed-tracing-otel/)を見れば「DB クエリに 800ms」「外部 API 待ちに 2s」の
ように犯人が名指しできる。経験上、疑う先の筆頭は DB と外部 API 呼び出し。

### 特定エンドポイントだけ遅い/エラー → そのサービスのログとトレース

範囲が絞れている分、原因もそのコードパスに閉じていることが多い。**デプロイ直後なら真っ先に
直近の変更を疑う**——差分を見る・ロールバックする方が、ログを掘るより速いことも多い。

### 全体的に遅い＆リソースも逼迫 → メトリクス

全体が一様に重いなら、個々のリクエストではなく**土台（リソース）**の問題。CPU・メモリ・
コネクション数のメトリクスを[平常時のベースライン](/posts/metrics-baseline-and-thresholds/)と
比べる。負荷増に対するスケール不足か、時間とともに漏れるリーク（メモリ・コネクション）が定番。

## どの症状でも、最初に確認する2つ

早見表に入る前に、共通のチェックが2つある。

- **直近にデプロイしたか**。障害の相当数は直近の変更が原因。症状が何であれ
  「最後に何を変えたか」は最初に思い出す。
- **[トラフィック](/posts/traffic-monitoring-signal/)は変わったか**。負荷が2倍になっていれば、
  遅さもリソース逼迫も「アクセス増」で説明がつくかもしれない。値は文脈とセットで読む。

## まとめ

- 症状 → まず見る場所の対応を**事前に**持っておくと初動が速い（障害中に考えない）
- 目安は「**エラーはログ、遅さはトレース、全体はメトリクス**」
- 早見表は「**エラーの有無**」×「**範囲（特定か全体か）**」の2軸から導出できる
- 特定エンドポイントだけ・デプロイ直後なら、**直近の変更**が筆頭容疑者
- どの症状でも、**直近デプロイ**と**トラフィックの変化**はまず確認する
