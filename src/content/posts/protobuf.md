**Protobuf（Protocol Buffers）** は Google 発の、**スキーマ先行のバイナリシリアライズ形式**。
`.proto` ファイルにデータの型を定義し、各言語のコードを**生成**して使う。[gRPC](/posts/grpc-vs-rest/) の土台として有名だが、
単体のデータ形式としても使える。この記事は gRPC ではなく、**Protobuf という"データの詰め方"**に絞る。

## なぜ Protobuf か — JSON との違い

用途はJSONと同じ「構造化データの受け渡し」。だが性質はかなり違う。

| | JSON | Protobuf |
| --- | --- | --- |
| 形式 | テキスト | **バイナリ** |
| サイズ | 大きい（キー名も毎回送る） | **小さい**（番号だけ送る） |
| 速度 | 遅め | **速い** |
| 人が読める | ○ | ✗（`.proto` が要る） |
| スキーマ | 任意（無くても送れる） | **必須**（`.proto`） |
| 多言語 | パーサ任せ | **コード生成で型付き** |
| 進化 | ゆるい（壊れやすい） | 番号で**厳密に管理** |

> ⭐ Protobuf の強みは「**小さい・速い・型がある**」。代償は「人が読めない・スキーマ必須」。
> 内部サービス間や大量通信のように**速さと型が効く所**で使い、人が直接読むAPIはJSONが向く、という住み分けになる。

## `.proto` を書いてコード生成する

まず型を `.proto` に宣言する。各フィールドには**番号**を振る。これが最重要だ。

```proto
syntax = "proto3";

message User {
  int32           id    = 1;   // 1,2,3… はフィールド番号（ワイヤ上の識別子）
  string          name  = 2;
  repeated string roles = 3;   // repeated = 配列
}
```

この `.proto` を `protoc` でコンパイルすると、Go や他言語の**型付きクラス/構造体が生成**される。手書きしない。

```text
ワイヤ上に流れるのは「番号2 → "Alice"」であって、"name" という文字列は送られない
   → だから小さい。ただし復元には .proto（番号↔名前の対応）が要る
```

> 🧭 Go は `protoc` ＋ `protoc-gen-go`、.NET は `Grpc.Tools` が `.proto` からコードを生成する。
> どちらも「**`.proto` が唯一の真実源、コードは生成物**」という点は同じ。生成物を手で直さないのは[このブログの記事と同じ](/posts/single-source-of-truth/)発想だ。

## スキーマ進化 — フィールド番号が主役

Protobuf が実務で強いのは**スキーマを壊さず育てられる**こと。鍵はフィールド**番号**にある。

- **足すのはOK**: 新しいフィールドは**新しい番号**で追加。古いコードは知らない番号を**無視**する（前方互換）。新しいコードは、無い時は**既定値**で読む（後方互換）
- **番号を再利用しない**: 一度使った番号を別用途に回すと、古いデータを誤解する。退役番号は `reserved` で封印する
- **名前の変更は安全**: ワイヤ上は番号だけなので、**番号を保てばフィールド名は変えてよい**
- **型の変更は基本NG**: 互換のない型変更はデータを壊す

> 💡 これは[DBスキーマ移行の expand-contract](/posts/db-schema-migration-expand-contract/)と同じ発想——**後方互換を保ったまま足し、古いものは封印する**。
> 番号という不変のIDに意味を紐づけることで、新旧のコードが同じデータを安全に読める。

## 注意点

- **人が読めない**: バイナリなので、デバッグには専用ツールや `.proto` が要る。ログに生で出しても分からない
- **`.proto` の共有が要る**: 送り手と受け手が**同じスキーマ**を持つ必要がある（スキーマ結合）。配布・版管理を仕組みにする
- **proto3 の既定値**: スカラは「未設定」と「ゼロ値」を区別しないのが既定。区別したいなら `optional` を付けて presence を持たせる
- **番号規律が命**: 番号の付け間違い・使い回しが最大の事故源。`.proto` のレビューを厳格に

## まとめ

- Protobuf は**スキーマ先行のバイナリシリアライズ**。`.proto` で型を定義し**コード生成**して使う（gRPCの土台だが単体でも可）
- JSONと比べ**小さい・速い・型がある**が、**人が読めず・スキーマ必須**。速さと型が効く所で使う
- ワイヤ上は**フィールド番号だけ**を運ぶ（名前は送らない）。だから小さく、復元に `.proto` が要る
- スキーマ進化は**番号が主役**——足すのはOK、**番号の再利用はNG**、名前変更は可。expand-contract と同じ発想
- 注意は**可読性・スキーマ共有・proto3の既定値・番号規律**。`.proto` を真実源として厳格に管理する
