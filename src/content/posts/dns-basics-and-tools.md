> 📚 シリーズ「ネットワークをちゃんと使う」(3 / 4)

ブラウザに `example.com` と打つと、裏で **名前 → IPアドレス** の変換が走る。これが **DNS（名前解決）**。
仕組みと主要レコード、`dig`/`nslookup` での調べ方を押さえ、「名前で繋がらない」を切り分けられるようにする。

## 名前解決の流れ

人は名前(`example.com`)で覚え、通信はIP(`93.184.216.34`)で行う。その橋渡しが DNS。
おおまかには、聞いて回って答えにたどり着く。

```text
アプリ → リゾルバ(問い合わせ役) → ルート → TLD(.com担当) → 権威サーバー(example.com担当)
                                                              └→ 「93.184.216.34 です」
```

- 一度引いた結果は一定時間 **キャッシュ**される（TTL）。だから毎回ルートまで聞きには行かない
- アプリから見れば「名前を渡すとIPが返る」仕組み。[OSのリゾルバ](/posts/user-space-vs-kernel-space/)が代行する

> 💡 電話帳に近い。「名前(会社名)」で引くと「番号(IP)」が返る。DNSはその電話帳を世界規模で分担管理している。

## 主要レコード — A と CNAME

DNSには種類(レコード)があり、よく使うのはこの2つ。

| レコード | 意味 | 例 |
| --- | --- | --- |
| A | 名前 → **IPアドレス**を直接指す | `example.com → 93.184.216.34` |
| CNAME | 名前 → **別の名前**へ転送(別名) | `www.example.com → example.com` |

- **A** が最終的な「住所」。**CNAME** は「あっちと同じ」という転送の別名
- CNAME の先をたどると最後は必ず A にたどり着く

> ⚠️ ドメインの頂点(`example.com` 自体)には CNAME を付けられない制約がある(仕様上)。
> このへんは AWS の Route 53 だと Alias レコードで回避する、という話につながる。

## dig / nslookup で調べる

「名前がどのIPに解決されるか」を手元で確認できる。トラブル時の必須技。

```bash
dig example.com +short        # 解決されたIPだけ表示（一番よく使う）
dig www.example.com           # 詳細（CNAMEの連鎖やTTLも見える）
dig example.com MX            # レコード種別を指定して引く

nslookup example.com          # 簡易版。dig が無い環境でも使えることが多い
```

> 💡 `dig +short` で**期待したIPが返るか**をまず確認。返ってこない/違うIPなら、接続以前に
> **DNSの設定ミス**を疑う。「サーバーは生きてるのに名前で繋がらない」の典型原因。

## まとめ

- DNSは **名前 → IPアドレス** を引く仕組み。リゾルバが分担サーバーに聞いて回る
- 結果は **TTL の間キャッシュ**される。毎回ルートまでは行かない
- **A** = IPを直接指す / **CNAME** = 別の名前への転送(別名)。最後は必ず A に着く
- **`dig 名前 +short`** で解決先IPを確認。期待と違えば DNS 設定ミスを疑う
- 「サーバーは生きてるのに名前で繋がらない」は DNS が原因のことが多い
