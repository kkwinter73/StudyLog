API の作り方には大きく2系統ある。**REST（リソース指向）**と **gRPC（手続き指向）**。
[内部通信でgRPCを使う構成](/posts/internal-service-grpc/)は別記事で触れたので、ここは
**APIスタイルとしての違い**——設計思想・契約・ストリーミング・ブラウザ対応——と選び方に絞る。

## REST — リソースを操作する

REST は「**データを名詞(リソース)として並べ、HTTPメソッドで操作する**」スタイル。

```text
GET    /users/1        ユーザー1を取得
POST   /users          ユーザーを作成
PUT    /users/1        ユーザー1を更新
DELETE /users/1        ユーザー1を削除
```

- URL が**リソースの住所**、メソッド(GET/POST/…)が**動詞**、結果は[ステータスコード](/posts/http-status-codes/)
- ステートレスで、`curl` やブラウザで**そのまま叩けて見える**のが強み

## gRPC — 関数を呼ぶ

gRPC は「**遠くのサービスの関数を、型付きで呼ぶ**」スタイル(RPC)。スキーマ(protobuf)を先に定義し、
そこからクライアント/サーバのコードを生成する。

```protobuf
service UserService {
  rpc GetUser(GetUserRequest) returns (User);
}
```

- URL やメソッドを設計するのではなく、**「呼びたい関数」を定義**する
- HTTP/2＋バイナリ(protobuf)で**高速・小さい**。型はスキーマから生成され厳格

> 🧭 Go でも .NET でも protobuf 定義から両言語のコードを生成でき、**言語をまたいで型を合わせられる**。
> C# の WCF を触ったことがあれば「契約(コントラクト)を先に決める」感覚が近い。

## 何が違うか

同じ「サービス間通信」でも、思想と得意分野が異なる。

| 観点 | REST | gRPC |
| --- | --- | --- |
| 設計の中心 | リソース(名詞)＋HTTPメソッド | 手続き(関数)＋メッセージ |
| 契約 | 緩い（OpenAPI等は任意） | protobuf で**必須・厳格** |
| データ形式 | テキスト(JSON) | バイナリ(protobuf) |
| ストリーミング | 不得意 | **双方向まで4様式**（下記） |
| ブラウザから直接 | そのまま可 | 原則不可（gRPC-Web が要る） |
| 人が叩く/デバッグ | 容易(curl) | 専用ツールが要る |

### gRPC の4つの通信様式

REST が基本「1リクエスト→1レスポンス」なのに対し、gRPC は流し続けられる。

- **Unary**: 1→1（普通の呼び出し）
- **Server streaming**: 1→多（進捗や一覧を流す）
- **Client streaming**: 多→1（分割アップロード等）
- **Bidirectional**: 多↔多（チャットやリアルタイム連携）

## どちらを選ぶか

用途で選ぶ。片方が常に正解ではない。

- **REST 向き**: 外部公開API、ブラウザ/モバイルが直接叩く、幅広い相手に手軽に提供したい
- **gRPC 向き**: [内部のサービス間通信](/posts/internal-service-grpc/)、低レイテンシ・高スループット、ストリーミング、多言語で型を揃えたい

> 💡 定番は「**外＝REST、内＝gRPC**」。外向きは手軽さと互換性、内向きは速度と型安全を取る。
> 1つのシステムで両方使い分けてよい。

> ⚠️ ブラウザから gRPC を直接は叩けない。フロント公開に gRPC を使うなら **gRPC-Web** やゲートウェイ経由が要る。
> ここを見落とすと「動かない」になりがち。

## まとめ

- REST は**リソース指向**（URL＋HTTPメソッド）、gRPC は**手続き指向**（型付き関数呼び出し）
- gRPC は **protobuf で契約が厳格**・バイナリで高速・**4つのストリーミング様式**を持つ
- REST は**手軽・ブラウザから直接・デバッグ容易**、gRPC はブラウザ直叩き不可（gRPC-Web が要る）
- 使い分けの定番は **外＝REST、内＝gRPC**。1システムで併用してよい
- 選ぶ軸は「相手が誰か（人/ブラウザ or サービス）」と「速度・型・ストリーミングの要否」
