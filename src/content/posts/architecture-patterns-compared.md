> 📚 シリーズ「Goで学ぶアーキテクチャとテスト」(2 / 7)

[前回](/posts/why-architecture-and-dip/)の「依存の方向を管理する」を、実際の形にしたのが
アーキテクチャパターン。代表的な3つを並べて違いを見る。

## レイヤードアーキテクチャ（N層）

最も基本的。層ごとに責務を分け、依存は上から下への一方通行。

```text
┌─────────────────────┐
│  Presentation 層     │  HTTP ハンドラ、リクエスト/レスポンス
├─────────────────────┤
│  Business Logic 層   │  ユースケース、ドメインルール
├─────────────────────┤
│  Data Access 層      │  DB 操作、外部 API 呼び出し
└─────────────────────┘
        依存の方向: 上 → 下
```

- **メリット**: 理解しやすい、分担しやすい
- **デメリット**: 下位層の変更が上位に波及しやすい（DB の都合がロジックに漏れる）

> 🧭 ASP.NET Core の典型構成 `Controller → Service → Repository` がまさにこれ。

## ヘキサゴナルアーキテクチャ（ポート & アダプター）

ビジネスロジックを中心に置き、外部との接続を **ポート（インターフェース）** と
**アダプター（実装）** で分離する。

```text
        ┌──── HTTP Adapter
        │
Port ── Core（ビジネスロジック）── Port
        │
        └──── DB Adapter
```

Core は外部を知らず、ポート越しにやり取りする。アダプターを差し替えれば技術を入れ替えられる。
[DIP](/posts/why-architecture-and-dip/) を構造にしたもの、と捉えるとわかりやすい。

## クリーンアーキテクチャ

ヘキサゴナルをさらに厳密にルール化したもの。**依存は必ず「外 → 内」**、内側は外側を知らない。
詳しくは次回扱う。

## 3つの比較

| パターン | 向き | 依存のルール |
| --- | --- | --- |
| レイヤード | 小〜中規模・シンプル | 上 → 下の一方通行 |
| ヘキサゴナル | 技術を差し替えたい | インターフェースで外部を抽象化 |
| クリーン | ロジックが複雑 | 依存は必ず外 → 内（内は外を知らない） |

> 💡 どれも目的は同じ ——「ビジネスロジックを外部技術（DB・Web・UI）から守る」。
> 規模と複雑さに合わせて選べばよく、いきなり最も厳密なものを選ぶ必要はない。

## まとめ

- レイヤードは最もシンプルだが、下位の変更が上位に波及しやすい
- ヘキサゴナルはロジックを中心に置き、ポート/アダプターで外部を抽象化
- クリーンは依存を「外→内」に固定した、より厳密な形
- 3つとも狙いは「ロジックを外部技術から守る」こと。規模で選ぶ
