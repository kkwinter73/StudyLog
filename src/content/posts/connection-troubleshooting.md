> 📚 シリーズ「ネットワークをちゃんと使う」(4 / 4)

シリーズの総まとめ。「繋がらない」と言っても原因は様々。鍵になるのが **refused と timed out の違い**。
ここまでの IP・ポート・[TCP](/posts/ports-and-tcp/)・[DNS](/posts/dns-basics-and-tools/) を総動員して、切り分けの順序を立てる。

## refused と timed out — 何が違う

このエラー2つは**原因が正反対**。ここを取り違えると調査が迷子になる。

| | connection refused | connection timed out |
| --- | --- | --- |
| 起きていること | 相手に**届いた**が断られた | 相手に**届かない**／返事が来ない |
| TCPの握手 | サーバーが拒否(RST)を即返す | [SYN](/posts/ports-and-tcp/)への返事が無く待ち続ける |
| 典型原因 | アプリが起動していない／そのポートで待ってない | ファイアウォール/SGで遮断・IP違い・経路無し |
| 体感 | **即座に**エラー | **数秒〜数十秒待って**からエラー |

> 💡 **「即拒否 = refused、待たされる = timed out」**で覚える。即答が返る時点で相手まで届いている。
> 待たされるのは、途中で握手の合図ごと捨てられている（＝間に壁がある）サイン。

## 切り分けの順序

繋がらない時は、上流から順に潰すと速い。

```text
1. 名前は引けてる?   → dig 名前 +short        （違うIP/解決不可なら DNS 問題）
2. 相手まで届く?     → ping 相手IP            （応答無しは経路/FW を疑う）
3. ポートは開いてる? → curl -v / nc -vz IP ポート（refused かtimeout かを見る）
4. サーバー側で待受? → ss -tlnp（サーバー上で） （0.0.0.0 で待ってるか）
```

- **refused** が出たら → サーバー側へ。アプリが起動し、そのポートで[待ち受けているか](/posts/ports-and-tcp/) `ss -tlnp`
- **timed out** が出たら → 間の壁を疑う。次のファイアウォール/SGへ

## ファイアウォール / セキュリティグループ

`timed out` の最頻原因がこれ。**通信が壁で黙って捨てられる**から、拒否すら返らず待ちぼうけになる。

- ファイアウォールは「どのIP/ポートからの通信を通すか」のルール集。許可されない通信は破棄される
- AWS の **セキュリティグループ(SG)** はインスタンス前の仮想ファイアウォール。
  「[`0.0.0.0/0`](/posts/ip-address-and-cidr/) から 443 を許可」のように [CIDR](/posts/ip-address-and-cidr/)＋ポートで書く

> ⚠️ 「ローカルでは動くのに本番(EC2)で timeout」の定番は **SG でそのポートが開いていない**。
> アプリやネットワーク経路より先に、SG のインバウンドルールを確認すると早い。

> 🧭 SG は「許可したものだけ通す(ホワイトリスト)」方式。既定では閉じていて、明示的に開けた分だけ通る。
> 「繋がらない＝そもそも開けてないのでは?」をまず疑うのが、この方式での鉄則。

## まとめ

- **refused = 届いたが拒否（即エラー）／ timed out = 届かない（待たされる）**。原因が正反対
- 切り分けは上流から: **DNS(dig) → 到達(ping) → ポート(curl/nc) → 待受(ss)**
- refused はサーバー側(アプリ起動・待ち受けポート)、timed out は間の壁(FW/SG)を疑う
- timed out の最頻原因は **ファイアウォール/SG の遮断**。許可ルールを CIDR＋ポートで確認
- 本番だけ繋がらないなら、まず **SG のインバウンドルール**を見る
