> 📚 シリーズ「AWSをちゃんと理解する」(1 / 6)

AWS のサービス名だけ見ても何者か分からない。このシリーズは **「なぜ要るのか → 何なのか → どう繋がるか」**
で概念から理解していく。1回目はネットワークと配信。

## なぜ「自分専用のネットワーク」が要るのか

サーバーをただインターネットに置くと、誰でも触れてしまう。そこで **自分専用の区画を作って囲い、
外に見せる部分だけ穴を開ける**。この区画が VPC。

## VPC とは（例え：囲った自分の敷地）

VPC は、広大な AWS の土地の中に **塀で囲った自分専用の敷地**。
使う IP アドレスの範囲を決め、その中にサーバーを置く。外との出入り口は自分で設計する。

### サブネット：敷地の中の区画分け

敷地内をさらに区画（サブネット）に分ける。重要なのは2種類。

- **パブリックサブネット** = 道路に面した区画。外から見えてよいもの（ALB など）を置く
- **プライベートサブネット** = 奥の区画。外から直接触られたくないもの（アプリ・DB）を置く

> 💡 アプリや DB を奥（プライベート）に隠し、入口（パブリック）だけ公開する。これが基本の守り方。

### NAT：奥の住人が「外に出かける」ための出口

プライベートの奥にいるアプリも、外部 API を叩いたり更新を取得したりで **外に出たい**ことがある。
でも外から入られたくはない。この「中から外へは行けるが、外から中へは入れない」出口が NAT。

| | 何か | 性質 |
| --- | --- | --- |
| **NAT Instance**（採用） | NAT 役を EC2 で自前運用 | 安いが冗長化・スケールは手動 |
| **NAT Gateway**（不採用） | AWS が用意するマネージドな NAT | 楽で高可用だが料金高め |

## 入口まわりのサービス

### ALB ＝ 入口の交通整理

**ALB（ロードバランサ）** は、来訪者を複数のアプリに振り分ける受付係。
混雑を分散し、`/api` はこっち…のように **パスで振り分け**もできる。

### CloudFront ＝ 近くのキャッシュ拠点（CDN）

毎回遠い本拠地まで取りに行くと遅い。**CloudFront** は世界中の拠点（エッジ）に内容を
**キャッシュして利用者の近くから配信**する。例えるなら近所のコンビニ受け取り。
**Lambda@Edge** は、そのエッジで小さな処理（リダイレクトやヘッダ書き換え）を挟む仕組み。

### Route 53 ＝ 住所の電話帳（DNS）

**Route 53** は `example.com` を正しい入口の場所へ変換する DNS。ドメインの名前解決を担う。

### ACM ＝ https の鍵

**ACM** は通信を暗号化する TLS 証明書（https の鍵）を発行・**自動更新**してくれる。
ALB や CloudFront に紐付けて使う。

## どう繋がるか

```text
ユーザー
  → Route 53        （名前を引いて入口を教える）
  → CloudFront      （近くから配信・ACM で https）
  → ALB             （交通整理して振り分け）
  → アプリ(プライベート)
        └─ 外部APIを叩くときは NAT 経由で外へ
```

## まとめ

- VPC ＝ 自分専用の囲われたネットワーク。中をサブネットで区画分けする
- 公開してよいものはパブリック、隠したいもの（アプリ・DB）はプライベートに置く
- NAT は「中から外へは出られるが外からは入れない」出口
- ALB は交通整理、CloudFront は近くのキャッシュ配信、Route 53 は DNS、ACM は https の鍵
- 流れは Route 53 → CloudFront → ALB → アプリ
