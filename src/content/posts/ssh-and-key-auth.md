[EC2](/posts/aws-compute-explained/) などのリモートサーバーに入る標準手段が **SSH**。
その認証の主役 **公開鍵認証** の仕組みと、実際の接続・つまずきどころを押さえてシリーズを締める。

## SSH とは

SSH は、リモートのマシンに**暗号化された通信**で安全にログインし、コマンドを実行する仕組み。

```bash
ssh alice@203.0.113.10        # ユーザー alice でそのIPに接続
ssh -i ~/.ssh/mykey.pem ec2-user@<EC2のIP>   # 鍵ファイルを指定して接続
```

通信は暗号化されるので、途中で盗み見されても中身は読めない。問題は「**本人かどう確かめるか**」——
ここでパスワードより安全な **公開鍵認証** が使われる。

## 公開鍵認証の仕組み

鍵を **秘密鍵（手元）** と **公開鍵（サーバーに置く）** のペアで作る。この2つは数学的に対応している。

| 鍵 | 置き場所 | 役割 |
| --- | --- | --- |
| 秘密鍵 (private) | **自分の手元だけ**（絶対に渡さない） | 自分が本人だと証明する |
| 公開鍵 (public) | 接続先サーバー（`~/.ssh/authorized_keys`） | 手元の秘密鍵と対応するか検証する |

> 💡 「公開鍵＝南京錠、秘密鍵＝その鍵」。錠（公開鍵）はサーバーにいくつ配ってもいいが、
> それを開けられる鍵（秘密鍵）は自分だけが持つ。だからパスワードを送らずに本人確認できる。

接続時、サーバーは公開鍵で「この相手は対応する秘密鍵を持っているか？」を確かめる。
秘密鍵そのものはネットワークに流れないのが安全な点。

## 鍵を作って接続する

```bash
# 1. 鍵ペアを作る（ed25519 が今の推奨）
ssh-keygen -t ed25519 -C "alice@example.com"
#   → ~/.ssh/id_ed25519（秘密鍵） と id_ed25519.pub（公開鍵）ができる

# 2. 公開鍵をサーバーに登録する
ssh-copy-id alice@203.0.113.10
#   （手動なら .pub の中身をサーバーの ~/.ssh/authorized_keys に追記）

# 3. 以降は鍵だけで入れる
ssh alice@203.0.113.10
```

> 🧭 EC2 は起動時にキーペアを指定し、AWS が公開鍵をインスタンスに登録してくれる。
> 手元に落とした `.pem`（秘密鍵）を `ssh -i key.pem ec2-user@...` で使う、という流れになる。

## ~/.ssh/config とハマりどころ

接続先が増えたら `~/.ssh/config` に名前を付けると `ssh 名前` だけで入れる。

```text
Host myec2
    HostName 203.0.113.10
    User ec2-user
    IdentityFile ~/.ssh/mykey.pem
```

```bash
ssh myec2     # 上の設定で接続
```

> ⚠️ **秘密鍵のパーミッションが緩いと SSH は接続を拒否する**（他人に読めるのは危険なため）。
> `chmod 600 ~/.ssh/mykey.pem` にする。[前回のパーミッション](/posts/linux-filesystem-and-permissions/)が効いてくる。

## まとめ

- SSH は**暗号化通信でリモートにログイン**する標準手段（EC2 接続もこれ）
- 認証の主役は **公開鍵認証**：秘密鍵は手元だけ、公開鍵をサーバーに置く
- 秘密鍵はネットワークに流れない。だからパスワードより安全
- `ssh-keygen` で作り、公開鍵を `authorized_keys` に登録、以降は鍵で入る
- `~/.ssh/config` で接続を名前付け。**秘密鍵は `chmod 600`** でないと拒否される
