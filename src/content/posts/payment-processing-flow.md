「決済ボタンを押したら課金される」の裏側を整理する。要点は、**アプリは生のカード番号を扱わず、
決済代行(PSP)に任せる**こと、そして決済は**与信→確定の2段階**で進むこと。登場人物・状態遷移・
冪等性・Webhook の順に見ていく。

## 登場人物

アプリが自分でお金を動かすわけではない。間に何者かが立つ。

| 登場人物 | 役割 |
| --- | --- |
| 顧客 / カード | 支払う人。カード情報を入力する |
| 自社アプリ | 注文を管理する。**カード番号は持たない**のが基本 |
| PSP（決済代行） | Stripe など。カード情報を預かり、ネットワークと話す |
| カードネットワーク / 発行銀行 | 残高・与信枠を確認し、最終的に承認/拒否する |

> ⚠️ 生のカード番号(PAN)を自社で保存・通過させると **PCI DSS** の重い準拠義務が発生する。
> フロントで PSP に直接送って**トークン化**し、アプリはそのトークンだけ扱うのが定石。

## 与信(authorization)と確定(capture)

カード決済は1回の操作に見えて、内部は2段階に分かれていることが多い。

```text
1. 与信 (authorization)
   - カード枠を「確保」する。まだお金は動かない
   - 在庫確保や本人確認のタイミングで行う

2. 確定 (capture)
   - 確保した枠から実際に「請求」する
   - 商品発送など、提供が確定したタイミングで行う
```

- **即時確定**（auth と capture を同時）にもできるが、物販では「発送時に確定」が安全
- 与信したまま確定しなければ、枠は一定期間後に自動で開放される
- 返金(refund)は確定後の取り消し、与信の取り消しは void と呼び分ける

> 💡 ホテルの「デポジット」と同じ発想。チェックイン時に枠を押さえ(auth)、チェックアウト時に
> 実額を請求する(capture)。

## 冪等性 — 二重課金を防ぐ

決済で最も怖いのが**二重課金**。ネットワークが不安定でレスポンスを取りこぼし、
クライアントがリトライすると、同じ決済が2回走りうる。

これを防ぐのが **冪等キー(idempotency key)**。リクエストごとに一意なキーを付け、
サーバは「このキーは処理済み」を記録して2回目以降は最初の結果を返す。

```go
// 注文ごとに1つのキーを生成し、リトライでは同じキーを使い回す
req.Header.Set("Idempotency-Key", orderID) // 例: 注文IDなど安定した一意値

// サーバ側のイメージ: キーで結果を覚えておき、再送には保存済みの応答を返す
if res, ok := store.Get(key); ok {
    return res // 課金は再実行しない
}
res := charge(req)
store.Save(key, res)
return res
```

> 🧭 C# でも考え方は同じ。HttpClient のリトライ(Polly など)を入れるときは、
> **リトライするリクエストに同じ冪等キーを乗せる**のがセット。キー無しのリトライは二重課金の元。

## 非同期確定と Webhook

決済の結果は、必ずしも HTTP レスポンスで同期的に返るとは限らない。
3Dセキュアの認証、銀行振込、コンビニ払いなどは**後から結果が確定**する。

そこで PSP は **Webhook**（イベント通知）でアプリに「確定した/失敗した」を後追いで知らせる。

- アプリは即時レスポンスを**確定の証拠にしない**。`payment_intent.succeeded` のような通知で確定扱いにする
- Webhook は**重複して届く**前提で、こちらも冪等に処理する（イベントIDで既処理を弾く）
- 通知が**なりすまし**でないか、PSP の署名を必ず検証する

> ⚠️ 「フロントが成功画面を出した＝課金成功」ではない。**真実源は PSP からの確定イベント**。
> 注文を発送可能にするのは Webhook を受けてから。

## 注文の状態遷移

ここまでをアプリ側の状態として持つと、流れが1本の線になる。

```text
pending（注文作成・与信待ち）
  → authorized（与信OK・確保済み）
    → captured（確定・請求済み）→ 発送可
    → canceled（与信取消 void）
  → failed（与信NG）
refunded（確定後の返金）
```

- 状態は**前進のみ**を基本にし、Webhook で受けたイベントで遷移させる
- 各遷移を記録(イベントログ)しておくと、問い合わせや突合(reconciliation)で効く

## まとめ

- アプリは**カード番号を持たず**、PSP にトークン化を任せる（PCI DSS 回避）
- 決済は**与信(auth)→確定(capture)の2段階**。物販は発送時に確定が安全
- リトライによる**二重課金は冪等キー**で防ぐ。結果を覚え、再送には同じ応答を返す
- **真実源は PSP の確定イベント(Webhook)**。即時レスポンスや成功画面を確定根拠にしない
- アプリ側は**注文の状態遷移**で管理し、各イベントを記録して突合に備える
