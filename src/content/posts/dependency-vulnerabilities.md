自分で書いたコードは無傷でも、依存しているライブラリに**既に公開された脆弱性**があれば
そこから破られる。攻撃者にとってこれは一番安い攻め口だ——脆弱性は公開済みで、対象は世界中に
たくさんいる。この記事は、その「既知のCVE」の仕組みと、Go の `govulncheck` で塞ぐ話。

## 攻撃の仕組み — 公開済みの穴を拾って回る

ゼロデイ（未知の穴）を自力で見つけるのは高くつく。攻撃者の多くはもっと楽をする。
**公開済みの脆弱性を、まだ直していないサービスに対して片っ端から試す**のだ。

- **CVE**（Common Vulnerabilities and Exposures）は個々の脆弱性に振られる世界共通の背番号
  （例: `CVE-2023-12345`）。「どのライブラリのどのバージョンが危ないか」がここに紐づく
- **CVSS** はその深刻度スコア（0〜10）。9.0以上は Critical で、優先して直す目安になる
- **セキュリティアドバイザリ**（GitHub Advisory / Go の脆弱性DB など）が
  「影響範囲・修正版・回避策」をまとめて公開する

やっかいなのは**推移的依存（transitive dependency）**。あなたが直接 import していなくても、
依存ライブラリがさらに依存する孫・ひ孫のパッケージに穴があれば、それはあなたのアプリの穴になる。

```text
your-app
└── github.com/foo/bar        ← 自分が入れたのはこれだけ
    └── github.com/baz/qux    ← bar が依存
        └── golang.org/x/...  ← ここに CVE があると、あなたも影響を受ける
```

> ⚠️ 「うちは危ないライブラリなんて使ってない」の大半は、直接依存しか見ていない。
> 穴は**依存ツリーの奥**に潜む。手で追いきれる量ではない。

## 対策の原則 — 新しく保ち、常に測る

依存の脆弱性対策は、派手な防御ではなく**地味な運用**で決まる。

- **依存を最新に保つ**: 修正版が出たら上げる。溜めるほど一気に上げるのが怖くなり、放置が進む
- **常時スキャンする**: 手動チェックは忘れる。CIに組み込んで**毎回自動で測る**
- **依存を増やしすぎない**: 依存が少ないほど攻撃面（拾いうる CVE の数）が小さい
- **速く上げられる道を用意する**: バージョン更新が一発で通るよう、テストとCIを整えておく
- **「入っている」と「実際に危ない」を区別する**: 依存ツリーに脆弱なパッケージがあっても、
  その脆弱な**関数を呼んでいない**なら実リスクは低い。ここを見分けられると騒音が激減する

> 💡 最後の「呼んでいるか」の区別が肝。単なる有無で警告を出すツールは誤検知だらけになり、
> やがて誰もアラートを見なくなる。**到達可能性（reachability）**まで見たい。

## Goでの防御 — govulncheck は「呼んでいる穴」だけ報告する

Go 公式の `govulncheck`（`golang.org/x/vuln`）は、**Go 脆弱性DB**と**静的解析**を組み合わせる。
依存ツリーに脆弱性があるかを調べるだけでなく、**あなたのコードがその脆弱な関数を実際に
呼び出す経路があるか**まで解析し、到達するものだけを報告する。だから警告が少なく、実効的だ。

```bash
# インストール（一度だけ）
go install golang.org/x/vuln/cmd/govulncheck@latest

# プロジェクト全体をスキャン
govulncheck ./...
```

呼んでいない脆弱性は情報として控えめに、**呼んでいる**脆弱性は「どのコードから届くか」の
呼び出し経路つきで出る。直し方は基本、修正版へ上げるだけ。

```bash
# 該当モジュールを最新に上げる
go get -u golang.org/x/text
go mod tidy
```

CI に一行足せば、脆弱性が入り込んだPRをその場で止められる。

```yaml
# GitHub Actions の1ステップ例
- name: govulncheck
  run: |
    go install golang.org/x/vuln/cmd/govulncheck@latest
    govulncheck ./...
```

さらに **Dependabot** を有効にすれば、修正版が出たときに更新PRを自動で立ててくれる。
「スキャンで見つける（govulncheck）」と「更新を運んでくる（Dependabot）」の両輪になる。

> 🧭 C#/.NET なら `dotnet list package --vulnerable` で既知の脆弱パッケージを一覧でき、
> NuGet の監査機能（`restore` 時の audit）でも警告が出る。ただし到達可能性まで見る
> govulncheck ほど絞り込まれず、「入っているか」寄りの判定になる点が違う。

## まとめ

- 攻撃者の一番安い攻め口は、あなたの古い依存にある**既に公開済みの脆弱性（CVE）**
- CVE=背番号、CVSS=深刻度スコア、アドバイザリ=影響と修正版のまとめ。**推移的依存**に穴が潜む
- 原則は**最新に保つ・常時スキャン・依存を減らす・速く上げる**、そして「入っている」と
  「実際に呼んでいて危ない」を区別すること
- Go は `govulncheck ./...` が Go脆弱性DB＋静的解析で**呼んでいる脆弱性だけ**を報告。
  CIに入れ、`go get -u` で更新、**Dependabot** で更新PRを自動化
- C# は `dotnet list package --vulnerable` ＋ NuGet audit。考え方は同じだが到達可能性までは見ない
