[push通知](/posts/push-notifications/)・[SMS通知](/posts/sms-notifications/)に続く通知系3本目。**メール通知**は
誰もがアドレスを持ち、**ほぼ無料でリッチな本文**を送れる。だが最難関は「送れる」ことではなく
**迷惑メール扱いされず受信箱に届く**こと——これが**到達性（deliverability）**だ。仕組み・送り方・
そして到達性の核 **SPF / DKIM / DMARC** 認証・バウンス処理・レピュテーションまでを押さえる。

## 仕組み — アプリから受信箱まで

自前のサーバから直接送ることもできるが、実務では **メール配信サービス（ESP）** を経由する四者リレー。

```text
①アプリサーバ ─API─▶ ②配信サービス ─SMTP─▶ ③受信側MTA ─▶ ④受信箱
 (自分のbackend)     (SendGrid/SES等)      (Gmail等の受信サーバ)
```

- **② ESP**（SendGrid / Amazon SES / Postmark 等）が、送信IPの評価管理・認証・バウンス処理を**代行**する
- **③ 受信側 MTA**（Gmail などの受信サーバ）が、認証とレピュテーションを見て**受信箱か迷惑メールか**を判定する
- 自前 SMTP で送ると IP 評価を一から育てる必要がある。だから ESP 経由が定番

> 💡 push/SMS と同じ四者モデルだが、決定的な違いは **③ 受信側が「受け取るか・どこに入れるか」を裁量で決める**こと。
> 「送信成功」と「受信箱に届いた」は別物で、そこを埋めるのが到達性の話。

## 送り方 — API で1通投げる

SMTP を直接話すより、ESP の **HTTP API** を使う方が楽で速い。Go で SendGrid 風の例:

```go
body := map[string]any{
    "personalizations": []any{map[string]any{
        "to": []any{map[string]string{"email": "user@example.com"}},
    }},
    "from":    map[string]string{"email": "noreply@myapp.com"},
    "subject": "パスワードを変更しました",
    "content": []any{map[string]string{"type": "text/plain", "value": "..."}},
}
// JSON を Authorization: Bearer <API key> 付きで POST
```

- 送信時点は「受付」。**バウンス・苦情・開封・クリック**は ESP が **Webhook（イベント通知）** で後追いする
- 受け口は[Webhook の実装](/posts/webhook-implementation/)と同じく、非同期に受けて記録・突合する
- 大量送信は**キュー経由**で流量を制御する（一気に送るとレピュテーションを落とす）

> 🧭 .NET の古い `SmtpClient` は非推奨。**SendGrid SDK** か **Azure Communication Services (Email)** を使う。
> Go も SDK か HTTP 直叩き。どちらも「送信＝受付、結果は Webhook」の二段構えは共通。

## 到達性の核 — SPF / DKIM / DMARC

受信側が「なりすましでない本物の送信者か」を検証する3点セット。**全部 DNS に設定**する。未設定だと迷惑メール直行。

| 仕組み | 何を証明するか | 置き場所 |
| --- | --- | --- |
| **SPF** | その IP から送ってよいと**ドメイン所有者が認めている** | DNS の TXT（許可IP一覧） |
| **DKIM** | 本文が**改ざんされておらず**、そのドメインが署名した | DNS の公開鍵＋メールに署名 |
| **DMARC** | SPF/DKIM 失敗時に**どう扱うか**の方針＋レポート | DNS の TXT（`p=none/quarantine/reject`） |

- **SPF** は「どのサーバが差出人ドメインを名乗ってよいか」を IP で宣言する
- **DKIM** は秘密鍵で署名し、受信側が DNS の公開鍵で検証する（改ざん・なりすまし検知）
- **DMARC** は上2つの結果と **From ヘッダのドメイン一致（アラインメント）** を見て、失敗時の処分（隔離/拒否）を指示する

> ⚠️ ESP を使う場合、**自ドメインで DKIM/SPF を設定**しないと ESP のドメイン名で届き、到達も信頼も下がる。
> 近年は Gmail/Yahoo が一定量以上の送信者に **DMARC 設定を事実上必須化**している。最初に必ず整える。

## バウンス・苦情・レピュテーション

認証を通しても、**送信者の評判（レピュテーション）**が悪いと受信箱に入らない。評判はフィードバックで決まる。

- **バウンス**: 宛先不達。**ハード**（存在しないアドレス＝恒久）と**ソフト**（一時的）。ハードは**即リストから除外**する
- **苦情（complaint）**: 受信者の「迷惑メール」ボタン。FBL（フィードバックループ）で通知され、**多いと評判が急落**
- **リスト衛生**: 無効・未同意の宛先を送り続けると評判が落ちる。**ダブルオプトイン**で質を保つ
- **購読解除**: `List-Unsubscribe`（ワンクリック解除）を付ける。解除しやすさが苦情を減らす
- **専用IPのウォームアップ**: 新しい送信IPは少量から徐々に増やして評判を育てる

> ⭐ 到達性は「一度設定して終わり」ではなく**運用**。ハードバウンスを溜めない・苦情率を監視する・
> 同意した宛先にだけ送る。これがレピュテーションを守り、結果的に受信箱に届き続ける唯一の道。

## push / SMS との使い分け

3つの通知チャネルは**コスト・即時性・リッチさ**で役割が違う。

| | メール | SMS | push |
| --- | --- | --- | --- |
| コスト | ほぼ無料 | 1通課金 | ほぼ無料 |
| 即時性・開封 | 遅め・見られないことも | 速い・高開封 | 速い（アプリ導入者のみ） |
| 表現力 | **リッチ（HTML/添付）** | 短文のみ | 短文＋簡単なデータ |
| 向く用途 | 領収書・お知らせ・長い内容 | OTP・重要連絡 | アプリ内の再訪促進 |

- **記録に残る/長い内容**（領収書、レポート、規約変更）はメール
- **今すぐ確実に**は SMS、**無料の反復接点**は push
- 実務は**併用＋フォールバック**（重要は SMS→ダメならメール、日常は push→未読ならメール）

## まとめ

- メール通知は**アプリ→配信サービス→受信側MTA→受信箱**の四者。最難関は「送信」でなく**到達性**
- 送信は API に POST、**結果（バウンス/苦情/開封）は Webhook** で後追い。送信成功≠受信箱到達
- 到達性の核は **SPF・DKIM・DMARC** を**自ドメインの DNS に設定**すること。未設定は迷惑メール直行
- **バウンス除外・苦情監視・オプトイン・ウォームアップ**でレピュテーションを守る＝運用そのもの
- **リッチ/記録はメール、確実な1回は SMS、無料反復は push**。用途で使い分けフォールバックを設計
