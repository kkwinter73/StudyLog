アプリを層に分けると「どこに何を書くか」が決まって迷わなくなる。よく出る4層——
**ドメイン / アプリケーション / プレゼンテーション / インフラ**——を、各層の役割と、
最後に一番間違えやすい **依存の向き** の順で押さえる。前提の考え方は
[なぜアーキテクチャが必要か（DIP）](/posts/why-architecture-and-dip/)を参照。

## 4層の全体像

まず一覧で「担当」と「置くもの／置かないもの」を掴む。

| 層 | 担当（一言で） | 置くもの | 置かないもの |
| --- | --- | --- | --- |
| プレゼンテーション | 外との入出力 | HTTPハンドラ、リクエスト/レスポンス変換 | 業務ルール |
| アプリケーション | ユースケースの調整 | 処理の流れ、トランザクション境界 | 業務ルールの中身・SQL |
| ドメイン | 業務ルールの核 | エンティティ、値オブジェクト、判断 | DB・HTTP・フレームワーク |
| インフラ | 技術の詳細 | DBアクセス、外部API、メール送信 | 業務判断 |

> ⭐ 迷ったら「これは**業務の話**か、**技術の話**か」で分ける。業務はドメイン、技術はインフラへ。

## ドメイン層 — 業務ルールの核

そのシステムが表現する **業務そのもの**。「注文の合計を出す」「在庫が足りるか判定する」
といったルールと、それを持つ**エンティティ**・**値オブジェクト**が住む。

```go
// 業務ルールはドメインの中に置く。DBもHTTPも知らない
type Order struct {
    Items []Item
}

func (o Order) Total() Money {  // 「合計の出し方」は業務ルール
    var sum Money
    for _, it := range o.Items {
        sum = sum.Add(it.Price.Mul(it.Qty))
    }
    return sum
}
```

> 🧭 C# でも同じ。ドメインの型に **EF Core の属性や `[JsonProperty]` を付けない**のが理想で、
> 永続化やシリアライズの都合はドメインに漏らさない。

## アプリケーション層 — ユースケースの調整役

「注文を確定する」のような **1つのユースケース** を実現する層。ドメインを呼び出して流れを組み立て、
**トランザクションの境界**を引くのが仕事。**業務ルールそのものは持たず**、順番に段取りするだけ。

```go
func (uc *OrderUseCase) Confirm(ctx context.Context, id OrderID) error {
    order, err := uc.repo.Find(ctx, id)   // 取得はインフラに任せる（下記）
    if err != nil { return err }
    if !order.CanConfirm() {              // 判断はドメインに委ねる
        return ErrNotConfirmable
    }
    return uc.repo.Save(ctx, order)       // 保存もインフラへ
}
```

> 💡 「調整役」なので、ここが厚く（if だらけに）なってきたら、判断がドメインに漏れているサイン。
> ルールはドメインへ引き戻す。

## プレゼンテーション層 — 外との入出力

ユーザーやクライアントとの **接点**。HTTP リクエストを受け、パース・形式チェック・認証をして
アプリケーション層を呼び、結果を JSON などに整形して返す。**UI/API の都合はここで吸収**する。

```go
func (h *Handler) Confirm(w http.ResponseWriter, r *http.Request) {
    id := OrderID(r.PathValue("id"))          // 入力を取り出す
    if err := h.uc.Confirm(r.Context(), id); err != nil {
        http.Error(w, err.Error(), 400)       // 出力の形に変換
        return
    }
    w.WriteHeader(http.StatusNoContent)
}
```

## インフラ層 — 技術の詳細

DB アクセス、外部 API、メッセージング、メール送信といった **技術的な実装**。上位層が定義した
**インターフェースを実装する**側で、業務判断は持たない。

```go
// repo のインターフェースは上位（アプリ/ドメイン）が定義し、実装をここに置く
type PostgresOrderRepo struct{ db *sql.DB }

func (r *PostgresOrderRepo) Find(ctx context.Context, id OrderID) (*Order, error) {
    // SQL などの詳細はこの層に閉じ込める
}
```

## 依存の向き — ここが一番間違えやすい

層に分けても、**依存の矢印が正しく向いていないと意味がない**。2つの流儀がある。

| 流儀 | 依存の向き | ドメインの立場 |
| --- | --- | --- |
| 素朴なレイヤード | 上→下（プレゼン→アプリ→ドメイン→インフラ） | インフラに依存してしまう |
| DDD/クリーン/オニオン | 外→内（インフラも内側に依存） | 何にも依存しない**中心** |

素朴な積み重ねだと、ドメインが DB（インフラ）に依存してしまい「DB を替えるとドメインも壊れる」。
そこで [依存性逆転(DIP)](/posts/why-architecture-and-dip/) を使い、**リポジトリのインターフェースを
アプリ／ドメイン側に置き、実装をインフラに置く**。すると矢印が内向きに揃い、ドメインが独立する。

> ⚠️ 「層に分けた＝クリーン」ではない。**インターフェースを内側に置いて依存を逆転**させて初めて、
> ドメインが技術から守られる。詳しくは[同心円と依存性のルール](/posts/clean-architecture-concepts/)。

## まとめ

- 4層は担当で分ける: **プレゼン(入出力) / アプリ(調整) / ドメイン(業務ルール) / インフラ(技術)**
- 迷ったら「**業務の話か技術の話か**」。業務はドメイン、技術はインフラへ
- アプリケーション層は**流れを組むだけ**。判断が増えたらドメインへ引き戻す
- 最重要は**依存の向き**。素朴な上→下だとドメインがインフラに依存してしまう
- [DIP](/posts/why-architecture-and-dip/) でインターフェースを内側に置き、依存を内向きに逆転させる
- 実際の Go 実装は[クリーンアーキテクチャをGoで実装する](/posts/clean-architecture-in-go/)へ
