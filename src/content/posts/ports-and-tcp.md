> 📚 シリーズ「ネットワークをちゃんと使う」(2 / 5)

IPアドレスが「どのマシンか」なら、**ポート番号**は「そのマシンのどのアプリか」。
ポートの確認方法と、TCP接続が始まる **3ウェイハンドシェイク** を押さえると、接続トラブルの読みが効く。

## ポート番号とは

1台のマシンでは Web・DB・SSH など複数のアプリが同時に動く。IPで届いたデータを
**どのアプリに渡すか**を決める番号がポート。`192.168.1.10:8080` の `:8080` がそれ。

| ポート | 用途 |
| --- | --- |
| 22 | SSH |
| 80 | HTTP |
| 443 | HTTPS |
| 5432 | PostgreSQL |

> 💡 IP＝建物の住所、ポート＝部屋番号。`IP:ポート` の組で「どのマシンのどのアプリか」が一意に決まる。

> 🧭 .NET の Kestrel が `localhost:5000`、Go の `http.ListenAndServe(":8080", ...)` で待つのも同じ話。
> アプリは「このポートで待ち受ける」と宣言して、そこに来た通信を受ける。

## 待ち受けの確認 — ss -tlnp

「アプリがちゃんとそのポートで待っているか」は `ss` で確認できる（旧 `netstat` の後継）。

```bash
ss -tlnp
# -t TCP  -l 待ち受け(LISTEN)のみ  -n 名前解決せず数字表示  -p プロセス名も表示
```

```text
State   Local Address:Port   Process
LISTEN  0.0.0.0:8080         users:(("myapp",pid=1234))
LISTEN  127.0.0.1:5432       users:(("postgres",pid=987))
```

> ⚠️ `0.0.0.0:8080` は全インターフェースで受付（外部から届く）、`127.0.0.1:5432` は
> **自分自身からのみ**。「ローカルでは繋がるのに外から繋がらない」時はここを最初に疑う。

## TCP通信の流れ — 3ウェイハンドシェイク

TCP は通信前に、両者で**3往復の挨拶**をして接続を確立する。これが3ウェイハンドシェイク。

```text
クライアント        サーバー
    │ ── SYN ──────▶ │   「繋ぎたい」
    │ ◀── SYN+ACK ── │   「いいよ、こちらも準備OK」
    │ ── ACK ──────▶ │   「了解、開始」
    └──── 確立。データ送受信へ ────┘
```

- この握手が成立して初めてデータが流れる。途中で相手が応じなければ接続は始まらない
- 「相手はいるが拒否」か「相手に届かない」かで、握手のどこで失敗するかが変わる
  （その違いが[次回の切り分け](/posts/connection-troubleshooting/)に直結する）

## まとめ

- **ポート番号**は「マシン内のどのアプリか」を指す。`IP:ポート` で宛先が一意に決まる
- 22=SSH / 80=HTTP / 443=HTTPS など定番ポートは覚えておくと速い
- **`ss -tlnp`** で待ち受け確認。`0.0.0.0`(外部可) と `127.0.0.1`(自分のみ) の違いに注意
- TCP は **3ウェイハンドシェイク**(SYN→SYN+ACK→ACK) で接続を確立してからデータを流す
- 握手のどこで失敗するかが、接続トラブルの種類の違いになる
