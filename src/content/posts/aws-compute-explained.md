> 📚 シリーズ「AWSをちゃんと理解する」(2 / 6)

[前回](/posts/aws-networking-explained/)の入口の奥にいる「アプリ」を、実際に **どこで動かすか**。
選択肢は大きく VM・コンテナ・関数の3つ。

## そもそも「コードを動かす場所」の3択

| 方式 | AWS | ざっくり |
| --- | --- | --- |
| VM（仮想サーバー） | EC2 | OS ごと1台借りる。何でもできるが管理も全部自分 |
| コンテナ | ECS / Fargate | アプリを箱に詰めて動かす。今回の主役 |
| 関数 | Lambda | コードの関数だけ置き、呼ばれた時だけ動く |

## コンテナとは（おさらい）

コンテナは **アプリ＋必要な依存をまとめた箱**。どの環境でも同じように動く。
Go なら[単一バイナリを小さなイメージに詰める](/posts/deploying-go-apps/)のと相性がよい。
問題は「その箱を本番で何台、どう動かし続けるか」——それを担うのが次の ECS。

## ECS / Fargate ＝ コンテナの司令塔

- **ECS** は「このコンテナを2台動かして、落ちたら立て直して」と **面倒を見る司令塔（現場監督）**。
- **Fargate** は、その土台となるサーバーすら意識しなくていいモード。
  コンテナを渡すだけで AWS が動かす場所を用意する。

> 💡 ECS（EC2 起動タイプ）だと土台の EC2 を自分で管理する必要があるが、**Fargate ならサーバー管理ゼロ**。
> 「現場監督＝ECS、作業場所もお任せ＝Fargate」と捉えると分かりやすい。

### Service Connect ＝ サービス同士を名前で繋ぐ

アプリが複数サービスに分かれると、互いを呼び合う必要が出る。だが台数やコンテナが入れ替わると
IP は変わる。**Service Connect** は各サービスに **名前**を与え、IP ではなく名前で通信させる仕組み
（サービスディスカバリ）。通信のメトリクスも取れる。

## Lambda ＝ 呼ばれた時だけ動く関数

**Lambda** は関数のコードだけ預けておき、リクエストやイベントが来た **時だけ起動**して、
動いた分だけ課金される。常駐サーバーを持たない。例えるなら人感センサー付きの照明。

> 🧭 .NET の Azure Functions に近い発想。Go でもハンドラ関数を書いてデプロイするだけで動く。

## 結局どう使い分けるか

```text
常駐する Web API・中〜大規模      → ECS / Fargate
軽い処理・低頻度・イベント駆動     → Lambda
特殊要件・OS まで細かく制御したい  → EC2
```

## まとめ

- コードを動かす場所は VM(EC2) / コンテナ(ECS) / 関数(Lambda) の3択
- ECS はコンテナの司令塔、Fargate は土台サーバーすら不要にするモード
- Service Connect はサービス同士を IP でなく名前で繋ぐ
- Lambda は呼ばれた時だけ動く関数。低頻度・イベント駆動に向く
- 迷ったら常駐は Fargate、軽い/イベントは Lambda
