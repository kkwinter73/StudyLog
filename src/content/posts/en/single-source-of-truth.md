**SoT (Single Source of Truth)** is a design idea: for any given piece of information, keep the "canonical original" in **exactly one place**, and have everything else reference it (or be generated from it). It's unglamorous, but it pays off in a lot of situations.

## Why you need it: duplicated data always drifts

If you write the same fact in two places, someone fixes only one of them and they **drift apart**. Once that happens, nobody can tell "which one is right?"

```text
Hard-code the tax rate in both A.go and B.go
  → a law changes and you fix only A
  → B stays stale
  → the same app produces conflicting results 💥
```

> ⚠️ The scary part isn't things "breaking" — it's **silent drift**. No error fires, and the wrong value just keeps getting used.

## What SoT is (analogy: only one original copy)

For a contract, you make only one "original," and everything else is a copy. Copies are just reproductions of the original, and changes are **always made to the original**. SoT is the same:

- The canonical version of a piece of information lives in **one place** (the source of truth)
- Everywhere else **references** it, or is **generated** from it
- When you fix something, you fix **only the source of truth**

## Benefits

| Benefit | Why |
| --- | --- |
| No drift | With one canonical version, there are no contradictions |
| One place to change | You don't hunt around for every spot to fix |
| Trustworthy | Everyone shares "look here and it's correct" |
| A starting point for automation | You can generate and verify derivatives from the source |

## Concrete examples: code, infrastructure, docs

**Code** — define constants and config in one place and import them.

```go
// ❌ hard-coded all over the place → drift
price := base * 1.10 // 0.10 here too
total := sub * 1.10  // and 0.10 over there too

// ✅ one source of truth
package tax
const Rate = 0.10 // the canonical tax rate lives here only; change this one line
```

> 🧭 This is also DRY ("Don't Repeat Yourself") in practice. Putting a type definition or enum in one place and reusing it everywhere is the same idea.

**Infrastructure** — with IaC (Terraform and the like), you write the "desired state" as code and make that the source of truth. Instead of poking at AWS by hand, you edit the code and apply it (manual changes create drift from the source of truth).

**Docs** — keep "why we decided this" in a decision log (ADR), in one place only. The trick is to **pick a source of truth per kind of thing**: specs live in code, progress lives in Issues.

## Generate derivatives, don't "copy" them

When you need a copy from the source of truth, **generate** it rather than copying by hand. Hand copies always fall out of sync eventually.

```text
Source of truth (one) ──generate──▶ derivative A (automatic)
                       └─generate──▶ derivative B (automatic)
Only the source of truth is edited. Derivatives can just be rebuilt.
```

## How this blog does it

This StudyLog itself is actually built around SoT.

- The canonical **frontmatter schema** for articles lives only in `src/content.config.ts`
- The canonical **site config** (title, nav) lives only in `src/site.ts`
- The canonical **decisions** live in `docs/decisions/`, and the index is [auto-generated by a script](/en/posts/why-architecture-and-dip/) (never hand-edited)

> 💡 Deciding "kind → the one place it lives" up front means you never wonder where to write something, and nothing drifts later.

## Summary

- SoT = "the canonical version of a piece of information lives in exactly one place." Everything else references or generates from it
- The danger of duplication is "silent drift." SoT prevents it at the root
- The benefits: no drift, one place to change, trustworthy, a starting point for automation
- Centralize constants/types in code, use IaC for infrastructure, and pick a source of truth per kind for docs
- Don't hand-copy duplicates — generate them. You always fix only the source of truth
