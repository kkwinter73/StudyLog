フロントエンドの JS・CSS・画像は、書いたままの姿でブラウザに届くわけではない。
**変換 → 結合 → 圧縮 → 指紋付け** という一連の加工ラインを通って「配信に都合のいい形」に作り替えられる。
このラインが**アセットパイプライン**だ。各工程が何の問題を解決しているのかを順に見る。

## 全体像 — src/ から dist/ への加工ライン

```text
src/（人間が書きやすい形）                          dist/（ブラウザ・ネットワークに都合のいい形）

 main.ts     ── 変換(transpile) ──┐
 style.scss  ── 変換 ─────────────┼── 結合(bundle) ── 圧縮(minify) ── 指紋(hash) ──▶  app-3f9c2b.js
 logo.png    ── 最適化 ───────────┘                                                  style-8ad1e0.css
                                                                                     logo-77b1c4.png
```

出発点は「**人間が書きやすい形**と**配信に効率のいい形**は別物」という事実。
TypeScript や SCSS はブラウザが直接読めないし、ファイルが細かいままだとリクエストが増え、
コメントや空白はただの無駄なバイトになる。その差を埋めるのがパイプラインの仕事で、
実体は Vite / webpack / esbuild といったビルドツールだ。

> 🧭 C# では ASP.NET の bundling & minification が同じ役割。「Asset Pipeline」という言葉自体は Rails 発祥。

## 変換 — 書きやすい形から動く形へ

最初の工程は、開発用の言語・記法をブラウザが実行できる形に落とすこと。

| 入力（書く形） | 出力（動く形） | 変換の名前 |
| --- | --- | --- |
| TypeScript | JavaScript | トランスパイル |
| SCSS / Sass | CSS | プリプロセス |
| 最新の JS 構文 | 古いブラウザでも動く JS | トランスパイル（ダウンレベル） |
| JSX / .vue / .astro | JS + HTML | コンパイル |

> 💡 「コンパイル」との違いは気にしなくていい。どれも「ソース → 実行できる形」への変換で、機械語ではなく別のテキストを吐くだけ。

## 結合と圧縮 — リクエスト数とバイト数を減らす

変換したファイルは、そのままだと import のたびに1リクエストずつ飛んで遅い。そこで:

- **結合（bundle）**: import の依存グラフを辿り、少数のファイルにまとめる
- **ツリーシェイキング**: export されているが**どこからも使われていない**コードを落とす
- **圧縮（minify）**: 空白・コメントを消し、変数名を1文字に縮める。動きは変えずバイトだけ減らす

> ⚠️ minify と gzip/brotli は別物。minify はビルド時に「コードそのもの」を縮め、gzip はサーバが配信時に「転送」を縮める。両方かけるのが普通。

なお HTTP/2 以降は多重化が効くので「全部1ファイルに」が最適とは限らず、
画面ごとに分割して必要な分だけ読む **code splitting** も使われる。

## 指紋付け — キャッシュ問題を「名前」で解く

パイプラインの中で一番効く工程がこれ。ファイル内容のハッシュをファイル名に埋め込む
（`app.js` → `app-3f9c2b.js`）。これを**フィンガープリント**と呼ぶ。

- 名前に内容のハッシュが入っている → **内容が変わればファイル名（URL）が変わる**
- だから古い URL のキャッシュは無効化する必要がない。**新しい HTML が新しい URL を指すだけ**
- 結果、アセットには `Cache-Control: max-age=31536000, immutable`（1年・不変）を付けられる

> ⭐ 「内容が変われば URL が変わる」— この一点で「キャッシュはいつ消すか」問題が消える。長期キャッシュと即時更新が両立する。

元の名前とハッシュ付きの名前の対応表が**マニフェスト**（`manifest.json`）。
HTML やテンプレートはこの表を引いて `<script src="/assets/app-3f9c2b.js">` を出力する。
配信は[キャッシュ](/posts/caching-cache-aside/)と相性のいい [CDN](/posts/web-server-proxy-lb-cdn/) が定番だ。

## Go のサーバから見ると

Go 自体はアセットパイプラインを持たない。フロントのビルドツールに `dist/` を吐かせて、
Go は**配る係**に徹するのが普通だ。小さい構成ならバイナリに同梱してしまえる。

```go
//go:embed dist
var assets embed.FS

func main() {
	sub, _ := fs.Sub(assets, "dist")
	http.Handle("/assets/", http.FileServer(http.FS(sub)))
}
```

テンプレートから参照するときは、ビルドが吐いた `manifest.json` を起動時に読み、
`"main.js" → "app-3f9c2b.js"` の対応をテンプレート関数で引けるようにしておく。
CI では「フロントのビルド → Go のビルド」の順で走らせ、成果物をひとつの
[デプロイ](/posts/cicd-github-actions/)に載せる。

> 🧭 C#（ASP.NET Core）の `asp-append-version="true"` は `?v=<ハッシュ>` をクエリで付ける方式。ファイル名ごと変える方式と発想は同じで、「URL を内容に連動させる」点が共通。

## まとめ

- アセットパイプラインは「人間が書く形」→「配信に都合のいい形」への加工ライン。実体は Vite / webpack 等のビルドツール
- **変換**でブラウザが読める形にし、**結合・圧縮**でリクエスト数とバイト数を減らす
- **指紋付け**（内容ハッシュ入りファイル名）で、長期キャッシュと即時更新を両立する
- 元名→ハッシュ名の対応表が**マニフェスト**。サーバ／テンプレートはこれを引いて URL を出す
- Go はパイプラインを持たず配る係。`go:embed` で同梱するか、CDN に置く

**次にやること**: Vite で小さなプロジェクトをビルドし、`dist/` のファイル名とマニフェストを実際に観察する。
