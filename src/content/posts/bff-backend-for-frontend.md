**BFF（Backend for Frontend）** は、**クライアント種別ごとに専用のバックエンド**を置くパターン。
Web用・モバイル用のようにフロントごとにBFFを1つ持ち、下流サービスの**集約**と画面向けの**整形**を任せる。
この記事は、BFFが何を解決し、どこに置き、どんな代償があるのかを掴むためのもの。

## 何を解決するのか — 汎用API1本の辛さ

複数のクライアント（Web・iOS・Android）を、**1本の汎用API**でまかなおうとすると無理が出る。

- **過剰取得（over-fetch）**: モバイルには不要な項目まで返る。通信も描画も重い
- **過少取得（under-fetch）**: 1画面のために API を何回も叩く（chatty）。モバイル回線で遅い
- **関心事の漏れ**: 「この画面だけ表示を変える」ようなクライアント固有の都合が、共通APIかフロントに漏れて肥大化する

> ⭐ 汎用APIは「全クライアントの最大公約数」になりがちで、どのクライアントにも最適でない。
> BFFは発想を変え、**画面（クライアント）ごとに"ちょうどいい"バックエンド**を用意する。

## どこに置くのか — クライアントと下流サービスの間

BFFはクライアントと、実際の業務ロジックを持つ下流サービスの**間**に立つ。

```text
Web       ─▶ Web用BFF   ─┐
iOS       ─▶ iOS用BFF   ─┼─▶ 認証 / 注文 / 在庫 / … （下流のマイクロサービス群）
Android   ─▶ Android用BFF ┘
```

- 複数の下流サービスの呼び出しを**集約**し、1画面に必要なデータを**まとめて1レスポンス**で返す
- レスポンスを**その画面にちょうどいい形**に整形する（要らない項目を落とす／名前を寄せる）
- クライアント固有の関心事（認証・セッション・表示ロジック）を**BFFに寄せて**下流を汚さない

> 🧭 .NET でいう API Gateway / 集約層に近いが、要点は**"クライアント種別ごと"に割る**こと。
> 全クライアント共通の1ゲートウェイとは違い、BFFは Web用・モバイル用と**分けて持つ**のが本質だ。

## 利点 — 画面に最適化できる

- **ちょうどいいレスポンス**: 各画面に必要なデータだけを、必要な形で返せる。過剰・過少取得が消える
- **ラウンドトリップ削減**: 集約をサーバ側（下流に近い所）でやるので、クライアントの往復回数が減る
- **下流の変化を吸収**: 下流サービスの改修をBFFが吸収し、クライアントを守る（変更の緩衝材）
- **クライアント別の認証をBFFに寄せられる**: 例えばSPAのトークンをブラウザに置かず、BFF側の
  Cookieセッションで持つ（[Cookie/セッション](/posts/cookie-session-token/)）。漏洩面を減らす定番構成

## トレードオフ — 増えるものと肥大化

BFFはタダではない。置く前に代償を見ておく。

- **数が増える**: クライアントの数だけBFFを作り・デプロイ・運用する。小規模には過剰
- **重複**: BFF間で似たコードが重複しがち。共通化しすぎると今度は汎用APIの辛さに逆戻り
- **肥大化リスク**: 便利ゆえに業務ロジックまでBFFに溜まり、下流と責務が曖昧になる。BFFは**整形と集約**に留める
- **所有権**: 一般に**そのフロントを作るチーム**がBFFを持つ。誰の持ち物か曖昧だと放置される

> 💡 「クライアントが1種類だけ」「APIが単純」なら BFF は過剰。まず汎用APIで始め、
> クライアントごとの都合が衝突し始めたら分ける、が現実的。**GraphQL**（クライアントが必要な形を問い合わせる）
> で過剰・過少取得を解く手もあり、BFFの代替になることもある。

## まとめ

- BFFは**クライアント種別ごとに専用のバックエンド**を置くパターン。Web用・モバイル用と分ける
- 解決するのは汎用API1本の辛さ——**過剰取得・過少取得（chatty）・クライアント固有の関心事の漏れ**
- BFFの仕事は下流の**集約**と画面向けの**整形**、そしてクライアント別の認証などを**寄せる**こと
- 利点は「ちょうどいいレスポンス・往復削減・下流変化の吸収」。SPAのトークンをBFFで持つ構成も定番
- 代償は**数の増加・重複・肥大化・所有権**。小規模やクライアント1種なら過剰で、GraphQLが代替になることも
