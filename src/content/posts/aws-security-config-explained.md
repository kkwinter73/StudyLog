> 📚 シリーズ「AWSをちゃんと理解する」(5 / 6)

ここまでのサービスを安全に使うための土台。バラバラに見えるが、根っこは
**「鍵を持たない・秘密をコードに書かない・暗号化する」** の3つに集約できる。

## IAM ＝ 誰が何をできるかの許可証

**IAM** は AWS のセキュリティの中心。「誰が・何に・何をしてよいか」を **ポリシー**で定義する。
原則は **最小権限**（必要な操作だけ許可）。

重要なのが **ロール**。アプリに固定の鍵を持たせる代わりに、ロールを割り当てると
**SDK が一時的な認証情報を自動取得**する。これで「鍵を持たない」が実現する。

> ⚠️ アクセスキーをコードや Git に書かない。EC2 / ECS / Lambda には IAM ロールを割り当てる。

### IAM Identity Center ＝ 人間のログインをまとめる

IAM が「アプリやサービスの権限」中心なのに対し、**Identity Center** は **人間のログイン**を束ねる
SSO・複数アカウント管理。社員が1つのログインで必要なアカウントに入れるようにする。

## 秘密と設定の置き場所

設定値や秘密を **コードに埋め込まず、外から渡す**ための置き場所。

| | 何を置くか | 特徴 |
| --- | --- | --- |
| **Secrets Manager** | DB パスワード等の機密 | 自動ローテーション対応 |
| **Parameter Store** | 設定値・ちょっとした秘密 | 安価（SSM の一部） |

> 🧭 迷ったら ——「機密＋自動更新が要る → Secrets Manager」「ただの設定値・安く → Parameter Store」。

## SSM ＝ サーバー運用の道具箱

**SSM（Systems Manager）** は運用ツールの集まり。代表が **Session Manager** で、
**SSH の鍵を使わずにサーバーへ安全に入れる**（踏み台や鍵管理が要らなくなる）。パッチ適用なども担う。

## KMS ＝ 暗号鍵の管理

**KMS** はデータを暗号化する **鍵そのものを安全に管理**する。S3 や RDS の暗号化に使われる。
この構成の dev では **CMK（自分で管理するカスタム鍵）は使わず**、AWS 管理の鍵で済ませている。

## WAF ＝ Web 攻撃を入口で弾く

**WAF（Web Application Firewall）** は SQL インジェクションや XSS のような
**Web アプリ層の攻撃を入口でブロック**する。この構成の dev では **不採用**（本番で検討する類）。

## まとめ

- 通底するのは「鍵を持たない・秘密を書かない・暗号化する」
- IAM は権限の許可証。アプリにはロールを割り当て鍵を持たせない
- Identity Center は人間のログイン(SSO)を束ねる
- 機密は Secrets Manager、設定値は Parameter Store
- SSM は鍵なし接続などの運用道具、KMS は暗号鍵の管理（dev は CMK 不使用）
- WAF は Web 攻撃を入口で防ぐ（dev は不採用）
