> 📚 シリーズ「AWSをちゃんと理解する」(4 / 6)

サービスが増えると「直接呼び合う」だけでは辛くなる。ここでは **非同期**の考え方と、
それを支える4サービスを整理する。

## なぜ非同期が要るのか

リクエストの中で重い処理（メール送信・画像変換・集計）まで全部やると、**遅く・失敗に弱い**。
そこで「**依頼だけ受け取って即返し、処理は後で**」にする。これが非同期。

```text
同期:   ユーザー →[受付〜重い処理まで全部]→ 返事（遅い）
非同期: ユーザー →[受付して即返事]            （速い）
                    └→ 後ろで重い処理を実行
```

## SQS ＝ 仕事を貯める待ち行列（キュー）

**SQS** は仕事を順番に貯める待ち行列。送る側はキューに入れて即終了、受け手が後で取り出して処理する。
急な大量依頼も貯めて捌けるし、送る側と受ける側が **直接繋がらない（疎結合）**。
例えるなら飲食店の注文票。

## SNS ＝ 一斉配信（pub/sub 通知）

**SNS** は1つの出来事を **複数の宛先へ一斉に配る**（ファンアウト）。トピックに publish すると、
購読している全員へ届く。例えるなら一斉放送やメーリングリスト。

> 💡 定番の組み合わせが **SNS → SQS**。SNS で複数チームに配り、各チームは自分の SQS で受けて
> 自分のペースで処理する。「配る」と「貯める」の合わせ技。

## EventBridge ＝ 条件で振り分けるイベントバス

**EventBridge** は「**どんな出来事が起きたら、どこへ送るか**」をルールで振り分ける仕組み。
SNS より賢いルーティングができ、**定期実行（cron のようなスケジュール）**も担える。

| | 役割 |
| --- | --- |
| SQS | 仕事を**貯める** |
| SNS | 同じ通知を**一斉に配る** |
| EventBridge | イベントを**条件で振り分ける**／定期実行 |

## SES ＝ メール送信の専門サービス

**SES** は確認メールや通知メールを **送る**ためのサービス。アプリから直接 SMTP を抱えるより、
到達率や管理を任せられる。

## まとめ

- 重い処理は「依頼だけ受けて後で処理」＝非同期にすると速く・壊れにくい
- SQS は仕事を貯める待ち行列（疎結合・バッファ）
- SNS は同じ通知を一斉配信（ファンアウト）。SNS→SQS が定番
- EventBridge はイベントを条件で振り分け＋定期実行
- SES はメール送信の専門サービス
