自分の書いたコードが完璧でも、**取り込んだ依存やビルド経路**が汚染されれば下流の利用者ごと侵害される。
これがサプライチェーン攻撃で、攻撃者は「1つ汚染すれば下流全員に届く」効率を狙う。仕組みと守りの原則、
そしてGoでの具体的な防御を整理する。

## 攻撃の仕組み — 「1つ汚して全員に届く」

自分のリポジトリではなく、その手前（依存・アカウント・CI）を狙う。代表的な経路はこれ。

- **悪意ある／タイポスクワット パッケージ**: `reqeusts` のように正規名に似せた偽物を公開し、うっかりの `import` を待つ
- **メンテナアカウントの乗っ取り**: 正規の人気パッケージの公開権限を奪い、正規名のまま悪意あるバージョンを出す
- **CIの汚染**: ビルドパイプラインに侵入し、成果物（バイナリ・イメージ）にビルド時点でコードを仕込む
- **依存混同（dependency confusion）**: 社内プライベート名と同名のパッケージを公開レジストリに置き、
  解決順の隙をついて公開側を引かせる

> ⚠️ 怖いのは、**正規の名前・正規の配布経路のまま中身だけ差し替わる**ケース。名前を見ても気づけない。
> だから「誰が・何を・いつ差し替えたか」を機械的に検証する仕組みが要る。

実例を短く。**event-stream**（2018）は乗っ取られたメンテナが依存を追加し暗号通貨ウォレットを狙った。
**SolarWinds**（2020）はビルドシステムに侵入し署名済み更新に仕込んだ。**xz backdoor**（2024）は
長期間かけて信頼を得たメンテナが圧縮ライブラリにバックドアを埋めた——いずれも「信頼された経路」の悪用だ。

## 対策の原則 — 固定して検証する

守りの中心は2つ。**入れるものを最小に絞る**ことと、**入れたものを固定して検証する**こと。

| 原則 | 具体策 |
| --- | --- |
| **固定と検証** | lockfile + チェックサムでバージョンを釘付けし、取得物のハッシュを毎回照合する |
| **依存を最小に** | 使わない依存を増やさない。攻撃面（取り込む他人のコード量）を減らす |
| **CIは最小権限** | パイプラインに与えるトークン・権限を絞る（[keyless auth](/posts/ci-oidc-keyless-auth/) で長期鍵を排す） |
| **来歴の証明** | 成果物に署名し、どのソース・どのビルドで作られたかの証明（provenance / SLSA）を付ける |
| **上げる前にレビュー** | バージョン更新は自動で丸呑みせず、差分を見てから上げる |

> 🧭 C# では NuGet に `packages.lock.json`（`RestorePackagesWithLockFile`）でバージョン固定でき、
> パッケージ署名の検証も設定できる。Goの `go.sum` に相当する役割だ。

## Goでの防御 — go.sum とチェックサムDB

Goはモジュールの完全性検証を言語ツールに組み込んでいる。中心は `go.sum` とチェックサムデータベースだ。

- **`go.sum`**: 依存モジュールの**期待ハッシュ**を記録する。`go build` 等は取得物をこれと照合し、
  一致しなければ失敗する。`go.mod` がバージョンを固定し、`go.sum` が中身の同一性を保証する
- **チェックサムDB（`sum.golang.org`）**: あるモジュール／バージョンのハッシュを**全世界で1つ**に固定する
  公開ログ。初めて見る依存はここと照合するので、後からこっそり中身を差し替える攻撃を検知できる
- **`go mod verify`**: ローカルのモジュールキャッシュが `go.sum` と一致するか明示的に検査する

```bash
go mod verify        # キャッシュの内容が go.sum と一致するか検証
go mod tidy          # 使う依存だけに整理（不要な攻撃面を落とす）
```

> ⚠️ **無効化フラグに注意**。`GONOSUMCHECK`（古い非推奨の環境変数）や `GOFLAGS=-insecure` 系で
> チェックを緩めると、せっかくの検証が効かなくなる。**CIで安易に無効化しない**こと。
> プライベートモジュールを公開DB照合から外したいだけなら、専用の `GONOSUMDB`／現在の `GOPRIVATE`
> （チェックサムDB照合をスキップする対象を指定）を使う。全体を無効化してはいけない。

### 取り込んだ後の防御: vendoring と govulncheck

- **vendoring**（`go mod vendor`）: 依存をリポジトリ内に取り込み、ビルドをレジストリから切り離す。
  取り込んだ時点のコードを固定でき、レビューもしやすい
- **`govulncheck`**: 既知の脆弱バージョンを検出する公式ツール。**実際に呼ばれている**関数だけを
  報告するので、ノイズが少ない。CIに入れて既知の悪いバージョンを弾く

```bash
go install golang.org/x/vuln/cmd/govulncheck@latest
govulncheck ./...    # 既知の脆弱性のうち、実際に到達するものを報告
```

> 💡 `go.sum` は「差し替え」を、`govulncheck` は「既知の穴」を守る。役割が違うので**両方**回す。
> どちらも [CIのパイプライン](/posts/cicd-github-actions/) に組み込んで自動で落とすのが定石。

## まとめ

- サプライチェーン攻撃は自分のコードの手前（**依存・アカウント・CI**）を狙い、下流全員に届く
- 実例（event-stream / SolarWinds / xz）はどれも**信頼された経路の悪用**だった
- 原則は **最小の依存**と、**lockfile + チェックサムで固定して検証**、CIは最小権限で成果物に来歴を付ける
- Goでは `go.sum` とチェックサムDB（`sum.golang.org`）が差し替えを検知、`go mod verify` で明示検査
- **無効化フラグ（`GONOSUMCHECK` 等）で検証を緩めない**。既知の穴は `govulncheck` をCIで回して弾く
