[移行のテスト](/posts/migration-testing/)をフェーズごとに揃えても、**本番で実際に1回流す瞬間**は別問題だ。
テストは「作り込む」話、実行は「事故らせない」話。この記事は、移行やリスクの高い変更を**1回実行する手順**を
**preflight → migrate → validate** の3つの門で挟む型を整理する。データ移行だけでなく、スキーマ変更・切替・
一括更新など「戻しにくい変更を1回流す」場面に共通で効く。

## 全体像 — 実行を3つの門で挟む

「流す」処理そのもの（migrate）を、前後の門（preflight / validate）で守る。**どの門も NG なら止まる**のが肝。

```text
          ┌── NG ─▶ 中止（まだ何も触っていない）
Preflight ┤
          └── OK
               │
               ▼
            Migrate ── 失敗 ─▶ ロールバック
               │
               ▼
          ┌ Validate ┐
   OK ◀───┘          └───▶ NG ─▶ ロールバック
（完了・確定）
```

| 門 | いつ | 役割 | 失敗したら |
| --- | --- | --- | --- |
| preflight | 触る**前** | 前提が揃っているか確認し、駄目なら**始めない** | 中止（無傷） |
| migrate | 実行**中** | 本体を安全に流す（冪等・再開可能・記録） | ロールバック |
| validate | 流した**後** | 成功条件を数字で確かめる | ロールバック |

> ⭐ 一番効くのは **preflight**。事故の多くは「始めてはいけない状態で始めた」こと。**まだ何も触っていない
> 段階で止まる**のが、いちばん安く、いちばん安全な失敗の仕方だ。

## ① Preflight — 触る前に「やめる」判断をする

migrate に入る前の**中止判断の門**。ここは「動かす準備」ではなく「**動かしてはいけない条件を探す**」ためにある。
1つでも引っかかれば、本体には一切触れずに中止する。

代表的なチェック:

- **バックアップ／戻せる状態**があるか（[ロールバック戦略](/posts/deploy-rollback-strategy/)。戻せないなら流さない）
- **リハーサルが合格済み**か（[移行リハーサルと突合](/posts/migration-testing/)を本番コピーで通したか）
- **時間窓・依存**は満たすか（許容ダウンタイム内か、上流ジョブは完了しているか、他の変更と衝突しないか）
- **入力データの健全性**（想定件数・NULL率・重複が事前の想定内か。桁違いなら止める）
- **べき等キー／ロック**があり、二重実行や割り込みが防げるか

```bash
#!/usr/bin/env bash
set -euo pipefail   # 1つでも失敗したら即中止（本体に進ませない）

preflight() {
  test -f "$BACKUP" || { echo "no backup"; exit 1; }        # 戻せるか
  [ "$(count_source)" -le "$MAX_ROWS" ] || { echo "too many rows"; exit 1; }
  in_maintenance_window || { echo "outside window"; exit 1; } # 時間窓
}
preflight   # ここを抜けるまで migrate は絶対に呼ばない
```

> 🧭 C# で関数の頭に引数チェックの guard clause を並べ、条件を満たさなければ早期 return / throw して本体に
> 進ませないのと同じ発想。preflight は、その **"実行前の guard clause" を運用手順に持ち込んだもの**だ。

## ② Migrate — 本体を安全に流す

preflight を抜けたら本体を実行する。ここで大事なのは速さではなく、**途中で落ちても被害を最小化できる**性質。

- **冪等（idempotent）**: 同じ処理を何度流しても結果が同じ。途中失敗後の再実行が怖くなくなる
- **再開可能**: 進捗（どこまで処理したか）を記録し、失敗した続きから再開できる。全件やり直しを避ける
- **記録を残す**: 何を・いつ・どれだけ処理したかをログに出す。validate と原因調査の材料になる
- **分割して流す**: 一括ではなくバッチ単位に切ると、失敗の影響範囲が狭まり、再開の粒度も細かくなる

> ⚠️ 「エラーなく最後まで流れた」は成功ではない。変換ミスは**静かに間違った値を入れて完走する**。
> 「流れたか」と「正しく入ったか」は別の門（validate）で分けて確かめる。

> 💡 冪等・再開可能にしておくと、preflight で弾けなかった想定外に本番中に気づいても、**安全に止めて直して
> 続きから**、が選べる。速い一発勝負より、止まれて再開できる方が本番では強い。

## ③ Validate — 成功を数字で確かめ、go / rollback を決める

流し終えた「後」の門。**主観の"動いてそう"ではなく、事前に決めた成功条件を数字で判定**する。ここが
migrate と validate を分けている理由そのものだ。

- **突合（reconciliation）**: [データ移行](/posts/data-migration/)の3点照合——**件数・合計・サンプリング**が一致するか。
  外部キーの相手が欠けていないか（参照整合性）も見る
- **健全性チェック**: 主要な経路が動くか（[スモークテスト](/posts/smoke-testing/)）。切替を伴うなら疎通・ログイン
- **新旧の突合**: 並行稼働なら[新旧APIの応答突合](/posts/api-response-diffing/)で、同じ入力に同じ出力かを確かめる
- **判定は自動で二値に**: 「合格／不合格」を人の目視ではなく閾値で機械判定し、**不合格なら即ロールバック**へ倒す

> ⭐ validate の成功条件は **preflight より前に決めておく**。流し終えてから「何をもって成功とするか」を
> 考え始めると、都合よく甘い基準を作ってしまう。門の合格ラインは、始める前に固定する。

## パターンとして使い回す

この型は移行専用ではない。**戻しにくい変更を1回流す**場面すべてに同じ骨格で効く。

| 場面 | preflight | migrate | validate |
| --- | --- | --- | --- |
| データ移行 | バックアップ・件数・リハ合格 | 冪等な変換を分割実行 | 件数・合計・サンプル突合 |
| デプロイ | ヘルスチェック・依存の準備 | 段階的に切替（[ロールバック](/posts/deploy-rollback-strategy/)前提） | スモーク・エラー率監視 |
| スキーマ変更 | 互換性確認（[expand/contract](/posts/db-schema-migration-expand-contract/)） | 後方互換な順で適用 | 旧新両コードが動くか |
| 一括更新 | 対象件数・WHERE 条件の確認 | バッチ＋べき等キー | 更新件数と想定の一致 |

> 💡 3つの門を「毎回同じ手順」に固定するほど、実行が属人的な度胸試しでなくなる。**手順書に preflight /
> migrate / validate の見出しを立てる**だけでも、抜けが減り、レビューもしやすくなる。

## まとめ

- 移行の**テストを作り込む**ことと、本番で**1回安全に流す**ことは別。実行は3つの門で挟む
- **preflight**: 触る前の中止判断。戻せるか・リハ合格・時間窓・データ健全性を確認し、駄目なら**始めない**（無傷で止まるのが最良）
- **migrate**: 本体を**冪等・再開可能・記録付き**で、できれば分割して流す。「完走＝成功」ではない
- **validate**: 事前に決めた成功条件を**数字で二値判定**し、不合格なら即ロールバック。成功条件は始める前に固定する
- この型は移行専用ではない。デプロイ・スキーマ変更・一括更新など、**戻しにくい変更を1回流す**すべてに使い回せる
