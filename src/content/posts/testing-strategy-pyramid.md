## テストピラミッド

下（単体）が多く、上（E2E）が少ないのが健全。

```text
        ╱  E2E  ╲        少数・遅い・高コスト（ブラウザ操作, API 全体）
      ╱──────────╲
    ╱  結合テスト  ╲      中間（DB 接続, コンポーネント間）
  ╱────────────────╲
 ╱    単体テスト     ╲    大量・速い・低コスト（関数 / メソッド単位）
╱────────────────────╲
```

| 種別 | 対象 | 速度 | DB | 例 |
| --- | --- | --- | --- | --- |
| 単体 | 関数・メソッド | 速い | 不要 | UseCase のロジック |
| 結合 | 複数コンポーネント | 中 | 要ることが多い | UseCase + Repository |
| E2E | システム全体 | 遅い | 必要 | HTTP リクエスト → レスポンス |

> ⚠️ ピラミッドが「逆三角形」になると（E2E ばかり）、遅くて壊れやすいテスト群になる。
> 速い単体テストを土台に厚くするのが基本。

## 何をテストすべきか

```text
✅ 優先する
  - ビジネスロジック（Entity, UseCase）
  - バリデーション（不正入力をはじく）
  - エラーハンドリング（異常系の振る舞い）
  - 境界値（0件, 1件, 上限値）

❌ 書かなくていい
  - フレームワークの機能そのもの（HTTP ルーティング等）
  - 外部ライブラリの動作確認
  - getter/setter のような自明なメソッド
```

## テストしやすいコード = 良い設計

テストの書きにくさは、設計の問題が表面化したもの。

| 書きにくいコード | 書きやすいコード |
| --- | --- |
| グローバル変数に依存 | 依存をインターフェースで注入 |
| 関数内で DB に直接接続 | 副作用（DB/ファイル/通信）を分離 |
| 1関数が多くをやる | 責務が1つ |
| 依存をハードコード（new） | 外から注入 |

> 💡 「テストしにくいから書かない」ではなく「書きやすいよう設計を見直す」が正しい向き。
> [クリーンアーキテクチャ](/posts/clean-architecture-concepts/)で依存を逃がしておくと、
> 単体テストが自然に書けるようになる。

## まとめ

- 単体を厚く、E2E を薄く ——「ピラミッド」を保つ
- 優先すべきはロジック・バリデーション・異常系・境界値
- フレームワークや外部ライブラリ自体はテストしない
- テストの書きにくさは設計を見直すサイン

最後に、テストを包む「QA（品質保証）」の考え方へ。
