> 📚 シリーズ「Goで学ぶアーキテクチャとテスト」(7 / 7・最終回)

シリーズの締めは QA。テスト単体ではなく、**品質を作り込む仕組み** としてまとめる。

## テストと QA の違い

- **テスト**: バグを見つける活動
- **QA**: バグが生まれにくい仕組みを作る活動（テストはその一部）

QA が含むもの ── テスト設計・実行 / コードレビュー / CI/CD / 静的解析(linter) /
モニタリング・アラート / 障害対応プロセス。

## 品質は「上流」で作り込むほど安い

```text
修正コスト
 高 │                      ╱  ← 運用中に見つかると最も高い
    │                  ╱
    │              ╱
 低 │  ╱──────╱
    └──────────────────────────→
     設計  実装  テスト  リリース  運用
```

> 💡 上流（設計・実装）で見つけるほど修正は安い。だから「後でまとめてテスト」ではなく
> 書いた瞬間に静的解析やテストで弾く方が結局速い。

## Go でできる品質施策

```bash
# 静的解析
go vet ./...          # Go 標準の静的解析
golangci-lint run     # 複数 linter をまとめて実行

# テスト
go test ./...         # 単体テスト
go test -race ./...   # レースコンディション検出
go test -cover ./...  # カバレッジ

# フォーマット（チーム共通の正解が1つ）
go fmt ./...
```

> 🧭 Go は `go fmt` で整形が一意に決まる。「インデント論争」が起きないのは地味に効く QA。

## CI に組み込む流れ

```text
Pull Request → CI が自動実行
  ├─ go fmt        フォーマットチェック
  ├─ go vet        静的解析
  ├─ golangci-lint linter
  ├─ go test -race 全テスト
  └─ go test -cover カバレッジ
全部通ったら → レビュー → マージ → 自動デプロイ
```

人間のレビューは「設計やネーミング」に集中し、機械的な指摘は CI に任せるのがコツ。

## アーキテクチャと QA の好循環

```text
依存が分離されている
  → 層ごとにテストできる → テストが速い → CI が速い
  → テストを書くハードルが下がる → テストが増える → バグが減る
  → 変更に自信が持てる → リファクタできる → 品質が上がる ──┐
  └──────────────────────────────────────────────────────┘
```

> ⭐ アーキテクチャとテストは別の話ではない。良い設計はテストを書きやすくし、
> テストを書く習慣は良い設計を維持する。互いに強化し合う関係。

## まとめ

- テスト=バグを見つける／QA=バグが生まれにくい仕組みを作る
- 品質は上流で作り込むほど安い。書いた瞬間に弾く
- Go は `vet` / `golangci-lint` / `test -race` / `fmt` が揃う
- CI で機械的チェックを自動化し、レビューは設計に集中する
- 良いアーキテクチャとテストは好循環を生む
