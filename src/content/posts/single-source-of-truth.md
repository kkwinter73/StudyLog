**SoT（Single Source of Truth＝単一の真実源）** は、「ある情報について“正しい元”は **ただ1か所** に置き、
他はそこを参照する（または自動生成する）」という設計の考え方。地味だが、効きどころが多い。

## なぜ要るのか：二重管理はいつか食い違う

同じ事実を2か所に書くと、片方だけ直して **食い違う**。そうなると「どっちが正しいの？」と
誰も判断できなくなる。

```text
税率を A.go と B.go の両方にベタ書き
  → 法改正で A だけ直す
  → B は古いまま
  → 同じアプリ内で計算結果が食い違う 💥
```

> ⚠️ 怖いのは「壊れる」ことより **静かにズレる** こと。エラーも出ず、間違った値が使われ続ける。

## SoT とは（例え：正本は1冊だけ）

契約書の「正本」は1冊だけ作り、あとは全部コピー。コピーは正本を写したものに過ぎず、
変更は **必ず正本に対して行う**。SoT も同じで、

- ある情報の **正は1か所**（＝真実源）
- 他の場所はそこを **参照** するか、そこから **生成** する
- 直すときは **真実源だけ** を直す

## メリット

| メリット | なぜ |
| --- | --- |
| 食い違わない | 正が1つなので矛盾が起きない |
| 変更が一箇所で済む | 直す場所を探し回らなくてよい |
| 信頼できる | 「ここを見れば正しい」が全員で共有される |
| 自動化の起点になる | 真実源から派生物を生成・検証できる |

## 具体例：コード・インフラ・ドキュメント

**コード** — 定数や設定は1か所に定義して import する。

```go
// ❌ あちこちにベタ書き → ズレる
price := base * 1.10 // ここにも 0.10
total := sub * 1.10  // あそこにも 0.10

// ✅ 真実源を1つに
package tax
const Rate = 0.10 // 税率の正はここだけ。変更はこの1行
```

> 🧭 「同じことを繰り返さない」= DRY 原則の実践でもある。型定義や enum を1か所に置いて
> 各所で使い回すのも同じ発想。

**インフラ** — IaC（Terraform など）で「あるべき構成」をコードに書き、それを真実源にする。
手で AWS をいじらず、コードを直して反映する（手動変更は真実源とのズレを生む）。

**ドキュメント** — 「なぜそう決めたか」は意思決定ログ（ADR）に1か所だけ残す。
仕様はコード、進捗は Issue、と **種類ごとに真実源を決める** のがコツ。

## 派生物は「コピー」ではなく「生成」する

真実源から複製が必要なときは、手でコピーせず **生成**する。手コピーは必ずいつか同期が切れる。

```text
真実源（1つ） ──生成──▶ 派生A（自動）
              └─生成──▶ 派生B（自動）
直すのは真実源だけ。派生は作り直せばよい。
```

## このブログでの実践

実はこの StudyLog 自体が SoT で組まれている。

- 記事の **frontmatter スキーマ** の正は `src/content.config.ts` だけ
- **サイト設定**（タイトル・ナビ）の正は `src/site.ts` だけ
- **意思決定** の正は `docs/decisions/` で、索引は[スクリプトで自動生成](/posts/why-architecture-and-dip/)（手編集しない）

> 💡 「種類 → 唯一の置き場所」を最初に決めておくと、どこに書くか迷わず、後から食い違わない。

## まとめ

- SoT ＝「ある情報の正はただ1か所」。他は参照か生成
- 二重管理の怖さは“静かにズレる”こと。SoT はそれを根本から防ぐ
- メリットは「食い違わない・変更が一箇所・信頼できる・自動化の起点」
- コードは定数/型を一元化、インフラは IaC、ドキュメントは種類ごとに真実源を決める
- 複製は手コピーせず生成する。直すのは常に真実源だけ
