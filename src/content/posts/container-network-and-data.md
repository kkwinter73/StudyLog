> 📚 シリーズ「コンテナをちゃんと理解する」(4 / 5)

複数のコンテナを動かすと「互いにどう通信するか」「データをどう残すか」が問題になる。
[DNS](/posts/dns-basics-and-tools/)の知識を土台に、**サービス名での名前解決**と、データを残す
**ボリューム/バインドマウント**を押さえる。

## コンテナ間の通信 — サービス名で名前解決

コンテナは[専用のネットワーク空間(NET namespace)](/posts/container-namespace-cgroup/)を持つ。
同じ Docker ネットワークに乗せると、互いに **サービス名（コンテナ名）で呼び合える**。

```yaml
# docker-compose.yml（抜粋）
services:
  web:
    image: myapp
    environment:
      DB_HOST: db          # ← IPではなく "db" という名前で繋ぐ
  db:
    image: postgres
```

- compose は同じネットワークを自動で作り、各サービス名を**ホスト名として[名前解決](/posts/dns-basics-and-tools/)**できるようにする
- `web` から `db:5432` で繋げる。コンテナの IP は起動のたび変わるが、**名前は変わらない**

> 💡 IP直書きだと再起動で壊れる。だから「名前で繋ぐ」。これは[AWS の Service Connect](/posts/aws-compute-explained/)
> （サービスを名前で繋ぐ仕組み）と同じ発想で、Docker 内蔵のDNSがそれを担っている。

## データは放っておくと消える

コンテナは使い捨てが前提。**コンテナを消すと、中で書いたファイルも一緒に消える**。
DBのデータやアップロードを残すには、外に逃がす仕組みが要る——それがマウント。

## ボリューム vs バインドマウント

データを永続化する方法は主に2つ。役割が違う。

| | ボリューム | バインドマウント |
| --- | --- | --- |
| 実体 | Docker が管理する領域 | **ホストの任意のディレクトリ**を直結 |
| 指定 | 名前（`pgdata:/var/lib/...`） | ホストのパス（`./src:/app`） |
| 主な用途 | 本番のデータ永続化(DB等) | 開発（ホストのコードを即反映） |
| 移植性 | 高い（環境に依存しにくい） | ホストのパス構成に依存 |

```yaml
services:
  db:
    image: postgres
    volumes:
      - pgdata:/var/lib/postgresql/data   # ボリューム：DBデータを永続化
  web:
    image: myapp
    volumes:
      - ./src:/app                        # バインドマウント：開発中のコードを直結
volumes:
  pgdata:
```

> ⚠️ 「コンテナを作り直したらデータが消えた」はマウントし忘れの定番。**残すデータはボリュームへ**、
> 開発で即反映したいコードはバインドマウントへ、と使い分ける。

## まとめ

- コンテナは NET namespace で**専用のネットワーク**を持つ。同じネットワーク上は**サービス名で通信**
- compose が内蔵DNSで「サービス名→IP」を解決。IPは変わっても名前は不変
- コンテナは使い捨て。**中に書いたデータは消える**ので外に逃がす
- **ボリューム**＝Docker管理の永続領域(本番DB向き) / **バインドマウント**＝ホストのパス直結(開発向き)
- データ消失の多くはマウント忘れ。残すならボリューム
