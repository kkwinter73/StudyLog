「削除したはずのデータを戻したい」「誰がいつ消したか追いたい」——こうした要求に応えるのが **論理削除**（soft delete）。
行を本当に `DELETE` する**物理削除**に対し、論理削除は `deleted_at` のような**印を立てて「消えた扱い」**にする。
便利だが、ユニーク制約・クエリでの除外漏れ・カスケード・性能と**落とし穴が多い**。使いどころ・実装・
そして「本当に消す」必要がある場面までを押さえる。

## 物理削除と論理削除 — 何が違うか

| | 物理削除 | 論理削除 |
| --- | --- | --- |
| 操作 | `DELETE FROM ...`（行が消える） | `UPDATE ... SET deleted_at = now()`（印を立てる） |
| 復元 | 基本できない（バックアップ頼み） | 印を消せば戻せる |
| 参照整合 | 参照先が消えると壊れうる | 行は残るので壊れない |
| 容量・性能 | 減る・速い | 溜まる・遅くなりうる |
| 既定クエリ | 何もしなくてよい | **毎回「生きてる行」に絞る**必要 |

- 論理削除は「削除」という名の **`UPDATE`**。行はテーブルに残り続ける
- 復元・監査・参照整合が要るドメイン（注文・ユーザー・会計）で効く
- 代償として、**以後すべての読み取りが「削除済みを除外する」責任**を負う

## なぜ論理削除を使うのか

消さずに残す理由は主に3つ。要求が無いなら物理削除で十分。

- **復元できる**: 誤削除やユーザーの「やっぱり戻して」に応えられる
- **監査・履歴**: 「誰が・いつ・何を消したか」が残る。会計や規制のある領域で必須になりがち
- **参照整合を壊さない**: 過去の注文が参照するユーザーを物理削除すると履歴が壊れる。残せば安全

> 💡 「消えて見えればいい」要求のほとんどは論理削除で足りる。逆に**法的に本当に消す必要**（後述）が
> 出た瞬間、論理削除だけでは要件を満たせなくなる。最初に「どっちの削除か」を問うのが設計の起点。

## 実装 — deleted_at で印を立てる

真偽フラグ `is_deleted` でもよいが、**`deleted_at`（タイムスタンプ）**が定番。「消えたか」に加え「いつ消えたか」も残る。

```sql
-- 削除（印を立てるだけ）
UPDATE users SET deleted_at = now() WHERE id = 42;

-- 通常の読み取りは必ず「生きてる行」に絞る
SELECT * FROM users WHERE deleted_at IS NULL;

-- 復元
UPDATE users SET deleted_at = NULL WHERE id = 42;
```

- NULL = 生存、非NULL = 削除済み（削除時刻）。`is_deleted` より情報量が多い
- 除外を書き忘れないよう、**「生存行だけのビュー」や ORM のデフォルトスコープ**で一元化するのが安全
- 部分インデックス `WHERE deleted_at IS NULL` を張ると、生存行への検索が速くなる

> 🧭 C# の EF Core なら **`HasQueryFilter(e => e.DeletedAt == null)`** でグローバルフィルタを一度定義すれば、
> 以後のクエリが自動で除外してくれる（Go の `gorm` の `gorm.DeletedAt` 相当）。手書き SQL 主体なら**ビュー**で同じ効果を作る。

## 落とし穴

論理削除の難しさは「消えていないのに消えたことにする」ズレから来る。代表的な4つ。

- **ユニーク制約**: `email` を UNIQUE にすると、削除済みの同じメールで**再登録できない**。
  解決は「`deleted_at` を含む部分ユニークインデックス（生存行のみ一意）」にする
- **除外漏れ**: どこか1つのクエリで `deleted_at IS NULL` を書き忘れると、削除済みが表に出る。**既定で除外**する仕組みで防ぐ
- **カスケード**: ユーザーを論理削除しても、その投稿・セッションは**自動では消えない**。関連の扱い（連鎖して印を立てるか）を決める
- **性能劣化**: 削除済みが溜まりテーブルが肥大化し、インデックスも太る。**部分インデックス**＋**定期的なアーカイブ/物理パージ**で抑える

> ⚠️ 一番怖いのは**除外漏れ**。集計や管理画面、JOIN 先など「見落としがちな経路」で削除済みが混じると、
> 数字が静かに狂う。**除外はアプリ全体で1箇所に寄せる**（ビュー / ORM スコープ）のが鉄則。

## 「本当に消す」必要があるとき

論理削除は「見えなくする」だけ。データは残っている。これが要件と衝突する場面がある。

- **法的な削除要求**（GDPR の「忘れられる権利」等）: 印では不十分。**物理削除か匿名化**が要る
- **容量・コスト**: 無限に溜めれば効く。一定期間後に物理パージするライフサイクルを決める
- 現実解は**ハイブリッド**: まず論理削除（猶予・復元可能）→ 一定期間後に**物理削除/匿名化**でバッチ消去

> ⭐ 「論理削除したから消えた」は誤解を生む。個人情報は**猶予期間つきで最終的に物理消去/匿名化**まで
> 設計して初めて「消した」と言える。削除は"状態"ではなく"ライフサイクル"で考える。

## まとめ

- 論理削除は `DELETE` ではなく **`deleted_at` を立てる `UPDATE`**。行は残り、復元・監査・参照整合に効く
- 代償は「**以後すべての読み取りが削除済みを除外する責任**」。ビュー / ORM スコープで**1箇所に寄せる**
- 落とし穴は **ユニーク制約・除外漏れ・カスケード・性能**。部分ユニーク/部分インデックスで対処
- **法的削除や容量**には論理削除だけでは足りない。**猶予つき論理削除→物理消去/匿名化**のハイブリッドで
- 設計の起点は「**どちらの削除が要るか**」を最初に問うこと
