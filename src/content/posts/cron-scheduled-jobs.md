「毎晩2時にバッチを回す」「5分ごとに集計する」——こうした**定期実行**を担う Linux の標準的な仕組みが **cron**。
常駐する cron デーモンが、**crontab** という設定に書かれた「いつ・何を実行するか」を見て、時間になったらコマンドを起動する。
この記事は crontab の書き方と定番のつまずき、そしてクラウド時代の定期実行までを押さえる。

## crontab の書き方 — 5フィールド＋コマンド

crontab の1行は、**時刻を表す5つのフィールド**と、実行する**コマンド**でできている。

```text
┌───── 分   (0-59)
│ ┌─── 時   (0-23)
│ │ ┌─ 日   (1-31)
│ │ │ ┌ 月   (1-12)
│ │ │ │ ┌ 曜日 (0-7, 0と7=日曜)
│ │ │ │ │
0 2 * * *  /usr/local/bin/backup.sh    ← 毎日 2:00 に実行
```

| 書き方 | 意味 |
| --- | --- |
| `0 2 * * *` | 毎日 2:00 |
| `*/5 * * * *` | 5分ごと |
| `0 9 * * 1` | 毎週月曜 9:00 |
| `0 0 1 * *` | 毎月1日 0:00 |
| `@daily` / `@hourly` / `@reboot` | 1日1回 / 1時間ごと / 起動時 |

- `*` は「毎」、`*/n` は「n ごと」、`,` で列挙（`0,30`）、`-` で範囲（`1-5`）
- 編集は `crontab -e`、確認は `crontab -l`。ユーザーごとに持てる

## よくあるつまずき — 「手元で動くのに cron で動かない」

cron のトラブルはほぼ**環境の違い**に集約される。手元のシェルと同じ前提で書くと必ずハマる。

- **環境変数・PATH が違う**: cron は最小限の環境で動く。`.bashrc` は読まれない。コマンドは**絶対パス**で書くか PATH を明示
- **作業ディレクトリ**: 既定はホーム。相対パスに頼らず、スクリプト側で `cd` するか絶対パスにする
- **出力とログ**: 標準出力/エラーは既定でメールに飛ぶ（見られず消える）。**必ずファイルへ**: `>> /var/log/job.log 2>&1`
- **タイムゾーン**: 実行時刻はサーバのTZ基準。ズレるなら `CRON_TZ=Asia/Tokyo` を指定
- **多重起動**: 前回が終わる前に次が始まると重なる。`flock` で**同時実行を防ぐ**
- **実行漏れ**: サーバが停止中の予定は**後追いされない**（取り戻したいなら anacron）

> 🧭 .NET でいう Windows タスクスケジューラや Quartz.NET / Hangfire と同じ役割。Linux ではこの cron が標準。
> 「環境が違うと動かない」は、[シェルと環境変数](/posts/shell-env-and-scripting/)の理解がそのまま効く所だ。

## クラウド/コンテナ時代の定期実行

1台のサーバの crontab は手軽だが、本番では弱点がある。だから今は別の仕組みに寄せることが多い。

- **単一障害点**: そのサーバが落ちれば定期実行も止まる。冗長化しづらい
- **観測しづらい**: 失敗に気づけない。成功/失敗の記録や通知を別途作り込む必要がある
- **今の定番**: スケジューラ（**EventBridge Scheduler** / k8s の **CronJob**）が、コンテナ化したバッチを時間で起動する。
  例えば AWS なら「**ECS のスケジュールタスク**」として、バッチ用イメージを時間になったら1回だけ動かす

> 💡 「cron の本番 image を [ECR](/posts/aws-ecr-container-registry/) に push」とは、この**コンテナ化した定期バッチ**の話。
> バッチを箱（イメージ）に詰めてレジストリに置き、スケジューラが時間になって pull・実行する——cron の役目をクラウドが担う形だ。

## まとめ

- cron は Linux 標準の**定期実行**。crontab の**5フィールド（分・時・日・月・曜日）＋コマンド**で「いつ何を」を宣言
- `*/5 * * * *`＝5分ごと、`0 2 * * *`＝毎日2時。`*` は毎、`*/n` は n ごと
- つまずきは**環境の違い**——PATH・作業ディレクトリ・出力先・タイムゾーン・多重起動・実行漏れ
- 絶対パス、ログはファイルへ、`flock` で多重防止、が定番の防御
- 本番は1台のcronを卒業し、**スケジューラ＋コンテナ**（ECSスケジュールタスク等）に寄せるのが今の主流
