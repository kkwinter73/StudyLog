# 開発環境（ハーネス）の仕組みと、その理由

このリポジトリの `.claude/` 一式は、「AIエージェント・ハーネスエンジニアリング」の原則を
**単一リポ・個人運用向けに身の丈で簡略化**して適用したもの。ここでは「何が・なぜ」あるかを説明する。

> ハーネスの一文: **賢いエージェントを安全に速くするのは、賢いプロンプトではなく賢い"馬具"。**
> ルールは必読の場所へインラインし、強制経路（hook）で裏打ちし、止め方は判定可能性で選ぶ。

## 構成

| 要素 | 役割 | 実体 |
| --- | --- | --- |
| 常時文脈 | エージェントが必ず読む規約 | ルート `CLAUDE.md`（自己完結・最重要1行をインライン） |
| 知識の住所 | 何をどこに書くか | `docs/`（[README](../README.md) の SoT 表・decisions） |
| 強制ガード | 規約をツールレベルで効かせる | `.claude/hooks/` ＋ `.claude/settings.json` |
| 自動注入ルール | 記事の書き方 | `.claude/rules/content.md` |
| 手順コマンド | 記事化フロー | `.claude/skills/new-log/` |

## ガード hook の "型" と、その判断軸

ハーネスの心臓部。**止め方は「機械的に判定できるか」と「取りこぼしの安全方向」で選ぶ**。

| hook | 発火点 | 型 | なぜこの型か |
| --- | --- | --- | --- |
| `guard-generated.sh` | PreToolUse(Write\|Edit) | **block** | 「dist/・.astro/ への書き込み」はファイルパスだけで確実に判定でき、止めるべき操作。構造化入力に絞るので誤検知ゼロ。 |
| `lint-post.sh` | PostToolUse(Write\|Edit) | **reminder** | frontmatter の不足は判定できるが、下書き途中の保存もある＝今すぐ完成とは限らない。止めずに注意文を出す。 |
| `full-check.sh` | Stop | **full gate** | turn を締める前に `astro check`。壊れたまま離れる事故を防ぐ最後の安全網。 |

### full-check の3つの安全弁（ブロッカーにしない工夫）

1. **無限ループ防止**: 再発火（`stop_hook_active`）では止めない。直せないエラーで永久ループに陥らせない。
2. **環境差で無音**: `npx` / `node_modules` が無ければ黙って通す。環境差で作業を止めない。
3. **邪魔をしない**: `src` に変更が無い turn（雑談など）はスキップ。

## なぜ block を増やさないのか

ハーネス原則：**判定できないものを block すると正規用途まで殺す**。
このリポで block は「生成物への書き込み」だけ＝パスで確実に判定でき、相手は善意のミス。
曖昧な判断（この内容は記事向きか等）は block せず、reminder かスキルの手順に委ねる。

## 単一リポなので省いたこと

元ドキュメントの「1セッション=1担当リポ」「テンプレSoT＋手動コピー」「越境allowlist」などは
**複数リポ前提の仕組み**なので、この個人プロジェクトでは採用していない。
原則（必読の場所へインライン＋hookで裏打ち／薄い記録／SoTを1つに）だけを引き継いでいる。
